## Screen 1 — Orientation（オリエンテーション）（2 min）

### このモジュールを終えるとできるようになること

Claude Code は、ターミナルで動く開発パートナーだ。

これまでのモジュールで、プロンプト、ツールスキーマ、コンテキストエンジニアリング、エージェントループ、マルチモーダル入力といった API の基本要素を整えてきた。このモジュールはその基盤の上に成り立つ。Claude Code は同じモデルをターミナルで動かし、パーミッションの層、設定の仕組み、チーム共有の機能をそこに加える。MCP が、外部サービスとの安全な連携を実現する。このモジュールでは、Claude Code と MCP を、堅牢なセキュリティと確実なデプロイに合わせて設定する方法を学ぶ。

このモジュールを終えると、次のことができるようになる。

1. Claude Code を explore → plan → code のループで動かし、作業のリスクに見合ったパーミッションモードを選ぶ。エージェントの生産性を保ちつつ、タスクに必要な分以上の権限は与えない。
2. AI が生成したコードを読み、出力を鵜呑みにせずに確認する。確度の高い所見は採用し、怪しいものは検証し、判断を誤ると被害が大きい箇所には人間のレビューを挟む。
3. CLAUDE.md、ルールファイル、hooks、サブエージェントを使って、プロジェクトのコンテキストを長く保つ。
4. ワークフローを skills、カスタムコマンド、プラグインとしてパッケージ化する。skill は一度書けば、Claude Code、Messages API、Agent SDK のどれでも同じように動く。
5. ツール、リソース、プロンプトを Claude に提供する MCP サーバーを構築する。クライアントとサーバーの通信方式に合ったトランスポートを選び、誰がそのサーバーを読み込むかをスコープで制御する。
6. Claude を社内システムに接続し、規制の厳しい顧客でも受け入れる認証方式を整える。コードの近代化のような案件を、セキュリティレビューに耐える形で要件として固める。

このモジュールは、コードで Claude を動かせるようになった開発者が、次にそれを設定可能に、共有可能に、そして実システムに安全に繋げられるようにするためのものだ。実践的で、コードを中心に据え、パターンで考える読者を想定している。Module 2 の API パターンには慣れていることを前提とする。エージェントループ、ツールスキーマ、コンテキストエンジニアリングは改めて説明しない。代わりに、動く統合を取り巻く技術判断を扱う。具体的には、ターミナルで Claude Code をパーミッションモデルの下で動かす方法、プロジェクトのコンテキストを長く保つ方法、そしてチームメイトがインストールできる形にワークフローをまとめる方法だ。さらに、クレデンシャルを漏らさず、セキュリティレビューも通る形で、MCP 経由で Claude を外部や社内システムに接続する方法も扱う。

### このモジュールの「ビルド」

このモジュールのすべては、ひとつの繰り返し現れる問題を軸に組み立てられている。自分のマシン、自分のセッション、ステージング環境で動いたコードが、他人が実行し、本番環境で、社の実システムに対して走るときにも持ちこたえなければならない、という問題だ。

自分のマシンの上では、パーミッションモードは安全に思える。プロジェクトのルールは守れる範囲に収まっている。skill は自分のスクリプトを見つけられる。クレデンシャルは設定ファイルの中にあった。接続はステージングのテストを通った。ところが、作業が自分のマシンを離れた途端、そのどれもが失敗に変わる。パーミッションモードがスコープに入っていなかったファイルを消す。ひとつのルールが何百行もの下に埋もれる。skill が他のどのマシンにも存在しないパスを指す。commit されたキーが数時間で漏れる。ステージング専用の設定ステップが本番の接続を落とす。このモジュールで学ぶのは、どの設定判断がどの失敗を防ぐのかを、チームメイトや監査人の目に触れる前に押さえることだ。

## Screen 2 — Permission Modes & Human Gates — Teaching（パーミッションモードと人間の確認）（17 min）

### Claude Code のエージェントループ、パーミッションモード、設定、そして人間の確認を挟む場所

Module 2 では、エージェントループが API レベルでどう動くかを確認した。モデルがツールを呼び出し、結果を受け取り、タスクが終わるまでそれを繰り返す。Claude Code は同じループをターミナルで動かすが、もう一段階レイヤーを重ねる。エージェントが行おうとする操作をすべて、パーミッションの仕組みで挟み込む。設定を変える前に、まずこのループがどう動き、パーミッションモードが何を制御するのかを把握しておく必要がある。

#### Claude Code がタスクを進める順序：explore → plan → code

Claude Code にタスクを渡しても、すぐに書き始めるわけではない。まずファイルを読み、関連するロジックを辿り、コードベースの全体像を頭に作る。これが explore（探索）のフェーズだ。変更を提案できるほど理解が進んだら、次に plan（計画）を作る。plan は、これから行おうとする編集を構造化して記述したものだ。あなたが plan を確認し、承認して初めて、code フェーズに移り、変更を書いて実行する。

この順序が重要なのには理由が二つある。一つは、出力の質が上がること。Claude Code は触れる前にコードベースを理解しているので、勝手な前提を減らし、下流への影響をより広く拾える。もう一つは、パーミッションモードがここで繋がること。plan モードは Claude Code を explore フェーズに留め置き、あなたが許可を出すまでファイルの編集もシェルコマンドもすべて止める。だから、見知らぬコードベースやミスが響く作業では、便利なデフォルトになる。

#### パーミッションモード：承認・確認・制約

パーミッションモードは、Claude Code がどのくらいの頻度で確認を求めて止まるかを制御する。どのモードも、速度と監視のバランスをそれぞれ異なる形でとる。どれを選ぶかは、コードベースをどれだけ把握しているかと、変更をどのくらい元に戻せるかで決まる。

各モードが何を自動で通し、何にまだ確認を挟み、どこに限界があるかは、各タブを選んで確認してほしい。

**default**
- **自動で通すもの:** 読み込みのみ。ほぼすべての編集やコマンドの前に確認を求める。
- **まだ確認を挟むもの:** すべてのファイル編集とシェルコマンドで承認が必要。
- **限界:** 安全だが、信頼できる作業では遅い。新規プロジェクトや見知らぬコードベースでのベースライン。

**acceptEdits**
- **自動で通すもの:** 読み込み、ファイル編集、作業ディレクトリ内でのよく使うファイルシステムコマンド（mkdir、touch、rm、rmdir、mv、cp、sed）。自動承認は作業ディレクトリ内のパスに限定され、保護対象のパスには引き続き確認を求める。
- **まだ確認を挟むもの:** それ以外のシェルコマンド、作業ディレクトリ外への書き込み、保護対象パスへの書き込み。
- **限界:** シェルの実行にも人間の目が必要な、信頼できるローカル作業向け。エージェントがスクリプトを実行しなければならない場合には適さない。

**plan**
- **自動で通すもの:** 読み込みのみ。調査と提案だけを行い、編集は一切しない。
- **まだ確認を挟むもの:** あなたが plan を承認するまで、すべてのファイル編集とシェルコマンド。
- **限界:** 機密性の高い、あるいは見知らぬコードベースでの探索と計画策定向け。出力を書かなければならないタスクには適さない。

**auto**
- **自動で通すもの:** すべて。ただし、独立した分類器がまず個々の操作を審査し、要求の範囲を超えてエスカレートするもの、認識されていないインフラを狙うもの、敵対的・不適切な内容に駆動されているものはブロックする。
- **まだ確認を挟むもの:** 本番デプロイやマイグレーション、大量削除、クレデンシャルの持ち出し、main への force-push はデフォルトでブロックされる。
- **限界:** 確認の回数は減るが、安全を保証するものではない。これはリサーチプレビューであり、機密操作のレビューを置き換えるものではない。利用可否はプラン、モデルバージョン、管理設定に依存する。ビルドの前に、常に最新の要件を確認すること。

**dontAsk**
- **自動で通すもの:** allow ルールで事前承認したツールと、読み込み専用のコマンドのみ。それ以外はすべて自動で拒否（DENY）する。
- **まだ確認を挟むもの:** 許可リストにないツール呼び出しはすべて拒否される。承認を待つキューはない。
- **限界:** ロックダウンされた CI やスクリプト向けに作られている。よく制限できるが、ローカルのインタラクティブな作業での摩擦を減らす手段ではない。

**bypassPermissions**
- **自動で通すもの:** すべてのツール呼び出し。確認プロンプトも安全チェックもない。
- **まだ確認を挟むもの:** 通常動作では確認を挟まない。標準のパーミッションチェックはバイパスされ、`rm -rf /` や `rm -rf ~` のような壊滅的な削除コマンドだけが最後の砦としてプロンプトを出す。
- **限界:** 環境が使い捨ての、隔離されたコンテナや VM の内部のみ。稼働中のコードベースに対向した開発者のワークステーションでは決して使わない。

#### 設定はどこに置き、誰に適用されるか

設定は複数のレベルに置ける。各レベルが、そこに書かれたルールの適用範囲を決める。

- **ユーザーレベル（`~/.claude/settings.json`）:** そのマシン上のすべてのプロジェクトに適用される。どこでも適用したい個人設定、例えば探索作業で好むデフォルトモードを置くのに適している。
- **プロジェクトレベル（`.claude/settings.json`、リポジトリに commit）:** リポジトリを clone する全員に適用される。チーム全体の規約、プロジェクトが使うツールの allow ルール、触ってはいけないパスの deny ルールを置くのに適している。
- **ローカルプロジェクトレベル（`.claude/settings.local.json`）:** ひとつのプロジェクト向けの個人の上書き。自動的に gitignore される。チーム全体には commit したくない、自分だけの設定を置くのに適している。
- **エンタープライズレベル（`managed-settings.json`、管理者が設定）:** ユーザーやプロジェクトファイルで上書きできない。環境ファイルの編集の拒否や、全プロジェクトにわたる特定シェルコマンドのブロックなど、組織全体のセキュリティ制御を置くのに適している。

allow ルールと deny ルールは、選んだモードの上に重なる。モードが何であれ、deny ルールは常に allow ルールに優先する。最も強固なガバナンス制御は、エンタープライズレベルの deny ルールだ。個々の開発者は削除できず、bypass モードが設定されていても適用される。

#### 人間がまだ見るべき場所：最悪のコストでレビューのゲートを置く

パーミッションモードと deny ルールが決めるのは、エージェントが確認なしで何をできるかだ。それらは、操作が適用される前に、あなた（人間）がどこでまだ目を通す必要があるかは決めない。その判断はひとつの問いに帰着する。安全なモードと危険なモードを分けるのと同じ問いだ。もし誰も確認せずにこの操作が走ったら、最悪の結果は何か。ミスのコストが低いほど、多くを通してよい。コストが高く、戻すのが難しいほど、そのステップには実行前に人間の確認が必要になる。

エージェントがコードを書いている場合でも、PR にコメントしたりブロックしたりするボットのように自動ステップで無人で走っている場合でも、ゲートを置く場所は同じ最悪ケースの問いで決まる。そこから三つの置き場所が導かれる。

- **リスクが低く、元に戻せる操作はゲートなしで通す。** フォーマット修正や作業ディレクトリ内に収まる編集は、間違えても被害が小さい。いちいち人間に承認を求めると、不要な監視のオーバーヘッドを招き、作業を遅くするだけだ。これが acceptEdits の想定するケースだ。
- **元に戻しにくい、あるいは機密パスに届く操作はゲートを挟む。** 作業ディレクトリ外への書き込み、破壊的なシェルコマンド、セキュリティに関わるファイルや保護対象ファイルへの編集がこれに当たる。そこでのミスのコストは高い。だからエージェントは、実行前に一旦止まって、その操作を人間に見せるべきだ。deny ルールがこれを確定的に強制し、default または plan モードは判断のあいだプロンプトを保持する。
- **チームが機密と記したコードへの変更では、エージェントを唯一のゲートにしない。** そこではエージェントの作業は人間の判断への入力であって、判断の置き換えではない。だから、エージェントやそのレビューがどれほど自信ありげに見えても、変更がマージされる前に人が確認しなければならない。

ゲートの置き場所とパーミッションモードの選択は、両面から見れば同じ一つの決定だ。モードはセッション全体のデフォルトを決める。ゲートは、デフォルトに任せるにはコストが高すぎる一つの操作のために、そのデフォルトを上書きする場所だ。どちらも、「これが確認なしで走ったら何が壊れるか」という問いから来る。

> **Cost · Complexity · Risk（コスト・複雑さ・リスク）**
> **Cost:** 信頼できる作業を default モードで走らせると、ツール呼び出しごとにプロンプトのレイテンシが乗り、長いリファクタではそれが積み重なる。
> **Complexity:** 複数レベルの設定とファイルが上書きの階層を持つため、一貫した注意が必要になる。エンタープライズレベルの deny ルールがプロジェクトレベルの allow ルールと矛盾する場合、プロジェクトの設定を保守する全員がそのことを理解していなければならない。
> **Risk:** リスクは、コンテキストに合わないモードを使うことにある。例えば、苛立ちから隔離されていないマシンで bypass モードを設定すると、エージェントと実ファイルのあいだの安全プロンプトをすべて取り除く。他のモードと違い、保護対象パスのガードも消える。

## Screen 3 — Permission Modes & Human Gates — Watch Out（パーミッションモードと人間の確認・要注意）（3 min）

### 機能するはずだったあのひとつのプロンプトを消した bypass モード

> **Setup**
> 作業がルーチンに思えたので、絶え間ないプロンプトを止めるために bypassPermissions に切り替えた。エージェントは何日もお行儀よく振る舞い、クリーンアップタスクはシンプルで、ツール呼び出しごとにプロンプトを出す挙動は無用の摩擦に思えた。

#### トランスクリプト

次のペアリングセッションは、3日間トラブルなく走っていたコードベースのクリーンアップで起きた。

Dev: "All right, I'm switching to bypassPermissions for this one. It's just renaming old API endpoint references. Nothing risky."

Claude Code: [Scanning files matching pattern /v1/legacy/ ... 47 files found]

Claude Code: [Updating endpoint references ...]

Claude Code: [Running post-rename cleanup.sh script ...]

Claude Code: [Deleted 3 files matching /v1/legacy/ in /deploy/config/prod/ ...]

Dev: "Wait. What was in /deploy/config/prod/?"

Claude Code: [Files contained environment-specific endpoint overrides for the production deployment. They have been removed.]

Dev: "That directory wasn't supposed to be in scope. I was working on /src/."

Claude Code: [The pattern /v1/legacy/ matched files in both /src/ and /deploy/config/prod/. In bypassPermissions mode, no confirmation was requested before the deletions.]

このミスを拾うはずだったプロンプトは、bypassPermissions をオンにした時点で無効になった。default または acceptEdits モードであれば、クリーンアップスクリプトは確認なしには走らず、ユーザーは本番の設定ファイルに削除が届く前に止められたはずだ。bypassPermissions では、パターンマッチは意図より広く働き、スクリプトと削除されるファイルのあいだにプロンプトはひとつも表示されなかった。

ゲートの正確な位置に注意。プロンプトを出したはずだったのはスクリプトの起動であって、rm の削除コマンド単体ではない。acceptEdits は、作業ディレクトリ内のパスに対する rm を含む、よく使うファイルシステムコマンドを自動承認する。もし Claude が削除を直接 rm コマンドとして出していたら、acceptEdits はそれを黙って通していた。default モードだけがそれらにプロンプトを出す。

> **What to Watch Out for（要注意なこと）**
> bypass モードは、予期していないものも含め、確認プロンプトをすべて黙らせる。ここでの失敗パターンは、チェックポイントのないモードで、エージェントが意図より広いファイル集合にマッチして走ることだ。bypassPermissions は、他のモードが保持する保護対象パスのガードもスキップする。だからリポジトリの状態も Claude 自身の設定も、自動プロンプトを失う。この隙間を埋めるには、モードを切り替える前に機密ディレクトリに deny ルールを設定しなければならない。安全網を失わずにプロンプトを減らしたいなら、完全な bypass ではなく、分類器で制御するモード（auto など）を使う。

## Screen 4 — Permission Modes & Human Gates — Checkpoint（チェックポイント）（4 min）

### Checkpoint 1: 設定ファイルを組み立て、人間のゲートを置く

やってみよう。信頼できるローカルでの決済モジュールのリファクタに向けて Claude Code を設定する。

リファクタではファイル編集を自動承認したいが、破壊的なシェルコマンドは絶対に走らせてはならない。また、`.env.production` はエージェントが読めてはならない。以下は settings.json の候補ピースだ。

#### Part 1: 正しい設定を組み立てる pieces を二つ選べ

二つ選ぶこと。

- ✓ **Piece A.** `{ "permissions": { "defaultMode": "default"} }`
- ✓ **Piece B.** `{ "permissions": { "defaultMode": "bypassPermissions" } }`
- ✓ **Piece C.** `{ "permissions": { "allow": ["Bash(npm run:*)"], "deny": ["Bash(rm:*)", "Bash(git push:*)"] } }`
- ✓ **Piece D.** `{ "permissions": { "deny": ["Read(.env.production)"] } }`
- ✓ **Piece E.** `{ "permissions": { "allow": ["Bash(*)", "Edit(*)"] } }`

#### Part 2

いまの設定はエージェントがファイルを自動で編集することを許している。リファクタの途中で、エージェントが複数の本番サービスが読むデプロイ設定ファイルへの変更を提案した。このひとつの操作に対して、人間のゲートはどこにあるべきか。最も適切な答えをひとつ選べ。

- a. Nowhere: 設定が既に編集を自動承認しているのだから、そのまま走らせる。
- b. 書き込みが実行される前に、デプロイ設定ファイルへの変更を人間が確認して承認する。そこでの誤った値は元に戻しにくく、ファイルの外のシステムに届くからだ。
- c. エージェントが一切止まらないよう、bypassPermissions を追加する。
- d. 書き込みの後、次回のプルリクエストのときにだけレビューする。

> **訳注（推定正解）:** Part 1 は C と D。C は npm run を許可しつつ rm と git push を拒否し、D は .env.production の読み込みを拒否する。A ではファイル編集の自動承認が実現せず、B・E は破壊的操作を無防備に通す。Part 2 は b。デプロイ設定は影響が大きく元に戻しにくいため、書き込み前の人間の確認が正しいゲートの位置。正式正解は原文の `Submit` で採点。

## Screen 5 — Durable Project Context — Teaching（継続するプロジェクトのコンテキスト）（20 min）

### CLAUDE.md、ルールファイル、hooks、サブエージェントで継続するプロジェクトのコンテキストを作る

前節までで、Claude Code がパーミッションモードと設定ファイルを通じて操作を挟み込む様子を見た。この設定の層が制御するのは、エージェントに許される作業の範囲だ。

これから扱うのはその上に乗るものだ。エージェントが何を知り、どう振る舞うかを設定する方法を学ぶ。一度セッションで定義したルールとプロジェクトのコンテキストが、次のセッションの最初にもまだ有効であるようにする。

#### CLAUDE.md: すべてのセッションに読み込まれるプロジェクトファイル

Claude Code がプロジェクトディレクトリで起動するたび、ルートにある CLAUDE.md というファイルを探して読む。その中身は、あなたからのメッセージが届く前に、プロンプトに追加される。つまり、CLAUDE.md に書いた規約も制約もコマンドも、すべて最初のプロンプトの段階で揃っている。毎回言い直す必要はない。

`/init` コマンドはコードベースをスキャンし、初期の CLAUDE.md を生成する。生成されたファイルは良い出発点だが、使う前に検証すべきだ。プロンプトの結果を左右するルールを手直しして書き込む。テストコマンド、フレームワークの規約、エージェントが触ってはいけないパス、デフォルトとは違うスタイルの判断だ。

サイズが主な失敗モードだ。新しい指示が来るたびに大きくなり続ける CLAUDE.md は、最も重要なルールを薄めてしまう。ファイルが大きいほどコンテキストウィンドウを消費し、個々の指示が読み込まれる全体に占める割合は小さくなる。すると、本当のミスを拾うはずのひとつのルールにエージェントが従う確率が下がる。CLAUDE.md は、振る舞いを変える制約だけを残し、それ以外はすべてオンデマンドで読み込まれる Skills に移す。

#### ルールファイル：ガイダンスを適用される場所に限定する

前節で、CLAUDE.md はすべてのセッションに読み込まれ、プロジェクト全体に当てはまる指示を置くべきだということを確かめた。では、コードベースの一部だけで意味を持つガイダンスはどう扱うか。そこで役立つのがルールファイルだ。指示を常に読み込むのではなく、関連する場所だけに適用できる。

CLAUDE.md は常にオンで、ルールファイルはそのベースラインの上に一段狭い層を足す。ルールファイルはプロジェクトの `.claude/rules/` ディレクトリに置き、YAML frontmatter の `paths` glob で特定のパスにスコープできる。こうしてスコープされたルールは、Claude Code がそのパターンにマッチするファイルを扱うときだけコンテキストに読み込まれる。これにより、コードベースの一部にだけルールを適用でき、他のコンテキストを汚さずに済む。

スコープは frontmatter から来るのであって、ファイルの置き場所からは来ない点に注意。ルールファイルは `.claude/rules/` のサブディレクトリ（例: `.claude/rules/database/`）に整理してもよいが、その構造は整理のためだけだ。`paths` フィールドのないルールファイルは、`.claude/rules/` のどこに置いても、起動時に無条件で読み込まれ、CLAUDE.md と同じ優先度を持つ。

実践上は、広いプロジェクトのメモリーと普遍的な制約は CLAUDE.md に置き、狭くパスに固有なガイダンスは `paths` でスコープしたルールファイルに置く。「データベーススキーマは絶対に変更しない」のような制約は、どこにでも適用されるから CLAUDE.md に置く。「データベースモジュール内のすべての SQL は明示的なトランザクション境界を持つ」のような制約は、次のように frontmatter を書いた `.claude/rules/database.md` に置く。

```
---
paths:
  - "src/db/**/*.sql"
---
```

こうしておけば、Claude がそれらのファイルを扱うときだけコンテキストに入る。

#### Hooks: ライフサイクルの固定ポイントで独自スクリプトを走らせる

hook を使うと、ツール呼び出しの実行前または実行後に、それを傍受して制御できる。CLAUDE.md に「ファイルを編集するたびに Prettier を走らせる」と書けば、エージェントはほとんどの場合それに従う。対して hook は、モデルがどう決めるかに依存せず、例外なく毎回それを実現する。hook はモデルの意思決定から独立して発火するからだ。

hooks は設定ファイルで定義し、`/hooks` コマンドで構成する。各 hook は、ライフサイクルイベント、それを特定のツール種別に限定するオプションの matcher、そしてイベントが発火したときに走るコマンドに紐づく。多くのガードレールと自動化のユースケースで中心になるイベントは次のとおりだ。

- **PreToolUse:** ツール呼び出しが実行される前に走る。一番最初に走るため、PreToolUse hook はツール呼び出しを審査し、終了コード 2 で抜けてそれをブロックできる。理由を stderr に書けば、エージェントが見るフィードバックになる。これが、CLAUDE.md の指示に頼るのではなく、設定の層でアクセス制御を強制する方法だ。
- **PostToolUse:** ツール呼び出しが完了した後に走る。呼び出しは既に起きているためブロックはできず、自動の副作用を置くのに適している。編集後のコードフォーマッタの実行、ファイル変更後のテストのトリガー、監査証跡のための操作のログ記録などだ。
- **UserPromptSubmit:** あなたがプロンプトを送信したとき、モデルがそれを処理する前に走る。作業が始まる前にコンテキストを注入したり、リクエストを検証したりするのに使う。
- **Stop:** モデルが応答を終えたときに走る。通知、クリーンアップタスク、監査ログの commit など、ターンの終わりに属する後続処理に使う。
- **Notification:** Claude Code が通知を送るとき、つまり Claude がツールの使用許可を必要とするときや、Claude Code が60秒間アイドルだった後に走る。それらのシグナルを外部チャネルやログシステムに流すのに使う。
- **SessionStart:** セッションが開始または再開されたときに走る。状態の初期化、環境変数の検証、必要なサービスへの到達確認をエージェントが作業を始める前に行うのに使う。
- **SessionEnd:** セッションが終了したときに走る。後片付け、最終的な監査の書き込み、セッションが閉じたことの通知に使う。

PreToolUse イベントで本番設定パスへの編集をブロックする hook は、パーミッションモードに関わらず、すべてのセッションのすべてのツール呼び出しでその制約を強制する。これがガードレールと規約の違いだ。

#### サブエージェント：隔離されたコンテキストに作業を委譲する

サブエージェントは Claude Code がタスクを委譲できる専門のアシスタントで、各アシスタントは自分自身の独立したコンテキストでタスクを走らせ、出力だけを返す。メインの会話履歴も、コンテキストに蓄積したファイルも、現在のセッションの状態も引き継がない。タスクをサブエージェントに送ると、白紙の状態から始めて作業を行い、結果を返す。

組み込みのサブエージェントは起動時に何を読み込むかが違う。この違いが、プロジェクトルールがどう適用されるかを決める。正確な一覧は Claude Code のドキュメントで常に確認すること。構成は時とともに増えているが、プロジェクトルールに影響する特定の分割はバージョンを通じて変わらない。Explore と Plan の組み込みサブエージェントは、調査を速く安く保つために CLAUDE.md と git status をスキップする。これらは速度に最適化されているので、CLAUDE.md で定義されたプロジェクトレベルのルールやリポジトリの状態は、実行時にはコンテキストにない。general-purpose サブエージェントは両方を読み込む。もし Explore または Plan にタスクを委譲し、CLAUDE.md のルールが当てはまるべきだったのに当てはまらなかったとすれば、それはコンテキストが読み込まれていないからだ。プロジェクトの制約を尊重しなければならないタスクでは、general-purpose サブエージェントか、必要なルールを明示的に読み込むカスタムサブエージェントを使う。

カスタムサブエージェントも、自動的にはスキルを見ない。`.claude/agents` にカスタムサブエージェントを定義し、それが特定のスキルを必要とするなら、エージェントの frontmatter にそのスキルを明示的に書かなければならない。組み込みのエージェントはスキルを事前読み込みしない。組み込みのエージェントがスキルに基づく振る舞いを必要とするなら、正しい道は、そのスキルを設定に書いたカスタムサブエージェントを作ることだ。

下のマップは、各仕組みの名前、読み込むもの、走るタイミング、コンテキストのコスト、そこに置くべきものを示す。それぞれがコンテキストのコストと適用の信頼性のあいだに別のトレードオフを持つので、どの仕組みが特定のプロジェクト知識を担うべきかを判断するのに使う。各カードをめくって全体像を見てほしい。

- **仕組み：CLAUDE.md**
  - 読み込むもの: セッション開始時にファイル全体がコンテキストの先頭に追加される。
  - 走るタイミング: すべてのセッションで、無条件に。
  - コンテキストコスト: セッションごとに永続。サイズで薄まる。
  - ここに置くもの: 普遍的なプロジェクトの制約、コマンド、フレームワークの判断。

- **仕組み：ルールファイル**
  - 読み込むもの: ファイルの中身。YAML frontmatter の `paths` glob でスコープする。`paths` がなければ CLAUDE.md と同じように読み込まれる。
  - 走るタイミング: Claude がルールの paths パターンにマッチするファイルを読むとき。パス指定のないルールはセッション開始時に読み込まれる。
  - コンテキストコスト: パススコープならトリガーされたときだけ追加。スコープなしなら CLAUDE.md と同じ永続コスト。
  - ここに置くもの: ほかの場所ではノイズになる、パスに固有なガイダンス。

- **仕組み：hook**
  - 読み込むもの: ライフサイクルイベントで独自スクリプトを走らせる。コンテキストには何も追加しない。
  - 走るタイミング: 構成したイベント（PreToolUse、PostToolUse など）のとき。
  - コンテキストコスト: 最小。Claude に戻すならスクリプトの出力だけ。
  - ここに置くもの: 強制ガードレール、自動の副作用、監査ログ。

- **仕組み：サブエージェント**
  - 読み込むもの: タスクのコンテキストのみ。メインセッションからは隔離される。
  - 走るタイミング: メインセッションが委譲タスクを送るとき。
  - コンテキストコスト: サマリーを返し、タスク履歴全体は返さない。
  - ここに置くもの: 探索、調査、および出力がメインコンテキストを膨らませてしまうタスク。分割して並列化できるタスクにも有用。

> **Handles well（うまく扱える）**
> 多くのセッションを通じて戻ってくるプロジェクト。安定したルール群、ディレクトリごとの違い、無条件のガードレールが、設定の手間を返してくれる。
> **Use a different approach（別の手法が合う）**
> 二度と戻らないひとつ限りのタスク。見知らぬコードベースを手早く探索するだけなら、設定のオーバーヘッドに見合わない。

## Screen 6 — Durable Project Context — Watch Out（継続するプロジェクトのコンテキスト・要注意）（4 min）

### ルールが効かなくなるまで膨らみ続けた CLAUDE.md

> **Setup**
> 新しいルールを追加するたび、どれも意味があるように思えた。個々には妥当で、ファイルがその置き場所としてふさわしいように思えた。しかし数週間で、ファイルは800行を超えた。

#### トレース

次のセッションログの抜粋は、チームが二か月にわたって追加を重ねた CLAUDE.md を持つプロジェクトのものだ。

```
Session context window loaded:
CLAUDE.md: 847 lines
Contents include: framework preferences (lines 1–40), testing conventions (lines 41–90), style guide (lines 91–210), dependency rules (lines 211–320), path restrictions (lines 321–360), historical decisions log (lines 361–700), archived notes (lines 701–847)

Claude Code action: [Editing auth/service.ts ...]
Claude Code action: [Reading /legacy/tokens/store.ts for reference ...]
Claude Code action: [Editing /legacy/tokens/store.ts to update token interface ...]
Path restriction from CLAUDE.md (line 347): "Do not modify files in /legacy/tokens/."
User: "You just edited /legacy/tokens/store.ts. I said not to touch that directory."
Claude Code: "I apologize. The /legacy/tokens/ restriction is in the project instructions. The edit was made in the course of updating the token interface. I did not flag the conflict before proceeding."
```

ルールはファイルの中にあり、エージェントもアクセスできていた。しかし失敗の原因は、ルールが薄められたことだ。ほかの846行が、重要なひとつの指示の実際の重みを下げていた。履歴の決定記録やアーカイブ済みのメモはどこかに記録されるべきだが、CLAUDE.md に置くべきではなかった。

> **What to Watch Out for（要注意なこと）**
> CLAUDE.md は現在のセッションの振る舞いを変える、働くルールの集合であって、膨らみ続ける追記ログではない。一行追加するたびに、ほかのすべての行の重みが下がる。ルールがパスに固有なら、それはルールファイルに属する。ルールが履歴のコンテキストなら、エージェントがオンデマンドで読む別の参照文書に属する。CLAUDE.md が数百行を超えたら、監査する。本当にセッションに必須なルールを残し、残りを移動させる。絶対に薄めてはいけないルールは、hook への最短経路にしておく。

## Screen 8 — Packaging Workflows — Teaching（ワークフローのパッケージ化）（8 min）

### ワークフローをプラグインとしてパッケージ化する：skills、カスタムコマンド、marketplace install

これまでに、Claude Code に継続するコンテキストを与え、振る舞いを強制する仕組みを扱った。常にオンのプロジェクトメモリーとしての CLAUDE.md、スコープしたガイダンスとしてのルールファイル、確定的なガードレールとしての hooks、隔離されたタスク委譲としてのサブエージェントだ。

これらの仕組みは `.claude` ディレクトリにあり、プロジェクトとともにバージョン管理される。次の問いは、そのセットアップをどうパッケージすれば、チームメイトが手動で設定を繰り返さずに、ひとつのステップでインストールできるかだ。

#### Skills は、エージェントがオンデマンドで読み込む再利用可能なワークフロー

skill は、`.claude/skills` に置かれた持ち運び可能な Markdown ファイル（SKILL.md）だ。frontmatter が skill を識別していつ適用されるかを述べ、本体が手順を保持する。同じ skill を Claude Code で走らせ、Messages API 経由で呼び出し、Agent SDK で読み込ませることができる。三つの間で変わるのはファイルそのものではなく、skill がどこで走り、どう読み込まれ、何に触れることが許されるかだ。Claude Code でしか skill を見たことのない開発者は、API では成り立たない仮定をしがちだ。そのため本節は、その違いを整理する。

#### 各環境で skill がどう読み込まれ、走るか

各タブを選ぶと、skill がどう読み込まれるか、手順がどこで走るか、何を知っておくべきかが見える。

**Claude Code**
- **skill の読み込み方:** ファイルシステム上の `.claude/skills` から発見される。description のマッチで読み込まれるか、名前で呼ぶと読み込まれる。
- **手順が走る場所:** ターミナルのセッション内で、ローカルのファイルに対して、有効なパーミッションモードと deny ルールの下で。
- **知っておくべきこと:** ファイルシステムベースであり、設定の層が制御する。

**Messages API**
- **skill の読み込み方:** リクエストとともに送られ、アプリケーションの環境ではなく、コード実行コンテナの中で走る。code-execution と skills の beta ヘッダーが必要。
- **手順が走る場所:** Anthropic のコード実行コンテナの中であり、自分のマシン上ではない。skill のファイルシステムとツールへのアクセスは、そのコンテナが提供するものに限定される。
- **知っておくべきこと:** ローカルのファイルやローカルのツールを前提とする skill は、ここでは同じようには振る舞わない。それらのファイルがある場所で走らないからだ。

**Agent SDK**
- **skill の読み込み方:** SDK が走らせるエージェントによって読み込まれる。ただし、ファイルシステムの設定（CLAUDE.md、skills）を読み込むかは `settingSources` 構成で制御される。デフォルトに頼らないこと。常に意図するソースへ明示的に設定し、ビルド時に Agent SDK のリファレンスで現在のデフォルトの振る舞いを確認する。TypeScript では `settingSources`、Python では `setting_sources` で設定する。
- **手順が走る場所:** ファイルシステムのソースを読み込むように指示すれば、SDK が走らせるプロセス、すなわち、あなたの環境内で走る。
- **知っておくべきこと:** よくある落とし穴は、Claude Code で動いた skill が SDK の下では何もしないこと。`settingSources` が設定されたことがなく、skill が読み込まれなかったからだ。

**Claude Managed Agents**
- **skill の読み込み方:** モデル、システムプロンプト、ツール、MCP サーバー、skills を名指しする API リソースとして一度定義する。エージェントの実行時に Anthropic がサーバー側で skill を読み込むので、こちらにはファイルシステムの発見ステップはない。
- **手順が走る場所:** Anthropic が用意して走らせるサンドボックスの中であり、あなたの環境ではない。アプリケーションはユーザーイベントを送り、ストリームされた結果を読み戻す。skill はそのマネージドサンドボックスが提供するものにアクセスし、ローカルファイルにはアクセスしない。
- **知っておくべきこと:** 現在はパブリックベータで、`managed-agents-2026-04-01` beta ヘッダーが必要。セッションはサーバー側に保存されるため、Managed Agents は現在、Zero Data Retention や HIPAA BAA の対象外。skill はセッション時ではなくエージェントリソースの定義時に紐付ける。利用可能な skills を変えるにはエージェント定義を更新する。

#### 三つの移植性のルール

- **description をマッチの基準として書く。** モデルはリクエストと description を比べて skill を読み込む。いつ適用されるかを特定する description はどのランタイムでも働くが、曖昧なものはどれでも読み込まれない。
- **skill の本体で、ローカルのファイルシステムやローカルのツールが存在すると仮定しない。** ローカルのコマンドをシェル経由で実行する skill は Claude Code では動くが、コマンドのないコンテナで走る Messages API では壊れる。skill の手順は、ランタイムが保証するものに限定するか、依存関係を明記する。
- **サブエージェントはスキルを引き継がないことを忘れない。** これは Module 2 でもそうで、ここでも同じだ。サブエージェントは白紙から始まるので、親が依存する skill は、サブエージェントをサポートするすべてのランタイムで、明示的にリストしなければならない。

実践上の教訓は、skill を一度書いて再利用できるが、複数のターミナルをまたいで使うためには、そうなるよう意識して設計しなければならないということだ。明確な description にスコープし、ローカル環境の仮定を含まない skill はランタイムをまたいで綺麗に移植する。特定のローカル環境を前提とするものはしない。

> **Handles well / Adds complexity / Use a different approach**
> - **うまく扱える:** 特定タスクの手順を一度書き、インタラクティブなターミナル、API の統合、ヘッドレスな SDK ジョブで再利用する。
> - **複雑さが増す:** 各ランタイムは skill の読み込みとサンドボックス化を別の方法で行うので、API の beta ヘッダーと SDK の `settingSources` を考慮しなければならない。
> - **別の手法が合う:** プロジェクトのすべてのセッションに適用しなければならない指示には、CLAUDE.md が依然として正しい道具。skills はオンデマンドで持ち運べる手順向けだ。

#### ワークフローに明示的な入り口を与える

カスタムコマンドは、定義済みの手順へのショートカットだ。現在の Claude Code では、明示的にも自動的にも起動できる形式として skills が推奨される。skill を `/skill-name` で直接呼ぶか、関連するときに Claude が自動で読み込む。古い `.claude/commands/` ディレクトリの形式もまだ動くが、レガシーな手順だ。明示的に呼んだときだけ走るワークフローを作りたい場合は、frontmatter に `disable-model-invocation: true` を付けた skills を使う。

プラグインのコマンドは自動的に名前空間化される。プラグイン名がプレフィックス（接頭辞）になり、payments というプラグインの run-tests コマンドは `/payments:run-tests` として呼ばれる。これが、二つのプラグインがどちらも run-tests コマンドを同梱しても衝突しない理由だ。作者はプラグイン名をインターフェースの一部として扱うべきだ。同梱するすべてのコマンドの前に付くし、プラグインを改名すればすべて改名されるからだ。

#### セットアップをインストール可能にするパッケージ層

プラグインは、skills、hooks、サブエージェント、MCP サーバーをひとつのインストール可能な単位にまとめる。プラグインは marketplace を通じてパッケージ化し配布できる。marketplace とは、誰かが作って共有したプラグインのカタログだ。公式の Anthropic marketplace は Claude Code の起動時に自動的に利用できる。`/plugin marketplace add <owner/repo>` のようなコマンドで、GitHub リポジトリでホストされるサードパーティの marketplace を追加できる。チームメイトは、同じセットアップを得るために単純なインストールコマンドをひとつ走らせるだけでよい。プラグインは、手動設定の手順のページを、バージョン管理され監査可能なインストールに置き換える。プラグインはコンポーネントを次のように配置する。

- skills は skills ディレクトリに入る。
- hooks、サブエージェント、settings はそれぞれの場所に入る。

プラグインマニフェストがバンドルを記述し、install コマンドがそれを対象のインストール先に組み込む。プラグインは個人単位でも、エンタープライズ全体でもダウンロードできる。

エンタープライズ管理者は、managed settings を通じてプラグインを組織全体にデプロイできる。managed marketplace の許可リストが、ユーザーが追加を許される marketplace ソースを制限する。組織がプラグインの由来を統制できるわけだ。許可リストはユーザーが追加できるものを制限するだけで、marketplace を自動的に登録はしない。ユーザーに add コマンドを実行させずに、全ユーザーへ marketplace を展開したい場合は、許可リストの設定に managed settings の `extraKnownMarketplaces` を組み合わせる。優先順位はデプロイのスコープから来る。managed settings は設定の階層においてユーザー設定とプロジェクト設定より上にあるので、managed スコープでデプロイされたプラグインは優先され、ユーザーやプロジェクトファイルで上書きできない。正確な設定名はリファレンスの層を確認すること。

#### パッケージ化の判断表

下の表は、各層が何か、誰向けか、いつ選ぶべきかを示す。

| 層 | 何か | 誰向けか | いつ選ぶか |
|---|---|---|---|
| Skill | `.claude/skills` の Markdown ファイル。description がタスクにマッチするか、名前で呼ばれたときに読み込まれる。 | インタラクティブに Claude Code を使う、個人またはチームの開発者。 | 特定タスクの手順を、必要になるまでコンテキストに出しておきたくないときに選ぶ。例えば PR レビューや、作業が必要になったときだけ読み込まれるデプロイチェックリスト。 |
| カスタムコマンド | 明示的に呼ぶと定義済みの手順を走らせる、名前付きのショートカット。 | 高頻度の手順に予測可能で明示的な入り口を欲する開発者。 | 手順に明確な名前があり、description のマッチに頼らず直接引き起こしたいときに選ぶ。 |
| Plugin | skills、hooks、サブエージェント、MCP サーバーの、バージョン管理されたバンドル。marketplace 経由で配布される。 | 共有の、バージョン管理されたセットアップをワンステップでインストールしたいチーム。 | 動くセットアップが今ひとつのマシンにあり、チームで共有し、バージョンを揃えたいときに選ぶ。 |

> **Cost · Complexity · Risk（コスト・複雑さ・リスク）**
> **Cost:** skills は起動時にコンテキストコストを加えるが、プラグインはインストールと保守のオーバーヘッドを加える。問うべきは、セットアップのコストを一度だけ払いたいか（プラグインのインストールのように）、それとも開発者各自が同じ手動手順を繰り返すたびに何度も払いたいかだ。
> **Complexity:** skill に絶対パスを書き込んだプラグインは、作者の環境では正しくインストールされ、ほかの全員では失敗する。skill や hook のコマンドにハードコードされた、パスや環境の前提が、マシンをまたいで最も壊れやすいからだ。
> **Risk:** プラグインは、バンドルしたコンポーネントをすべてのインストール先に持ち込む。作者がローカルで頼っていた deny ルールや hook は、バンドルの一部として明示的に書かれない限り含まれない。skills や hooks が、バンドルに含まれないガードレールに結びついているなら、その保護はチームメイトのマシンには届かない。

## Screen 10 — Packaging Workflows — Watch Out（ワークフローのパッケージ化・要注意）（3 min）

### 自分のマシンではインストールされ、ほかの全員では失敗したプラグイン

> **Setup**
> プラグインが綺麗にインストールされたということは、パッケージが正しく組み立てられたことを示す。しかしプラグインが有効に動くことを示すわけではない。インストールと実行は別物だからだ。インストールはファイルを所定の場所にコピーする。実行は、それらのファイルが指すパスと変数を、走っているマシンに対して解決する。プラグインの作者が自分のマシンのレイアウトを skill にハードコードすると、インストールはどこでも成功するが、実行は作者自身のセットアップ以外のどこでも失敗する。この隙間は、作者には見えない要因から生じる。

#### 何が起きたか

ある開発者がデプロイワークフローの skill を構築し、プラグインとしてパッケージ化し、ローカルでテストした。ローカルのテストは通り、プラグインは社内の marketplace を通じてチームに出た。チームメイト全員のインストールは成功したが、チームメイトが skill を走らせた途端、失敗した。

根本原因は skill の SKILL.md の中にあった。`/Users/alexmorgan/projects/deploy-utils/validate.sh` を指すコマンドだ。

そのディレクトリは作者のマシンにしか存在しなかった。skill は作者のホームディレクトリへの絶対パスを持ち歩き、チームメイトの実行はどれも、自分のシステムにも skill にもないファイルを探すことになった。

同じプラグインの二つ目の skill は、環境変数 `DEPLOY_TOKEN` に依存していた。作者は自分のシェルプロファイルにそれを設定していたが、プラグインの README には一切書かれていなかった。三人のチームメイトが二時間デバッグに費やして、二つ目の失敗を欠けていた変数まで辿った。

プラグインは作者のマシンをチームのマシンと誤って扱い、それが破壊を招いた。上の二つの失敗はどちらも同じ根本原因と同じ絶対パスを持つ。それは SKILL.md の中にプレーンテキストとしてあり、ファイルを読むレビュアーが捕捉できる。環境変数への依存は危険になり得る。パッケージの中にその依存関係を示すものが何もないため、skill はその変数を必要とするステップに達するまではうまく走り、そこで初めて失敗する。これが、修正に三人で二時間を要した理由だ。

> **What to Watch Out for（要注意なこと）**
> skill、hook のコマンド、プラグインのコンポーネントの中のパス参照は、すべてプロジェクトルートからの相対か、ベースパスに環境変数を使うものでなければならない。プロジェクトに置いたスクリプトは `$CLAUDE_PROJECT_DIR` で、プラグイン自身にバンドルしたスクリプトは `${CLAUDE_PLUGIN_ROOT}` で参照する。そうすれば、誰のマシンで走っても、セッションがどのディレクトリで始まっても、パスが正しく解決する。プラグインが依存するスクリプト、設定ファイル、その他のアセットは、プラグインの中にバンドルするか、共有のプロジェクトの場所に置き、インストール後にどのチームメイトも同じファイルにアクセスできるようにすること。プラグインが必要とする環境変数をすべて文書化し、インストール時に検証する。そうすれば、欠損は実行の途中ではなく、インストールの時点で発覚する。配布前にクリーンなマシンでインストールをテストすること。ビルドマシンが隠している問題をこれが捕捉する。

## Screen 12 — MCP Servers — Teaching（MCP サーバー）（21 min）

### MCP サーバーの構築と設定：トランスポート、スコープ、GitHub サーバー

これまでの節では、プラグインを、skills、hooks、サブエージェント、MCP サーバーをひとつのインストール可能な単位にまとめるパッケージ層として紹介した。

この節では、MCP サーバーのバンドルとは何か、どう構築するかをさらに説明する。MCP サーバーは、コードベースの外から Claude にツールを提供する層だ。MCP サーバーを構築するとき、最初の決断のひとつは、適切なトランスポート方式を定め、サーバーのスコープを定義することだ。

#### MCP サーバーとは何か。ツールを直接配線するのと何が違うのか

アプリケーションにツールを直接配線するとき、ツールのスキーマと機能の定義はあなたの責任で、どちらもそのアプリケーションのコードの中にある。三つの異なるアプリケーションが同じ外部サービスを必要とするなら、それぞれが独自の統合を保守する。Model Context Protocol（MCP）は、ツールの定義を個々のアプリケーションから切り離し、サーバーと呼ばれるプロセスに変える。

MCP サーバーは、MCP クライアントが使えるツール、リソース、プロンプトを提供するプロセスだ。Claude Code は組み込みの MCP クライアントを持つ。MCP サーバーに接続すると、Claude Code はそのサーバーが提供するツールを発見し、セッションの中で呼び出せる。MCP サーバーを使えば、機能を一度構築するだけで、そこに接続するすべての MCP クライアントが、統合を再実装することなくアクセスできる。

#### MCP サーバーはリソースとプロンプトも提供する

MCP サーバーはツール、リソース、プロンプトを提供する。ツールは既に学んだ。モデルが呼べるアクションだ。残る二つは、ツール呼び出しでは必要なものが得られないケースをカバーする。

リソースは、サーバーが読み込み専用のデータとして公開するもので、モデルがツールを呼んで取得するのではなく、クライアントが直接フェッチしてコンテキストに置ける。クライアントはアドレスでリソースを要求し、サーバーがデータを返す。リソースには二つの形がある。直接リソースはパラメータを取らないデータに固定のアドレスを持つ。利用可能な文書の一覧などだ。テンプレート化されたリソースは、アドレスの中にパラメータを置く。文書識別子を取る文書アドレスなどだ。ターンの最初から既知のデータをコンテキストに入れておきたいとき、リソースを選ぶ。ツール呼び出しで取りに行くよりも、直接取り込むほうが安く、予測可能なときにこれが欲しい。リソースのサポートは MCP クライアントによって異なる。このパターンに頼る前に、クライアントがコンテキストへリソースを注入する仕組みを持つことを確認すること。

プロンプトは、サーバーが公開する事前記述済みの指示テンプレートで、ユーザー各自に書かせるのではなく、検証済みのプロンプトを名前で呼べるようにする。ユーザーはすでに自分の言葉で大半のタスクをモデルに頼める。だからプロンプトが有用なのは、特定の言葉遣いが必要なときだ。注意深く作られた指示が、ユーザーが打つものよりも明確に良い結果を出すタスクで、かつどのクライアントにも同じ品質を届けたいとき。指示をサーバーにパッケージしておけば、プロンプトは一か所で保守され、サーバーが繋がるどこでも再利用される。

#### トランスポート：Claude Code がサーバーとどう話すか

トランスポートは、MCP クライアントと MCP サーバーのあいだの通信路だ。適切なトランスポートは、サーバーがどこで走るかで決まる。各タブを選ぶと、それが何か、いつ使うかが見える。

**stdio**
- stdio はサーバーを、クライアントと同じマシンのローカルプロセスとして走らせる。クライアントはサーバーをサブプロセスとして起動し、標準入出力で通信する。ローカルのツール、個人のスクリプト、自分のマシンで走らせる開発サーバーには、これが正しい選択だ。チームで共有したいサーバーや、リモートでホストしたいサーバーには使えない。

**HTTP**
- HTTP は、ローカルで走らないサーバーに推奨されるトランスポートだ。標準的な HTTP 接続で通信し、別のマシンでホストされるサーバーをサポートする。HTTP サーバーを登録するとき、URL を指定し、クライアントがネットワーク経由で接続する。共有のチームサーバーやホストされる統合は HTTP を使う。

**SSE**
- SSE（Server-Sent Events）は、現在の HTTP トランスポートより前の、古いトランスポート手段だ。HTTP トランスポートに取って代わられ、新しいサーバーにはもはや推奨されない。既存の設定や文書で SSE に遭遇した場合は、現在の推奨ではなく、レガシーな選択肢として扱うこと。

#### コンテキストコスト

接続した各 MCP サーバーはツール定義を提供し、それを前もって読み込めばコンテキストウィンドウを占める。デフォルトでは、Claude Code はこれらの定義を前もって読み込まず遅延させ、タスクが求めるときにだけ、検索のステップで関連ツールを発見して読み込む。呼び出されたツールだけがコンテキストに入る。

オプトインのモードは、ツール定義がコンテキストウィンドウのほぼ10%に収まるなら前もって読み込み、その上限を超えるときだけ遅延させる。いずれにせよ、必要なサーバーだけを接続することが各リクエストを引き締まったまま保つ。接続したサーバーが増えると、モデルが考慮しなければならない定義のプールにそれぞれが加わるからだ。

#### プロンプトキャッシュ：再利用可能なリクエストに一度だけ払う

いま見た MCP サーバーのコンテキストコスト問題には、コストとウィンドウの両面がある。すべてのリクエストは入力をゼロから再処理し、前回のリクエストと同じ部分も含めて、再処理に毎回払う。プロンプトキャッシュは、同じ安定した内容に二度払うのを止められる。

キャッシュは、リクエストの安定したプレフィックスに対して行われた処理を保存する。後続のリクエストは同じ処理を再利用でき、同じトークンを再処理しなくて済む。最初のリクエストがプレフィックスをキャッシュに書き込み、後続のリクエストは、同じ位置まで同一の内容を、コストのごく一部で送る。内容は完全に一致しなければならない。キャッシュブレークポイントより前に一文字でも変わると、キャッシュは無効になり、書き直しが強制される。だからキャッシュに最も適しているのは、ほとんど変わらないリクエストの部分だ。長いシステムプロンプト、大きなツール定義の集合、あるいは複数の質問を投げる参照文書などだ。

キャッシュは、キャッシュブレークポイントを設定することで有効になる。キャッシュ全体を一括で有効にする設定はない。Messages API では、キャッシュしたい最後のブロックに `cache_control` フィールドを `type: ephemeral` で追加する。これで、そのブロックまでを含めてキャッシュする。ブレークポイントは最大四つ置ける。リクエストはツール、システムプロンプト、メッセージの固定順で処理される。だからツールの後にブレークポイントを置けば、ツール定義をキャッシュしつつ、メッセージは動的に保てる。

キャッシュには時間制限がある。デフォルトのキャッシュ寿命は最後の読み取りから5分。ブレークポイントに `ttl: 1h` を設定すれば、オプトインで1時間の寿命が使える。5分のデフォルトは、リクエストが数分おきに届くやり取り型のモデルに合う。読み取りごとにタイマーがリセットされるからだ。1時間の選択肢は、リクエストのあいだに長い間が空くワークロードに合う。ステップのあいだに休むエージェントなどで、5分のウィンドウでは次のリクエストまでに切れてしまう場合だ。ウィンドウが次のリクエスト前に切れると、読み取りの恩恵なしに、再び書き込みコストを払うことになる。キャッシュは最小トークン閾値（現在の大半のモデルで1,024トークン）を超えた場合にのみ適用されることに注意。ブレークポイントを設定しても、短いプロンプトはキャッシュされない。

#### 検索拡張生成：Claude はリクエストが必要とする知識だけをどう取り込むか

MCP サーバーで見たコンテキストコストの問題は、大量の参照資料が生むのと同じ問題だ。モデルはすべてのリクエストでコンテキストウィンドウの中身をすべて読む。だから前もって読み込む文書が多いほど、コンテキストは使われ、作業の余地は減る。通常 RAG と略される検索拡張生成が、これを解くパターンだ。すべての文書をコンテキストに読み込む代わりに、システムは資料をコンテキストウィンドウの外に置き、現在のリクエストに最も関連する部分を見つけ、リクエスト時にその部分だけをモデルに渡す。モデルは図書館全体からではなく、取り出されたスライスから回答を生成する。

RAG には二つの形がある。

古典的 RAG は重い準備作業を前もって済ませる。誰かが質問する前に、元資料をチャンクに分け、各チャンクを、意味を数学的に捉える数値の集合（埋め込みと呼ばれる）に変換する。その数値をデータベースに格納する。ユーザーが質問したとき、システムは質問を同じ種類の数値に変換し、最も似た数値を持つチャンクを探す。図書館員が、開館前にすべての本を読んで章ごとの正確な要約カードを書いておくようなものだ。あなたが質問を持って来たとき、正しいカードを即座に引ける。

エージェント的検索は、前もったインデックス作りを完全に飛ばす。事前構築されたデータベースはない。代わりに、モデルはあなたが尋ねた瞬間に必要なものを割り出し、それから取りに行く。ライブのソースを検索し、オンデマンドで文書を読み、タスクが展開するに従って結果を取り込む。質問されたとき、用意されたカードを参照するのではなく、自ら答えを探しに行く研究者のようなものだ。

エージェント的検索という名前を知らなくても、すでに目にしたことがあるかもしれない。Claude Code では、多くの外部ツール（MCP サーバー）に接続しているとき、Claude はすべてのツール定義を前もって読み込まない。一度に保持するには多すぎるからだ。代わりに、現在のタスクに必要なツールだけを発見して読み込む。Claude.ai Projects は、アップロードされた文書にも同じことをする。プロジェクトのナレッジベースがアクティブウィンドウに収まらなくなると、すべてを読み込むのではなく、各質問に最も関連する文書のセクションだけを取り出して提示する。

二つのアプローチは、根本的には同じことをする。関連するスライスを見つけ、そこから生成する。違いはタイミングだ。古典的 RAG は事前に作ったインデックスに対するマッチでスライスを見つける。エージェント的検索は必要な瞬間に検索で見つける。

検索に手を伸ばす前に、理解しておくべき性質が二つある。

- **スケールする。** 元資料が大きくなっても、各リクエストのコストは平坦に保たれる。モデルが受け取るのはその質問に関連するスライスだけで、図書館全体ではないからだ。ナレッジベースが何千もの文書に増えても、ひとつの質問が取り戻すテキスト量はほぼ同じまま。検索がスケールするのはこれのおかげだ。元資料は増え続けられても、リクエストのサイズは大きくならない。
- **見つかるもの次第。** モデルは受け取ったスライスの上で推論する。検索のステップが必要な文書を見落とすと、モデルはそれを決して見ない。だから資料の整理の仕方が重要になる。曖昧な名前のファイル（`notes_final_v3.pdf`）は、説明的な名前のファイル（`Q3 refund policy, updated August 2024`）より検索で拾われにくい。関連するファイルを一緒にまとめておくのも役立つ。良い検索は、よく整理された元資料から始まる。

#### 構成スコープ：誰がサーバーを読み込むか

スコープは、どのユーザーとプロジェクトがサーバーを読み込むかを決める。各スコープは異なる構成の場所に対応する。

- **Local スコープ** は、サーバー構成を `~/.claude.json` に置く。登録されるのは現在のプロジェクトのパスの下で、現在作業中のプロジェクトにだけ適用され、チームメイトとは共有されない。リポジトリに commit する準備ができていない、特定のプロジェクトのコンテキストに結びついたサーバーや、ひとつのプロジェクトでしか意味を持たないツールに適したスコープだ。
- **User スコープ** はサーバー構成を個人の Claude 設定に置き、すべてのプロジェクトで使えるようにする。ただしまだ個人用で、チームメイトには見えず、リポジトリにも書き込まれない。どのコードベースを扱っていても頼るローカルのデータベースツールやスクリプトなど、すべてのプロジェクトで使う個人のユーティリティに適したスコープだ。
- **Project スコープ** はサーバー構成をリポジトリルートの `.mcp.json` に書き込む。そのファイルをバージョン管理に commit すると、リポジトリを clone する全員が同じサーバーを自動的に得る。チーム全体がアクセスできるサーバーに適したスコープだ。構成がコードと一緒に移動するからだ。留意すべき点がひとつ。project スコープのサーバーは、各チームメイトのマシンから走る。stdio サーバーの場合、commit された構成は起動コマンドを格納し、各 clone が独自のローカルサブプロセスを立ち上げる。だからチームメイト各自が、ローカルにランタイム（npx で起動するサーバー用の Node など）をインストールしていなければならない。
- **Enterprise スコープ** は管理者が制御する集中管理された構成でデプロイする。管理者は、個別の設定手順なしに、組織の全ユーザーへサーバーを配れる。共有の社内サービス、セキュリティツール、組織全体に存在しなければならず、個々の開発者の設定に任せられないサーバーに適したスコープだ。

#### サーバー全体ではなく、ひとつの MCP ツールを対象にするパーミッションルール

サーバーに接続すると、そのツールリスト全体が公開される。しかし、エージェントがそれらのツールすべてに確認なしで届いてほしいことは稀だ。パーミッションモードの節のパーミッション層は MCP ツールにも及び、ルールはサーバー全体ではなく、個々のツールを名指しできる。

MCP ツールはパーミッションルールの中で、サーバー名とツール名で `mcp__server__tool` と識別される。`mcp__github__create_issue` への allow ルールは、GitHub サーバーの他のすべてのツールにはまだプロンプトを出したまま、そのひとつのツールだけをプロンプトなしで走らせる。書き込み可能なツールへの deny ルールはそれをブロックしつつ、同じサーバーの読み込み専用ツールは使い続けられる。これが、広いサーバーに繋ぎつつ、エージェントをその能力の狭いスライスの中に留める方法だ。ひとつのツールへの deny は、サーバーへの allow より優先する。

API MCP コネクターももう一つの有用な制御だ。API MCP コネクター経由でサーバーに届く場合、`mcp_toolset` オブジェクトでツールごとに `enabled` フラグを設定できる。このフラグで、サーバーを登録しつつ、モデルに見せたい特定のツールだけを公開できる。パーミッションルールが公開されたツールを走らせてよいかを決め、`enabled` フラグはモデルがそのツールを見るかどうかを決める。前者はガバナンスの制御で、後者はコンテキストコストとスコープの制御だ。これらは一緒に使われることが多い。公開前に、正確なルールの構文とコネクターの beta ヘッダーを文書で常に確認すること。

#### GitHub MCP サーバー：具象例におけるトランスポート、スコープ、認証

GitHub MCP サーバーは、GitHub が保守するリモートサーバーで、プルリクエストのレビュー、Issue の作成、コード検索など、リポジトリ管理向けのツールを提供する。接続の手順を辿ることで、トランスポート、スコープ、認証が、他者の保守するサーバーの中でどう連携するかが見える。

GitHub サーバーは GitHub がリモートでホストするため、HTTP トランスポートを使う。サーバーの URL を指定して登録し、クライアントがネットワーク経由で接続する。スコープについては、チーム全体が同じリポジトリツール群にアクセスする必要があるなら project スコープを、自分だけがサーバーにアクセスするなら local スコープを選ぶ。

GitHub MCP サーバーの認証には Personal Access Token を使う。GitHub でトークンを生成し、MCP 構成のリクエストヘッダーで Bearer トークンとして渡す。トークンは環境変数で供給し、構成ファイルの中で参照しなければならない。`.mcp.json` にインラインで commit してはならない。commit されたファイルに直接書き込まれたトークンはリポジトリ履歴に入り、後の commit でファイルを上書きしても取り除けないからだ。

OAuth は別の認証機構で、ブラウザベースのサインインフローで個々のユーザーを認証するサーバーで使われる。Linear はこのパターンを使うサーバーの例だ。Linear MCP サーバーに初めて接続するとき、クライアントは Linear のサインインページにリダイレクトする。アクセスを承認すると、トークンが発行され、自動的に保存される。クレデンシャルは誰も手でコピーしたり管理したりしない。サービスの認可モデルがユーザー本人の識別に結びついている統合では、OAuth が正しいパターンだ。

GitHub MCP はユーザーが生成して保存するサービスクレデンシャルを使う。Linear MCP はサインインフローを起動し、クレデンシャルをユーザーに代わって処理する。どちらもリモートの HTTP サーバーで、どちらも同じトランスポートとスコープの論理に従う。違うのは認証のステップだ。

#### MCP セットアップのリファレンス

下の表は、各デプロイのコンテキストにおけるトランスポートとスコープの決定をまとめる。

| コンテキスト | トランスポート | スコープ | 構成の場所 | シークレットの扱い |
|---|---|---|---|---|
| 個人のローカルツール（自分のマシンでのみ走る） | stdio | Local | `~/.claude.json`（プロジェクトごとのエントリ） | 環境変数のみ。設定ファイルには書かない。 |
| 共有のチームサーバー（全チームメイトが同じサービスに接続） | HTTP | Project（`.mcp.json`） | リポジトリルートに commit する `.mcp.json` | OAuth または環境変数。API キーを `.mcp.json` に commit してはならない。 |
| 個人の実験（共有の準備ができていない） | stdio または HTTP | Local | 個人の Claude 設定 | 環境変数のみ。 |
| 組織全体のデプロイ（管理者管理） | HTTP | Enterprise | managed settings（管理者制御） | シークレットは管理者が管理。上書きを防ぐため構成はロック。 |

> **Cost · Complexity · Risk（コスト・複雑さ・リスク）**
> **Cost:** 接続した各 MCP サーバーはコンテキストウィンドウにツール定義を加える。接続するサーバーが多いほど、すべてのリクエストが大きくなる。指定のタスクが必要なサーバーだけを読み込むこと。
> **Complexity:** トランスポートとスコープは独立した決定だが、相互作用する。stdio サーバーはひとつのマシンでしか走らないため、共有用に project スコープにできない。スコープを選ぶ前に、サーバーが走る場所にトランスポートを合わせること。
> **Risk:** API キーを `.mcp.json` の中にバージョン管理へ commit するのが、この節で最もよくある間違い。キーはリポジトリ履歴に入り、後でローテートしても暴露を取り除くには不十分になる。シークレットは環境変数に置く。構成ファイルが持つのはサーバーのアドレスだけだ。

> **Handles well / Adds cost or complexity / Use a different approach**
> - **うまく扱える:** 複数の Claude Code セッションを通じて使い、チームと共有したい再利用可能な統合。機能が別プロセスとして保守するのに十分なほど安定している場合。GitHub サーバーは良い例だ。
> - **コストや複雑さが増す:** 環境のシークレットを慎重に管理していないチームは、注意深く見る必要がある。MCP サーバーを追加すると、シークレットが誤扱われる場所が増える。リスクはリポジトリに commit される `.mcp.json` ファイルに集中する。
> - **別の手法が合う:** ツールのロジックがコードベースの中に直接置け、セッションやアプリケーションをまたいで再利用する必要のない、ひとつ限りのタスク。ひとりのユーザーが使う単一プロジェクトの統合なら、サーバーを保守するよりも、API 呼び出しの中でツールを直接配線するほうが単純かもしれない。

## Screen 13 — MCP Servers — Watch Out（MCP サーバー・要注意）（4 min）

### 構成ファイルと一緒にリポジトリへ流れた API キー

> **Setup**
> サーバーは動いていた。チームは共有のセットアップを必要としていた。認証の方法の整理は、引き渡しの後でやれるように思えた。その近道は、構成ファイルが commit された瞬間に、一時的な直書きの API キーを、共有のクレデンシャル暴露に変えた。

#### 何が起きたか

ある開発者が、サービスアカウントの API キーを使ってデータウェアハウスの MCP サーバーに接続した。セットアップ中に素早くサーバーを動かすため、キーは `.mcp.json` 構成ファイルに直接置かれた。チームにセットアップを共有する前に環境変数へ移す計画だった。

開発者は `.mcp.json` をプロジェクトのリポジトリに commit し、チームメイトがリポジトリを clone すれば同じサーバーに接続できるようにした。するとキーも一緒に commit された。48時間以内に、三人のチームメイトがリポジトリを clone し、CI パイプラインが新しい clone をトリガーした。キーは四か所にあった。ローカルマシン、リポジトリ履歴、三人のチームメイトのマシン、CI ランナーのファイルシステムだ。

これに気づいた開発者は、キーを環境変数に移し、`.mcp.json` を更新し、修正したファイルを commit した。しかしキーはまだ commit 履歴の中にあり、サービスアカウントをローテートしなければならなかった。ローテーションで、同じキーで設定されていた二つの外部サービスが壊れ、これを直すのに三時間を要した。

修正後の `.mcp.json` は、インラインの値ではなく環境変数の参照を使う。

Before（do not use）

```
{
  "type": "http",
  "url": "https://warehouse.internal/mcp",
  "headers": {
    "Authorization": "Bearer sk-abc123..."       .............inline credential
  }
}
```

After（correct）

```
{
  "type": "http",
  "url": "https://warehouse.internal/mcp",
  "headers": {
    "Authorization": "Bearer ${WAREHOUSE_MCP_TOKEN}"    ............env variable reference
  }
}
```

> **What to Watch Out for（要注意なこと）**
> 構成ファイルに commit された API キーは、リポジトリ履歴に入る。後の commit でファイルを上書きしても、履歴からキーは取り除かれない。現在のバージョンから取り除かれるだけだ。commit されたファイルにインラインで書かれたクレデンシャルは、漏洩したものとして扱い、ローテートしなければならない。正しいパターンは、値を環境変数に置き、構成ファイルの中でその変数を参照することだ。
> エージェントがクレデンシャルの値を commit されるファイルに直接書き込むのを防ぐには、二つの層を使う。第一に、CLAUDE.md に規約の指示を書き、クレデンシャルの値を `.mcp.json` にインラインで書いてはならないと述べる。これで各セッションでモデルにルールを伝える。第二に、その指示を PreToolUse hook で補強する。`.mcp.json` に対する書き込みと編集を審査し、インラインのクレデンシャル値と思われるパターンを検出したら、ブロックする終了コードで抜ける。CLAUDE.md の指示が意図を伝え、hook はモデルがどう決めようと確定的にそれを強制する。これが、継続するプロジェクトのコンテキストの節で扱った hook と指示の同じ区別だ。CLAUDE.md の指示は、ファイルが膨らんだりコンテキストが移動したりするときに一貫性なく従われることがある。hook は例外なく、関連するすべてのツール呼び出しで発火する。

## Screen 14 — MCP Servers — Checkpoint（チェックポイント）（4 min）

### Checkpoint 5: 各デプロイシナリオにトランスポートとスコープを対応させる

やってみよう。下の各デプロイシナリオについて、正しいトランスポートとスコープを選ぼう。

## Screen 15 — Enterprise Integration — Teaching（エンタープライズ統合）（18 min）

### Claude を社内システムに接続し、安全に認証する

モジュールの前半で、MCP サーバーを構築し、トランスポートとスコープを設定する方法を扱った。社内プロジェクトでチームだけが使うサーバーなら、GitHub の Personal Access Token の例が認証パターンをカバーする。

この節で扱うのは、統合が規制された環境で動かなければならないときに変わる点だ。プロトタイプが通常無視する、識別、シークレットの扱い、データレジデンシーという三つの問いが、本番のデプロイでは回答が必須の要件になる。

#### 動くプロトタイプと、エンタープライズ統合は何が違うのか

社内サービスに Claude を繋ぐプロトタイプは、ひとつの問いに答える。接続は動くか。本番のエンタープライズ統合は、さらにいくつかの問いに答えなければならない。モデルは誰として振る舞っており、その識別は監査可能か。モデルはどのデータにアクセスでき、そのデータは組織の外にどこへ出るのか。管理者は、個々の開発者が認証のセットアップを変更できないように構成をロックできるか。アクセスは、コンプライアンス監査を満たす形で記録できるか。

これらの問いはエンタープライズソフトウェアにとって新しいものではない。規制されたデータに触れる外部システムすべてに適用されるのと同じ、識別、アクセス、コンプライアンスの要件だ。これらを統合の設計の一部として扱うことが、デモとデプロイ可能なものを分ける。

#### サービス種別ごとの認証パターン

適切な認証機構は、サービスがどこで走るかと、どんな識別モデルをサポートするかで決まる。各タブを選ぶと、パターンといつ使うかが見える。

**ユーザー識別を持つリモートサービス**
- OAuth を使う。MCP サーバーが 401 Unauthorized を返して認証が必要なことを示す。クライアントはブラウザベースのサインインフローを起動する。ユーザーがアクセスを承認すると、トークンが発行され保存される。誰も手でシークレットをコピーしない。OAuth フローは、クラウドサービス、SaaS ツール、そしてユーザーの識別が認可モデルの一部である統合すべてに期待されるパターンだ。このモジュールの前出の Linear MCP サーバーがこのパターンを使う。対して GitHub サーバーはヘッダーとして渡す Personal Access Token で認証する。

**サービス識別を持つリモートサービス**
- 環境変数で渡す API キーを使う。キーはサービスアカウントを識別する。キーを構成ファイルに commit してはならない。実行の場で環境に置く。Agent SDK を使う CI パイプラインでは、キーはコードにハードコードされず、パイプラインランナーがシークレットとして注入する。

**ファイルシステムアクセスを持つローカルサービス**
- ネットワーク認証のない stdio トランスポート。セキュリティ境界はファイルシステムのパーミッションモデルだ。設定ファイルの deny ルールがガバナンスの層になる。

#### シークレットそのものの管理が、安全な認証のもう半分

認証パターンの選択が接続を確立するが、それを保つのは別の問題だ。前に触れた MCP キーの漏洩は、認証方法の悪い選択ではなく、間違った場所にあり、広がったあとには除去できなかったクレデンシャルだった。三つの実践がこれを防ぐ。それぞれが、クレデンシャルが露出する特定の経路に対処する。

第一の実践は分離だ。クレデンシャルは、それを参照する構成と一緒に移動しない。構成ファイルは変数の参照だけを持ち、値はファイルが置かれていない場所に置かれる。これが、漏洩したキーの失敗が破ったルールだ。これが重要なのは、仕組み上の理由による。構成ファイルは commit され、共有され、clone される。インラインに書かれた値はそれらのコピーすべてに一緒に載って、commit された値は上書きでは取り除けない形でリポジトリ履歴に入る。値をファイルの外に保てば、ファイルは共有しても安全なままになる。

第二の実践は、値をファイルから出したあと、それをどこに置くかだ。ひとつのマシンやひとつのパイプラインの実行でしか生きない値なら、実行の場で注入される環境変数で十分だ。CI ランナーがシークレットとして設定し、構成は名前で読み取り、何もディスクに書かれない。複数のサービスや人が必要とする値なら、シークレットストアがよい。シークレットストアは、クレデンシャルを保持し、実行時に認可された呼び出し元に返し、誰が何を読んだかを記録するマネージドサービスだ。値を一元化するので、一度のローテーションですべての消費者を一度に更新できる。そして、各サービスが独自のクレデンシャルを独自のファイルに持つことで蓄積するコピーを取り除く。シークレットがローカルで短命なら環境変数を、共有されたり監査されたりするならシークレットストアを選ぶ。

第三の実践はローテーションだ。クレデンシャルは新しい値で定期的に、そして漏洩が疑われたら直ちに置き換える。ローテーションは漏洩したキーに対する唯一の適切な対応だ。一度露出したキーは、再び秘密に戻せないからだ。新しいものを発行しなければならない。インラインのクレデンシャルというやり方がこれほど高くつくのはこのためだ。commit されたコードにハードコードされた値は綺麗にローテートできない。古い値は履歴に残り、それにハードコードされたすべての消費者が変更で壊れる。シークレットストアや環境変数から読まれるクレデンシャルは、それを使うコードに触れることなくローテートできる。コードは値を名前で参照し、その背後の値が変わっても名前は変わらないからだ。

二つの習慣がローテーションを安くする。各クレデンシャルは、そのタスクが必要とする最も狭いアクセスにスコープする。そうすれば漏洩したキーは、その統合が必要としたものにしか届かない。そして、各クレデンシャルを使うサービスの記録を保つ。そうすればローテーションの際に、どこで使われているかが一目で分かる。

前出の漏洩したキーの失敗は、間違いを特定した。commit されたファイルにインラインで書かれたクレデンシャルだ。これが起きないようにするには、三つの実践を適用する。分離が値をファイルの外に保ち、シークレットストアか環境変数が、ファイルとは別の場所に値を置き、ローテーションは最初の二つが守られたときにのみ回復を助けられる。

#### 規制産業が、動く認証の上に加えるもの

金融サービスやヘルスケアの顧客は、「認証は動くか」以上のことを問う。データはどこで処理されるか。アクセスはどう記録されるか。管理者は、開発者が監査ウィンドウの間に認証のセットアップを変更できないように構成をロックできるか、と。

前出のエンタープライズの managed configuration が最後の問いに答える。管理者がデプロイしたサーバー構成を個々のユーザーが上書きできなければ、認証のセットアップは組織全体で一貫し、各開発者の設定ファイルが正しいことに依存しない。

監査 hooks が記録の問いに答える。PostToolUse hook がすべてのツール呼び出しとそのパラメータを監査ストアに記録すれば、コンプライアンスレビューが必要とする記録が得られる。hook はモデルがどう決めようと、すべての呼び出しで確定的に発火し、そのログをモデルがスキップすることはできない。

データレジデンシーが処理の問いに答える。特定のリージョンにある HTTP エンドポイントで構成されたサーバーと、処理をそのリージョンにピン留めするプラットフォームのデプロイを組み合わせれば、データがどこへ行くかというコンプライアンスレビュアーの問いに、チェックできる答えを与えられる。だからインフラの要件と、モジュールの前半のプラットフォームの選択は、ビルド時だけでなく監査時に影響する。

#### コードの近代化：モジュール全体をレガシーの変更に適用する

コードの近代化は、このモジュールが扱う内容すべてを試す、よいテストケースだ。各ツールが管理するよう設計されたリスクに集中するからだ。見知らぬレガシーコードベースへの大規模な変更は、影響範囲が大きく、依存が予測不能で、元に戻しにくい。このモジュールのツールは、作業が始まる前に適用すれば、それぞれのリスクに直接対処する。explore、plan、code のループが、この種の作業の中心のワークフローだ。plan モードはエージェントを読み込み専用の explore フェーズに留め、変更への信頼を築くあいだそこに置く。提案された編集をレビューし、想定外のパスに触れるものを特定し、ひとつのファイルが変更される前に差し戻せる。hooks は、最も機密性の高いフェーズで、特定のパスへの編集を防ぐガードレールを強制する。CLAUDE.md は新しいターゲットパターンの規約を運ぶ。その結果、エージェントは変更の全スコープにわたって新規約を一貫して適用し、周囲のコードからレガシーの作法へ引き戻されることがない。

高リスクの作業に対する責任あるスコープの取り方は、セッションが始まる前に三つの問いに答える。

- **何かが間違ったとき、影響範囲（blast radius）はどれほどか。** 変更されるコードにどのシステムが依存し、ひとつの編集が間違っていれば下流で何が壊れるか。
- **変更はどう監査されるか。** すべてのツール呼び出しを記録する PostToolUse hook はあるか。そのログは、エージェントが触ったものをレビューする必要がある人を満足させるか。
- **次のフェーズに進む前に、各フェーズを誰が承認するか。** plan モードは探索と実行のあいだの境界を強制するが、承認の決定そのものは、作業が始まる前にあなたが定義し文書化すべきものだ。

これらの問いは近代化の作業に固有ではない。高リスクなエージェント的タスクすべてに当てはまる。コードの近代化がこれらの問いを明確に浮かび上がらせるのは、スコープが大きく、コードベースは見知らぬもので、ミスのコストが高いからだ。

#### 認証と統合のチェックリスト

下の表は、各サーバー種別の主要な決定を挙げる。

| サービス種別 | 認証方法 | シークレットの場所 | 記録されるもの | 誰が構成をロックできるか |

> **Cost · Complexity · Risk（コスト・複雑さ・リスク）**
> **Cost:** OAuth フローはユーザーごと、サービスごとに一度のセットアップステップを加える。API キーの管理にはシークレットのローテーションプロセスが必要で、PostToolUse hook を通じた監査ログ記録はすべてのツール呼び出しに小さなオーバーヘッドを加える。
> **Complexity:** 規制された環境は、プロトタイプには現れない要件を加える。スコープの段階でそれらを特定するのが、統合をスケジュール通りに保つ規律だ。
> **Risk:** リスクは、プロトタイプが本番に向かうときに集中する。ハードコードされたクレデンシャルを使い、監査ログがなく、中央でロックできないシステムは、規制された顧客のセキュリティレビューを通らない。修正は難しくないが、レビューの前に注意を要する。

> **Handles well / Adds cost or complexity / Use a different approach**
> - **うまく扱える:** 規制された顧客が気にするデータに触れる統合で、同じツール群がすでにエンタープライズの managed settings と監査 hooks をサポートしている場合。セキュリティ要件を前もってスコープすることはオーバーヘッドをわずかにしか加えず、最終レビューで統合が止まるのを防ぐ。
> - **コストや複雑さが増す:** OAuth フローやエンタープライズのシークレット管理に馴染みのないチーム。これらのパターンは、大半の規制された組織でセキュリティや IT チームとの調整を要し、タイムラインはそれを織り込む必要がある。
> - **別の手法が合う:** 本番データに決して触れないプロトタイプや概念実証。デモ専用の統合に完全なエンタープライズ統合のチェックリストは不要だが、シークレットに環境変数を使う習慣を適用するのはコストゼロで良い実践だ。

## Screen 16 — Enterprise Integration — Watch Out（エンタープライズ統合・要注意）（3 min）

### ステージングでは動いたのに本番で失敗した OAuth 接続

> **Setup**
> OAuth 接続はステージングでエンドツーエンドで動いた。そのため本番への移行はルーチンの切り替えのように思えた。チームが見落としたのは、OAuth のリダイレクト URI がホストごとに登録され、しばしば環境ごとに統制されるということだった。だからステージングでの成功は、本番のホストがサインインフローを完了できると承認されていることを意味しなかった。

#### 抜けていたステップを明らかにした会話

次のやり取りは、本番で MCP 統合が失敗したあとのデプロイ後レビューで起きた。統合はすべてのステージングテストに通っていた。

Security reviewer: "Every production sign-in attempt through the MCP connection is failing. The error is a redirect URI mismatch. Where was the OAuth app registered?"

Developer: "I registered it for staging.mycompany.com during development. We moved to production last week. The connection worked all through staging."

Security reviewer: "That's the issue. The OAuth provider only accepts redirect URIs you've explicitly registered, and production.mycompany.com is not on the allowed list. Every sign-in attempt hits the check, fails the URI match, and loops back to the sign-in screen."

Developer: "So I just need to add the production URI to the app registration?"

Security reviewer: "Yes, and before you do, check whether your staging app registration should be a separate app from production. Most enterprise customers require separate OAuth app registrations for each environment as part of their security policy, so using the same app registration across environments is the second issue I'd flag."

開発者はステージングで OAuth フローをエンドツーエンドでテストし、動くことを確認していた。つまり本番の失敗はコードの欠陥ではなかった。それはホストごと、環境ごとに適用される構成ステップであり、開発者は本番についてそれを行うことを知らなかった。

> **What to Watch Out for（要注意なこと）**
> OAuth のリダイレクト URI はホストごとに登録される。だからステージングで動く OAuth 接続は、本番の接続が構成されていることを意味しない。OAuth 認証の MCP 統合を新しい環境に移す前に、新しいホストのリダイレクト URI を OAuth アプリの登録に追加する。規制されたエンタープライズの顧客については、ステージングと本番で別々の OAuth アプリの登録が必要かを確認する。登録のステップをデプロイのチェックリストに含め、最初の本番サインインの試行で発見されることがないようにすること。

## Screen 20 — Recap Key Takeaways（振り返り・重要なポイント）（6 min）

### 七つの重要なポイント

節ごとにひとつ、モジュールを貫くポイント。

**1. パーミッションモードはリスクの決定であって、速さの決定ではない。**
Claude Code は、すべてにプロンプトを出すものから、何にもプロンプトを出さないものまでを範囲に持つ。パーミッションモードは、プロンプトを減らしたいという個人の希望ではなく、作業と環境のリスクプロファイルに合わせるべきだ。開発者のワークステーションで、稼働中のコードベースに対する bypass モードは、エージェントとファイルのあいだのチェックポイントをすべて取り除く。触れてはならないパスへの deny ルールを、プロジェクトまたはエンタープライズレベルで設定すれば、モード単独では覆われない隙間をカバーできる。

**2. AI のコードレビューがくれるのは、トリアージすべき所見の集合であって、適用すべき評決ではない。**
レビュアーが目の前の差分から証明できる所見、例えば抜けた null チェックや閉じていないリソースは信頼し、それが引用する行で確認する。実行時の振る舞いや別のシステムに関する主張は、検証すべき仮説として扱う。レビュアーはそれを証明する証拠を持たずにその主張をしたからだ。所見が元に戻しにくいアクションに変わるポイントに人間のゲートを置き、レビュアーが推測しなければならない規約を与えることで、その精度を上げる。

**3. skill は持ち運べるが、「どこでも動く」は設計して作るものだ。**
同じ SKILL.md が Claude Code、Messages API、Agent SDK で動く。ただし、それぞれが読み込みとサンドボックス化を別の方法で行う。Claude Code のファイルシステム発見、API の beta ヘッダーとコード実行コンテナ、SDK の `settingSources` だ。明確な description にスコープし、ローカル環境の仮定を含まない skill は綺麗に移植する。書かれたターミナルを前提とするものはしない。どのランタイムでも、サブエージェントは白紙から始まる。自動的にはスキルを事前読み込みしない。

**4. 継続するコンテキストには、関心事ごとに正しい仕組みが必要だ。**
CLAUDE.md はセッションに永続するプロジェクトのメモリーだが、サイズで薄まる。ルールファイルはガイダンスを適用される場所に限定する。hooks は確率的ではなく、確定的にガードレールを強制する。サブエージェントは探索の作業をメインコンテキストの外に出す。この四つの仕組みはそれぞれ別の問題を解く。だからすべてを CLAUDE.md に詰め込むと、保守しにくく、無視されやすいひとつのファイルになる。

**5. 共有可能なセットアップには、持ち運べるコンポーネントが必要だ。**
作者のホームディレクトリへの絶対パスを参照するプラグインは、ひとつのマシンにインストールされ、ほかのすべてでは失敗する。共有される skills、hooks、プラグインのコンポーネントは、プロジェクトルートからの相対パスを参照しなければならず、環境変数の要件はすべて文書化されるか、インストール時に検証されなければならない。配布前にクリーンなマシンでインストールをテストすること。

**6. トランスポートとスコープは独立した決定だが、依存した結果を持つ。**
stdio は自分のマシンで走るサーバー向け。HTTP はリモートでホストされるものや、複数の開発者がアクセスするものすべて向け。Local スコープはサーバーを個人に保ち、project スコープは `.mcp.json` 経由でリポジトリと共有する。組み合わせはデプロイの意図に合致しなければならない。共有のチームサーバーは HTTP トランスポートと project または enterprise スコープを必要とする。`.mcp.json` の stdio サーバーは、共有可能に見えて、そうではない構成だ。

**7. エンタープライズ統合は、デプロイの前にセキュリティ要件を特定することを要する。**
規制された顧客は、識別、データレジデンシー、アクセスの記録、構成の統制について問う。答えは、ユーザー識別のサービスには OAuth から、サービスクレデンシャルには環境変数から、監査ログには PostToolUse hook から、構成のロックにはエンタープライズの managed settings から来る。これらはどれも実装は難しくないが、本番のデプロイがセキュリティレビューに失敗したあとに後付けするのは難しい。

## Screen 20B — Glossary Key Terms（用語集・重要な用語）（3 min）

### このモジュールの重要な用語

アルファベット順。用語をクリックすると定義が展開する。

- **Claude Agent SDK:** Claude Code がターミナルで走らせるのと同じエージェントループを公開する、プログラム可能なインターフェース。開発者がコードからループを呼び出し、パーミッションモードと利用可能なツールを設定し、インタラクティブなセッションなしでタスクを走らせられる。ターミナルで適用されるのと同じパーミッションモデルと deny ルールが SDK でも適用される。

- **CLAUDE.md:** Claude Code プロジェクトのルートに置かれる Markdown ファイル。その中身はすべてのセッションの開始時にコンテキストウィンドウの先頭に追加される。すべてのセッションで無条件に適用すべき、普遍的なプロジェクトの制約、規約、コマンドを保持する。およそ200〜300行を超えて大きくなったファイルは、内容の重みで重要なルールを薄めるリスクを負う。

- **hook:** Claude Code の実行のライフサイクルイベント（PreToolUse、PostToolUse、UserPromptSubmit、Stop）に紐づけられたコマンド。CLAUDE.md の指示と違い、hooks はモデルがどう決めようと、構成したイベントで確定的に走る。PreToolUse hook は終了コード 2 で抜けて、ツール呼び出しを実行前にブロックできる。

- **MCP（Model Context Protocol）:** Claude Code のような MCP クライアントが、ツール、リソース、プロンプトを提供する MCP サーバーに接続できるようにする、開かれた通信層。プロトコルは、クライアントがサーバーのツールをどう発見し呼ぶかを定める。MCP を使うと、ツールの定義と保守が個々のアプリケーションのコードから出て、どの MCP クライアントでも attach できる再利用可能なサーバーに移る。

- **MCP transport:** MCP クライアントと MCP サーバーのあいだの通信路。stdio はサーバーをクライアントと同じマシンのローカルサブプロセスとして走らせる。HTTP はネットワーク経由でリモートにホストされたサーバーに接続する。トランスポートの選択が、サーバーがどこで走れるか、誰が接続できるかを決める。

- **Permission mode:** Claude Code の設定で、ツール呼び出しを実行する前に、エージェントがどのくらいの頻度で確認を求めて止まるかを制御する。モードは default（ほぼすべてのアクションの前にプロンプト）から bypass 系（プロンプトなし）までを範囲に持つ。deny ルールはどのモードよりも優先する。エンタープライズ設定レベルの deny ルールは、個々の構成ではバイパスできない。

- **Plugin:** Claude Code のコンポーネント（skills、hooks、サブエージェント、MCP サーバー構成）の、バージョン管理されたバンドル。marketplace を通じて配布される。プラグインをインストールすると、受け手はひとつのステップで作者と同じセットアップを得る。エンタープライズ管理者は managed settings を通じてプラグインを組織全体にデプロイできる。

- **Rules instruction file:** Claude Code で、ガイダンスを特定のパスや条件にスコープするファイル。すべてのセッションで無条件に読み込まれる CLAUDE.md と違い、ルールファイルは Claude Code がその管理するディレクトリで作業しているときだけ起動する。パスに固有なガイダンスを、メインのプロジェクトメモリーファイルの外に保つのに使う。

- **Subagent:** Claude Code が委譲されたタスクを扱うために立ち上げる、別の実行コンテキスト。サブエージェントはメインの会話のコンテキストも蓄積したファイルも引き継がない。白紙から始めてタスクを行い、サマリーだけを返す。探索や調査にサブエージェントを使うと、再利用されない内容でメインセッションのコンテキストが埋まるのを防げる。
