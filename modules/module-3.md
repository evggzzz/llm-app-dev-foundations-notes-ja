## Screen 1 — Orientation（オリエンテーション）· 2 min

### このモジュールを終えるとできること

Claude Code は、ターミナルネイティブな開発パートナーだ。

前のモジュールで、プロンプト・ツールスキーマ・コンテキストエンジニアリング・エージェントループ・マルチモーダル取り込みという不可欠な API コンポーネントを設定した。このモジュールはその基盤の上に直接積み上げる。Claude Code は、同じモデルをターミナル環境で動かせるようにし、パーミッションレイヤー・設定システム・チーム指向の共有機能を導入する。MCP プロトコルは、外部サービスとの安全な統合を実現する。このモジュールでは、堅牢なセキュリティと効果的なデプロイのために Claude Code と MCP をどう設定するかを学ぶ。

このモジュールを終えると、次ができるようになる。

1. Claude Code を explore・plan・code のループで走らせ、作業のリスクレベルに合うパーミッションモードを選ぶ。タスクが必要とする以上の権限を与えずに、エージェントを生産的に保つ。
2. AI 生成コードを読み、校正された信頼（calibrated trust）で出力をレビューし、信頼できる所見は行動に移し、そうでない所見は検証し、誤呼び出しのコストが高い所には人間のレビューゲートを置く。
3. CLAUDE.md・ルール指示ファイル・hooks・サブエージェントを使い、Claude Code に永続的なプロジェクトコンテキストを与える。
4. ワークフローを skills・カスタムコマンド・プラグインとしてパッケージ化する。Claude Code・Messages API・Agent SDK で同じように走る skill を一度だけ書く。
5. Claude にツール・リソース・プロンプトを利用可能にする MCP サーバーを構築し、クライアントとサーバーがどう通信するかに合うトランスポートを選び、誰が読み込むかを制御する設定スコープを決める。
6. Claude をエンタープライズシステムに接続し、規制顧客が受け入れるパターンでそれらの接続を認証し、セキュリティレビューで持ちこたえるようにコードモダナイゼーションのエンゲージメントをスコープする。

このモジュールは、コードで Claude を既に動かしており、その作業を設定可能・共有可能・実システムへの接続において安全なものにしなければならない Developer 向けだ。実践的で、コードファーストで、パターン志向であることを前提とする。Module 2 の API パターンに慣れていることを前提とし、エージェントループ・ツールスキーマ・コンテキストエンジニアリングは教え直さない。稼働する統合の周りにある工学の意思決定を教える。ターミナルで Claude Code をパーミッションモデルの下でどう走らせるか、永続的なプロジェクトコンテキストをどう与えるか、チームメイトがインストールできるようにワークフローをどうパッケージ化するか、そしてクレデンシャルを漏らさず・セキュリティレビューに失敗せずに MCP で Claude を外部やエンタープライズシステムにどう接続するか。

### このモジュールにおける「ビルド」

このモジュールのすべては、1つの繰り返す問題を軸に組み立てられている。自分のマシン・セッション・ステージングで動くコードが、誰か他の人がプロダクションで、実社のシステムに対して走らせるとき、持ちこたえなければならない、という問題だ。

自分のマシンではパーミッションモードは安全に感じ、プロジェクトルールは従うのに十分小さく、skill はスクリプトを見つけ、クレデンシャルは設定ファイルにあり、接続はステージングテストで動いた。作業が自分のマシンを離れた瞬間、それらの便利さはそれぞれ失敗になりうる。パーミッションモードがスコープ外のファイルを削除する、1つのルールが何百行の下に埋もれる、skill が他のどのマシンにも存在しないパスを指す、commit されたキーが数時間で漏れる、ステージング専用の設定ステップがプロダクションの接続を落とす。このモジュールの作業は、チームメイトや監査人の前に現れる前に、どの設定の決定がそれらの失敗のどれを防ぐかを学ぶことだ。

## Screen 2 — Permission Modes & Human Gates — Teaching（パーミッションモードと人間のゲート）· 17 min

### Claude Code のエージェントループ、パーミッションモード、設定、人間のゲートの置き所

Module 2 は、エージェントループが API レベルでどう働くかを確立した。モデルがツールを呼び・結果を得て・タスクが終わるまで続く。Claude Code はその同じループをターミナルで走らせるが、追加のレイヤーを加える。エージェントが取ろうとするすべての行動を門番するパーミッションシステムだ。何かを設定する前に、ループがどう走り、パーミッションモードが何を制御するかを理解する必要がある。

#### Claude Code がタスクをどう進めるか：explore・plan・code

Claude Code にタスクを渡すと、すぐには書き始めない。ファイルを読み・関連ロジックをたどり・コードベースの像をまず構築する。これが exploration フェーズだ。そして、変更を提案するのに十分理解したら、計画を作る。計画は、行う予定の編集の構造化された記述だ。あなたが計画をレビューし承認した後でのみ、code フェーズに移り、変更を書いて実行する。

この順序は2つの理由で意味を持つ。1つ目、より良い出力を生む。Claude Code は何かに触れる前にコードベースを理解するので、より少ない想定で、より多くの下流効果を捉える。2つ目、パーミッションモードがここで繋がる。plan モードは Claude Code を explore フェーズに留め、あなたが解放するまで全ファイル編集とシェルコマンドをブロックする。だから不慣れなコードベースや利害の大きい作業に有用なデフォルトになる。

#### パーミッションモード：承認・ゲート・制約

パーミッションモードは、Claude Code がどれくらい頻繁に確認を求めて止まるかを制御する。各モードは速度と監督の異なるトレードオフを作る。正しい選択は、コードベースをどれだけ知っているかと、変更がどれくらい可逆的かによる。各モードが何を自動承認し、何をまだゲートし、その制限は何か、各タブを選ぶ。

**default**
- **自動承認するもの:** 読み込みのみ。ほぼすべての編集やコマンドの前にプロンプトを出す。
- **まだゲートするもの:** すべてのファイル編集とシェルコマンドに確認が必要。
- **制限:** 安全だが、信頼済み作業では遅い。新規プロジェクトや不慣れなコードベースの基準。

**acceptEdits**
- **自動承認するもの:** 読み込み、ファイル編集、作業ディレクトリ内の一般的なファイルシステムコマンド（`mkdir`, `touch`, `rm`, `rmdir`, `mv`, `cp`, `sed`）。自動承認は作業ディレクトリ内のパスにスコープされ、保護パスはまだプロンプトを出す。
- **まだゲートするもの:** その他すべてのシェルコマンド。作業ディレクトリ外への書き込み。保護パスへの書き込み。
- **制限:** シェル実行にまだ人間の目が必要な、信頼済みのローカル作業。エージェントがスクリプトを走らせなければならない場合は不適切。

**plan**
- **自動承認するもの:** 読み込みのみ。調査し提案する。編集はしない。
- **まだゲートするもの:** あなたが計画を承認するまで、すべてのファイル編集とシェルコマンド。
- **制限:** 機密性の高い、または不慣れなコードベースでの探索と計画。出力を書かなければならないタスクには不適切。

**auto**
- **自動承認するもの:** すべて。ただし別の分類器がまず各行動をレビューし、要求を超えてエスカレートする・認識されていないインフラを狙う・敵対的または不適切な内容に駆動されるように見えるものをブロックする。
- **まだゲートするもの:** プロダクションデプロイとマイグレーション、大量削除、クレデンシャル持ち出し、main への force-push はデフォルトでブロック。
- **制限:** プロンプトを減らすが安全を保証しない。これは研究プレビューで、機密操作のレビューの代わりではない。可用性はプラン・モデルバージョン・管理者設定による。ビルド前に常に現在の要件を確認。

**dontAsk**
- **自動承認するもの:** allow ルールで事前承認したツールと、読み取り専用コマンドのみ。それ以外はすべて自動拒否（DENY）。
- **まだゲートするもの:** allow リストにないすべてのツール呼び出しは拒否。確認のキューはない。
- **制限:** ロックダウンされた CI とスクリプト向けに構築。よく制限するが、ローカルの対話作業の摩擦を減らす手段ではない。

**bypassPermissions**
- **自動承認するもの:** すべてのツール呼び出し。確認プロンプトも安全チェックもない。
- **まだゲートするもの:** 通常操作では何も。標準のパーミッションチェックはバイパスされる。`rm -rf /` や `rm -rf ~` のような壊滅的な削除コマンドだけが、最後の手段のプロンプトをまだ出す。
- **制限:** 環境が使い捨ての、隔離されたコンテナや VM の中のみ。ライブコードベースに対する開発者ワークステーションでは決して使わない。

#### 設定はどこにあり、誰に適用されるか

設定は複数レベルに置け、各レベルがそれが含むルールのスコープを決める。

- **User level（`~/.claude/settings.json`）:** マシン上のすべてのプロジェクトに適用。探索作業の好みのデフォルトモードなど、どこでも自分に付いて回るべき好みの置き場所。
- **Project level（`.claude/settings.json`・リポジトリに commit）:** リポジトリを clone するすべての人に適用。チーム全体の規約、プロジェクトが使うツールの allow ルール、触れてはならないパスの deny ルールの置き場所。
- **Local project level（`.claude/settings.local.json`）:** 1プロジェクトの個人的な上書き。自動的に gitignore される。チーム全体に commit すべきでない自分の好みの置き場所。
- **Enterprise level（`managed-settings.json`・管理者が設定）:** ユーザーやプロジェクトファイルで上書きできない。環境ファイルへの編集の拒否や、全プロジェクトでの特定シェルコマンドのブロックなど、組織全体のセキュリティ統制の置き場所。

allow と deny のルールは、選んだモードの上に重なる。deny ルールは、有効なモードにかかわらず常に allow ルールに勝つ。最も永続的なガバナンス統制は、エンタープライズレベルの deny ルールだ。いかなる個々の開発者も削除できず、bypass モードが設定されていても適用される。

#### 人間がまだ見なければならない場所：最悪ケースのコストでレビューゲートを置く

パーミッションモードと deny ルールは、エージェントが聞かずに何ができるかを決める。それは、行動が着地する前に、あなた（人間）がまだ見る必要がある場所を決めない。その決定は1つの問いに載る。安全なモードと危険なモードを分けるのと同じ問いだ。この行動が人間のチェックなしで走ったら、最悪の結果は何か？ 間違えたときのコストが低いほど、多くを通せる。コストが高く、元に戻しにくいほど、そのステップは実行前に人間のゲートを必要とする。

同じ最悪ケースの問いは、エージェントがコードを書いているか、プルリクエストにコメントしたりブロックしたりするボットのような自動化ステップで無監視で走っているかにかかわらず、ゲートを置く。3つの配置がそれから導かれる。

- 低利害で可逆な行動はゲートなしで通す。フォーマット修正や作業ディレクトリに閉じた編集は、間違ってもほとんどコストを運ばない。だからそれぞれに人間の承認を求めるのは不要な監督を買い、作業を遅くする。これが acceptEdits のためにあるケースだ。
- 元に戻しにくい、または機密パスに届く行動はゲートする。作業ディレクトリ外への書き込み・破壊的シェルコマンド・セキュリティ関連や保護ファイルへの編集。そこでの誤呼び出しのコストは高い。だからエージェントは、走る前に一時停止し行動を人間に表面に出すべきだ。deny ルールがこれを決定論的に強制し、default や plan モードはあなたが決めるあいだプロンプトを保つ。
- チームが機密と印したコードへの変更で、エージェントだけがゲートになることは決してない。そこではエージェントの作業は人間の決定への入力で、代替でない。だからエージェントやその独自のレビューがどれほど自信満々でも、変更がマージする前に人がレビューしなければならない。

ゲートの配置とパーミッションモードの選択は、両側から見た同じ決定だ。モードはセッション全体のデフォルトを決め、ゲートは、デフォルトに任せるにはコストが高すぎる1つの行動のためにそのデフォルトを上書きする場所だ。両方とも、これがチェックされずに走ったら何が壊れるかを問うことから来る。

#### コスト・複雑性・リスク
- **コスト:** 信頼済み作業を default モードで走らせると、毎ツール呼び出しにプロンプトレイテンシが加わり、長いリファクタでは蓄積する。
- **複雑性:** 上書き階層を持つ複数レベルの設定とファイルは、一貫した注意を要する。エンタープライズレベルの deny ルールがプロジェクトレベルの allow ルールと矛盾する場合、プロジェクト設定を保守する全員が理解する必要がある。
- **リスク:** リスクは、文脈に誤ったモードを使うことにある。例えば、隔離されていないマシンで焦って bypass モードを設定すると、エージェントとライブファイルの間のすべての安全プロンプトを取り除き、他のモードと異なり、保護パスガードも取り除く。

## Screen 3 — Permission Modes & Human Gates — Watch Out（注意）· 3 min

### 重要な1つのプロンプトを消した bypass モード

#### 設定

作業が日常的に感じられたので、絶え間ないプロンプトを止めるために bypassPermissions に切り替えた。エージェントは何日もよく振る舞い、クリーンアップタスクは単純で、毎ツール呼び出し前にプロンプトを出す振る舞いは不要な摩擦に感じた。

#### トランスクリプト

次のペアリングセッションは、3日間問題なく動いていたコードベースのクリーンアップで起きた。

> **Dev:** "All right, I'm switching to bypassPermissions for this one. It's just renaming old API endpoint references. Nothing risky."
> **Claude Code:** [Scanning files matching pattern /v1/legacy/ ... 47 files found]
> **Claude Code:** [Updating endpoint references ...]
> **Claude Code:** [Running post-rename cleanup.sh script ...]
> **Claude Code:** [Deleted 3 files matching /v1/legacy/ in /deploy/config/prod/ ...]
> **Dev:** "Wait. What was in /deploy/config/prod/?"
> **Claude Code:** [Files contained environment-specific endpoint overrides for the production deployment. They have been removed.]
> **Dev:** "That directory wasn't supposed to be in scope. I was working on /src/."
> **Claude Code:** [The pattern /v1/legacy/ matched files in both /src/ and /deploy/config/prod/. In bypassPermissions mode, no confirmation was requested before the deletions.]

この間違いを捉えたはずのプロンプトは、bypassPermissions をオンにしたときにオフにされた。default や acceptEdits モードなら、クリーンアップスクリプトは確認なしには走らず、ユーザーは削除がプロダクション設定ファイルに届く前に止められた。bypassPermissions では、パターンマッチは意図より広く、スクリプトとそれが削除したファイルの間にプロンプトはなかった。

ゲートの正確な位置に注意。プロンプトを出したのは rm 削除コマンド単独でなく、スクリプト呼び出しだった。acceptEdits は作業ディレクトリ内のパスの rm を含む一般的なファイルシステムコマンドを自動承認する。Claude が削除を直接 rm コマンドとして発行していたら、acceptEdits は黙って通していた。default モードだけがそれらにプロンプトを出す。

#### 注意すべきこと

bypass モードは、予期しなかったものを含め、すべての確認プロンプトを黙らせる。ここでの失敗パターンは、チェックポイントのないモードで走るエージェントが、意図より広いファイルセットにマッチすることだ。bypassPermissions は他のモードが保つ保護パスガードもスキップするので、リポジトリ状態と Claude 自身の設定でさえ自動プロンプトを失う。この隙間を覆うには、モードを切り替える前に機密ディレクトリに deny ルールを設定しなければならない。安全網を失わずにプロンプトを減らしたいなら、完全な bypass でなく、分類器で門番するモード（auto など）を使う。

## Screen 4 — Permission Modes & Human Gates — Checkpoint（チェックポイント）· 4 min

### Checkpoint 1：settings ファイルを組み立て、人間のゲートを置く

やってみよう。payments モジュールの信頼済みローカルリファクタのために Claude Code を設定している。

リファクタはファイル編集を自動承認すべきだが、破壊的シェルコマンドを決して走らせてはならず、ファイル `.env.production` はエージェントに決して読まれてはならない。以下は settings.json の部品だ。

#### Part 1: 正しい構成を組み立てる settings.json の部品を選ぶ

2つ選べ。
- □ Piece A. `{ "permissions": { "defaultMode": "default"} }`
- □ Piece B. `{ "permissions": { "defaultMode": "bypassPermissions" } }`
- □ Piece C. `{ "permissions": { "allow": ["Bash(npm run:*)"], "deny": ["Bash(rm:*)", "Bash(git push:*)"] } }`
- □ Piece D. `{ "permissions": { "deny": ["Read(.env.production)"] } }`
- □ Piece E. `{ "permissions": { "allow": ["Bash(*)", "Edit(*)"] } }`

#### Part 2

設定はエージェントが自動的にファイルを編集することを許す。リファクタ中、エージェントが複数のプロダクションサービスが読むデプロイ設定ファイルへの変更を提案した。その1つの行動に人間のゲートはどこに置かれるべきか？ 最も良い答えを1つ選べ。
- a. Nowhere：設定は既に編集を自動承認するので、走らせる。
- b. デプロイ設定ファイルへの変更は、書き込みが実行される前に人間がレビューし承認する。そこでの誤った値は元に戻しにくく、ファイルの外のシステムに届くから。
- c. エージェントが決して一時停止しないよう bypassPermissions を追加する。
- d. 書き込みの後、次のプルリクエストでのみ変更をレビューする。

**ボタン:** `Submit` | `Skip for now`

> **訳注（推定正解）:** Part 1 は C（allow `npm run` / deny `rm`・`git push`）と D（deny `Read(.env.production)`）。Part 2 は b（プロダクション影響・不可逆だから書き込み前ゲート）。正式正解は原文の `Submit` で採点。

## Screen 5 — Durable Project Context — Teaching（永続的なプロジェクトコンテキスト）· 20 min

### CLAUDE.md・ルールファイル・hooks・サブエージェントによる永続的なプロジェクトコンテキスト

前節で、Claude Code がパーミッションモードと設定ファイルで行動をどう門番するかを見た。その設定レイヤーは、エージェントに許されることを制御する。

このクラスタはその上に積み上げる。エージェントが何を知りどう振る舞うかを設定する方法を学ぶ。セッションで定義したルールとプロジェクトコンテキストが、次のセッションの開始時にも効いているように。

#### CLAUDE.md：毎セッションに読み込まれるプロジェクトファイル

Claude Code がプロジェクトディレクトリで起動するたび、ルートの CLAUDE.md という名前のファイルを探して読む。内容は、あなたからのメッセージが届く前にプロンプトに追加される。つまり CLAUDE.md に置いたすべての規約・制約・コマンドは、毎セッションの最初のプロンプトから存在する。あなたが言い直す必要はない。

`/init` コマンドはコードベースを走査し、初期の CLAUDE.md を生成する。生成されたファイルは素晴らしい基準だが、使う前に検証すべきだ。プロンプトの結果を制御するルールを保持するよう洗練させる。テストコマンド・フレームワークの規約・エージェントが触れてはならないパス・デフォルトと異なるスタイルの決定。

サイズが主な失敗モードだ。新しい指示ごとに成長し続ける CLAUDE.md は、最も重要なルールを薄める。より大きいファイルはより多くのコンテキストウィンドウを消費し、単一の指示を読み込まれる全体の中でより小さい割合にし、それが本当の失敗を捉える1つのルールにエージェントが従う確率を下げる。CLAUDE.md は振る舞いを変える制約に留め、他はすべてオンデマンドで読み込まれる Skills に移す。

#### ルール指示ファイル：指針を適用される場所にスコープする

前節で、CLAUDE.md が毎セッションに読み込まれ、プロジェクト全体に当てはまる指示を保持すべきだと確立した。次の問いは、コードベースの一部にしか意味がない指針をどうするかだ。それがルール指示ファイルの出番で、指示を毎セッションに読み込む代わりに、relevant な場所にだけ適用できる。

CLAUDE.md は常にオンで、ルールファイルはその基準の上に狭いレイヤーを加える。プロジェクトの `.claude/rules/` ディレクトリに置かれ、YAML frontmatter の paths glob で特定パスにスコープできる。こうスコープされたルールは、Claude Code がパターンに合致するファイルを扱うときだけコンテキストに読み込まれる。これにより、残りのコンテキストを散らかさずに、コードベースの1部にルールを適用できる。

スコープは、ファイルの配置でなく frontmatter から来ることに注意。ルールファイルは `.claude/rules/` のサブディレクトリ（例 `.claude/rules/database/`）に整理できるが、その構造は整理のみ。paths フィールドのないルールファイルは、`.claude/rules/` のどこにあっても、起動時に無条件で読み込まれ、CLAUDE.md と同じ優先度を持つ。

実践には、広いプロジェクトの記憶と普遍的な制約は CLAUDE.md に置き、狭いパス固有の指針は paths でスコープしたルールファイルに置く。「データベーススキーマを決して変更するな」のような制約はどこにでも適用されるから CLAUDE.md に。「データベースモジュールのすべての SQL は明示的なトランザクション境界を含め」のような制約は frontmatter を伴う `.claude/rules/database.md` に置く。

```yaml
---
paths:
- "src/db/**/*.sql"
---
```

こうして、Claude がそれらのファイルを扱うときだけコンテキストに入る。

#### Hooks：ライフサイクルの固定点で独自スクリプトを走らせる

Hook は、ツール呼び出しの実行前後にそれを傍受し制御できる。CLAUDE.md に、ファイル編集後に毎回 Prettier を走らせるよう特定のルールを書けば、エージェントはほとんどの場合それに従う。一方 hook は、モデルが何を決めるかに依存せず、例外なく毎回それを起こす。hook は独立して発火するからだ。

hooks は設定ファイルで定義され、`/hooks` コマンドで構成する。各 hook は、ライフサイクルイベント・特定のツール型にスコープする任意の matcher・イベント発火時に走るコマンドに束縛される。ほとんどのガードレールと自動化ユースケースの中核イベントは以下。

- **PreToolUse:** ツール呼び出しの実行前に走る。最初に走るので、PreToolUse hook はツール呼び出しを調べ、exit code 2 でブロックでき、理由を stderr に書いてエージェントが見るフィードバックにする。これが、エージェントが CLAUDE.md の指示を尊重することを望む代わりに、設定レイヤーでアクセス制御を強制する方法だ。
- **PostToolUse:** ツール呼び出し完了後に走る。呼び出しは既に起きたので、このイベントはブロックできない。それが自動副作用に適した場所にする。編集後にコードフォーマッタを走らせる・ファイル変更後にテストを起動する・監査証跡のために操作をログする。
- **UserPromptSubmit:** プロンプトを送信時、モデルが処理する前に走る。作業が始まる前にコンテキストを注入または要求を検証する必要があるときに使う。
- **Stop:** モデルが応答を終えたときに走る。通知・クリーンアップタスク・監査ログの commit のように、ターンの終わりに属するフォローアップに使う。
- **Notification:** Claude Code が通知を送るとき（ツール使用の許可が必要なとき・60秒アイドルの後）に走る。それらの信号を外部チャネルやロギングシステムに流すのに使う。
- **SessionStart:** セッションの開始・再開時に走る。状態の初期化・環境変数の検証・エージェントが作業を始める前の必要サービス到達確認に使う。
- **SessionEnd:** セッション終了時に走る。解体タスク・最終監査書き込み・セッションが閉じた通知に使う。

PreToolUse イベントでプロダクション設定パスへの編集をブロックする hook は、パーミッションモードにかかわらず、毎セッションの毎ツール呼び出しでその制約を強制する。それがガードレールと規約の違いだ。

#### サブエージェント：隔離されたコンテキストに作業を委譲する

サブエージェントは、Claude Code がタスクを委譲できる専門化されたアシスタントで、各アシスタントは自身の別個のコンテキストでタスクを走らせ、出力だけを返す。メインの会話履歴・コンテキストに蓄積したファイル・現在のセッション状態を継承しない。サブエージェントにタスクを送ると、白紙から始め、作業し、結果を返す。

組み込みサブエージェントは、起動時に何を読み込むかで異なる。この違いがプロジェクトルールの適用を決める。現在のリストは Claude Code ドキュメントで常に確認すること。セットは時と共に大きくなったが、プロジェクトルールに影響する特定の分割はバージョンをまたいで保たれる。Explore と Plan の組み込みサブエージェントは、調査を速く安く保つため CLAUDE.md と git status をスキップする。速度に最適化されているので、CLAUDE.md で定義されたプロジェクトレベルのルールとリポジトリ状態は、走るときコンテキストにない。general-purpose サブエージェントは両方を読み込む。Explore や Plan にタスクを委譲し、CLAUDE.md のルールが適用されるべきなら、そのコンテキストが読み込まれなかったことが原因だ。プロジェクトの制約を尊重しなければならないタスクでは、general-purpose サブエージェントか、必要なルールを明示的に読み込むカスタムサブエージェントを使う。

カスタムサブエージェントも、自動的にはあなたの skills を見ない。`.claude/agents` にカスタムサブエージェントを定義し、特定の skill が必要なら、エージェントの front matter にその skill を明示的にリストしなければならない。組み込みエージェントは事前読み込みされた skills を持たない。組み込みエージェントが skill 支援の振る舞いを必要とするなら、正しい経路は、それらの skills を構成にリストしたカスタムサブエージェントを作ることだ。

| メカニズム | 読み込むもの | 走るタイミング | コンテキストコスト | 属するもの |
|---|---|---|---|---|
| **CLAUDE.md** | セッション開始時にコンテキストの前に追加される完全なファイル内容。 | 毎セッション・無条件。 | セッションあたり永続。サイズで薄まる。 | 普遍的なプロジェクト制約・コマンド・フレームワークの決定。 |
| **ルールファイル** | ファイル内容。YAML frontmatter の paths glob でスコープ。paths なしは CLAUDE.md のように読み込む。 | ルールの paths パターンに合致するファイルを Claude が読むとき。スコープなしはセッション開始時。 | パススコープ：トリガ時だけコンテキストに追加。スコープなし：CLAUDE.md と同じ永続コスト。 | 他の場所ではノイズになるパス固有の指針。 |
| **Hook** | ライフサイクルイベントでスクリプトを走らせる。コンテキストに内容は追加されない。 | 構成されたイベント（PreToolUse・PostToolUse 等）。 | 最小：Claude に戻すならスクリプト出力のみ。 | 強制ガードレール・自動副作用・監査ロギング。 |
| **サブエージェント** | タスクコンテキストのみ。メインセッションから隔離。 | 委譲タスクのためにメインセッションが派遣するとき。 | 完全なタスク履歴でなく要約を返す。 | 探索・調査・出力がさもなければメインコンテキストを膨らませるタスク。分解して並列化できるタスクにも有用。 |

- **Handles well（うまく扱える）:** 多くのセッションをまたいで戻るプロジェクトで、安定したルールセット・ディレクトリごとの変動・無条件ガードレールが設定に報いるとき。
- **Use a different approach（別の手法を使う）:** 二度と戻らない単発タスク。不慣れなコードベースの quick な探索には、設定のオーバーヘッドは割に合わない。

## Screen 6 — Durable Project Context — Watch Out（注意）· 4 min

### ルールが効かなくなるまで成長し続けた CLAUDE.md

#### 設定

新しいルールごとに追加する価値があるように感じたので、CLAUDE.md は成長し続けた。各追加は個別には妥当で、ファイルは各ルールの正しい置き場所に見えた。しかし数週間で、ファイルは800行を超えた。

#### トレース

次は、2か月のチームの追加が蓄積した CLAUDE.md を持つプロジェクトのセッションログ抜粋だ。

```
Session context window loaded:
CLAUDE.md: 847 lines
Contents include: framework preferences (lines 1–40), testing conventions (lines 41–90),
  style guide (lines 91–210), dependency rules (lines 211–320), path restrictions (lines 321–360),
  historical decisions log (lines 361–700), archived notes (lines 701–847)

User prompt: "Refactor the auth module to use the new token service. Do not modify the /legacy/tokens/ directory."
Claude Code action: [Reading auth module ...]
Claude Code action: [Editing auth/service.ts ...]
Claude Code action: [Reading /legacy/tokens/store.ts for reference ...]
Claude Code action: [Editing /legacy/tokens/store.ts to update token interface ...]
Path restriction from CLAUDE.md (line 347): "Do not modify files in /legacy/tokens/."

User: "You just edited /legacy/tokens/store.ts. I said not to touch that directory."
Claude Code: "I apologize. The /legacy/tokens/ restriction is in the project instructions.
  The edit was made in the course of updating the token interface.
  I did not flag the conflict before proceeding."
```

ルールはファイルにあった。エージェントはそれにアクセスした。しかし失敗は希釈（dilution）だった。他の846行が、重要な1つの指示の有効な重みを下げた。履歴の決定ログとアーカイブされたメモはどこかに記録されるべきだったが、CLAUDE.md に属するものではなかった。

#### 注意すべきこと

CLAUDE.md は、成長する追加ログでなく、現在のセッションの振る舞いを変える稼働中のルールセットだ。追加するすべての行が、他のすべての行の重みを下げる。ルールがパス固有なら、ルールファイルに属する。ルールが履歴コンテキストなら、エージェントがオンデマンドで読む別の参照ドキュメントに属する。CLAUDE.md が数百行を超えたら、監査する。本当にセッションに不可欠なルールを特定し、残りを移す。薄めたくない1つのルールは、hook への最短経路であるべきだ。

## Screen 7 — Durable Project Context — Checkpoint（チェックポイント）· 3 min

### Checkpoint 2：正しい値をドラッグする

やってみよう。パス制限を強制する hook を設定していて、下の構成に2つの空欄がある。

正しいものを選べ。ツール呼び出しの実行前に走るライフサイクルイベントと、`.env.production` の読み取りをブロックするために hook が走らせるコマンド。

```json
{
  "hooks": {
    "________": [
      {
        "matcher": "Read",
        "hooks": [{ "type": "command", "command": "________" }]
      }
    ]
  }
}
```

- **Blank 1（ライフサイクルイベント）:** PreToolUse / PostToolUse / UserPromptSubmit / SessionStart
- **Blank 2（コマンド）:**
  - A. ツール呼び出しを stdin から読み、ファイルパスをチェックし、パスが `.env.production` のとき exit code 2 で終わる（理由を stderr に書く）スクリプト。
  - B. ツール呼び出しを監査ファイルにログし、exit 0 するスクリプト。
  - C. 警告を表示し、無条件で exit 0 するスクリプト。

**ボタン:** `Submit` | `Skip for now`

> **訳注（推定正解）:** Blank 1 = PreToolUse（実行前ブロック）。Blank 2 = A（exit 2 でブロック・stderr に理由）。正式正解は原文の `Submit` で採点。

## Screen 8 — Packaging Workflows — Teaching（ワークフローのパッケージ化）· 8 min

### プラグインとしてのワークフロー：skills・カスタムコマンド・marketplace インストール

前節では、Claude Code に永続的なコンテキストと振る舞いの強制を与えるメカニズムを扱った。常にオンのプロジェクト記憶には CLAUDE.md、スコープされた指針にはルールファイル、決定論的ガードレールには hooks、隔離されたタスク委譲にはサブエージェント。

これらのメカニズムは `.claude` ディレクトリにあり、プロジェクトと共にバージョン管理される。次の問いに移る。その設定を、チームメイトが手動の構成を手で繰り返す代わりに、1ステップで簡単にインストールできるようにパッケージ化するには、どうすればよいか？

#### Skills はエージェントがオンデマンドで読み込む再利用可能なワークフロー

skill は、`.claude/skills` に置かれた移植可能な Markdown ファイル（SKILL.md ファイル）だ。front matter が skill を識別し適用されるべきときを記述し、本文がステップを持つ。同じ skill は Claude Code で走り、Messages API を通じて呼ばれ、Agent SDK で読み込まれる。3つで変わるのはファイルそのものでなく、skill がどこで走るか・どう読み込まれるか・何に触れてよいかだ。Claude Code でしか skills を見たことがない開発者は、API では成り立たないことを想定しがちなので、この画面は違いを概説する。

##### 各環境で skill がどう読み込まれ走るか

**Claude Code**
- **読み込み方:** ファイルシステムの `.claude/skills` から発見。description の合致または名前で呼んだときに読み込む。
- **走る場所:** ターミナルセッションで、ローカルファイルに対して、有効なパーミッションモードと deny ルールの下で。
- **知っておくべきこと:** ファイルシステムベースで、設定レイヤーが治める。

**Messages API**
- **読み込み方:** リクエストと共に送られ、アプリケーションの環境でなく code execution container の中で走る。code-execution と skills の beta ヘッダーが必要。
- **走る場所:** 自分のマシンでなく、Anthropic の code execution container の中。skill のファイルシステムとツールアクセスは、そのコンテナが提供するもの。
- **知っておくべきこと:** ローカルファイルやローカルツールを想定する skill は、ここでは同じようには振る舞わない。それらのファイルがある場所で走っていないから。

**Agent SDK**
- **読み込み方:** SDK が走らせるエージェントが読み込む。ただしファイルシステム設定（CLAUDE.md・skills）が読み込まれるかは `settingSources` configuration で制御される。デフォルトに頼らない。常に意図するソースに明示的に設定し、ビルド時に Agent SDK リファレンスで現在のデフォルト挙動を確認する。`settingSources`（TypeScript）／`setting_sources`（Python）で設定する。
- **走る場所:** SDK が走らせるプロセスで、ファイルシステムソースを読み込むよう指示すれば、自分の環境。
- **知っておくべきこと:** よくある驚き：Claude Code で働いた skill が SDK では何もしない。settingSources が設定されず skill が読み込まれなかったから。

**Claude Managed Agents**
- **読み込み方:** モデル・system prompt・ツール・MCP サーバー・skills を名指しする API リソースとして一度定義。エージェントが走るとき Anthropic が skill をサーバー側で読み込むので、自分の側にファイルシステム発見ステップはない。
- **走る場所:** 自分の環境でなく、Anthropic がプロビジョニングし走らせるサンドボックスの中。アプリケーションはユーザーイベントを送り、ストリーム結果を読み戻す。skill は自分のローカルファイルでなく、そのマネージドサンドボックスが提供するものにアクセスする。
- **知っておくべきこと:** 現在パブリックベータで `managed-agents-2026-04-01` beta ヘッダーを必要とする。セッションはサーバー側に保存されるので、Managed Agents は現在 Zero Data Retention や HIPAA BAA カバレッジの対象ではない。skills はエージェントリソースの定義時に、セッション時でなく、付属させる。利用可能な skills を変えるにはエージェント定義を更新する。

##### 3つの移植性ルール

1. **description をマッチング基準として書く。** モデルは要求を skill の description と比較して skill を読み込む。だから skill が適用されるべきときを識別する description はすべてのランタイムで働くが、曖昧なものはすべてで読み込みに失敗する。
2. **skill の本文でローカルファイルシステムやローカルツールの存在を想定しない。** ローカルコマンドにシェルアウトする skill は Claude Code では働くが、コマンドのないコンテナで走る Messages API では壊れる。skill のステップをランタイムが保証して提供するものに限定するか、依存をドキュメント化する。
3. **サブエージェントは skills を継承しないことを忘れない。** これは Module 2 でそうだったし、ここでもそうだ。サブエージェントはクリーンに始まるので、親が依存した skill は、サブエージェントをサポートするすべてのランタイムで、サブエージェントに明示的にリストされなければならない。

実用的な要点は、skill を一度書いて再利用できるが、ターミナルをまたいで使えるようには特別に設計しなければならないことだ。明確な description にスコープされ、ローカル環境の想定がない skill はランタイムをまたいで綺麗に移植するが、特定のローカル環境を想定するものはしない。

- **Handles well（うまく扱える）:** 一度書かれ、対話ターミナル・API 統合・ヘッドレス SDK ジョブをまたいで再利用される、タスク固有の手順。
- **Adds complexity（複雑さを加える）:** 各ランタイムは skill を違って読み込みサンドボックス化するので、API の beta ヘッダーと SDK の settingSources を考慮しなければならない。
- **Use a different approach（別の手法を使う）:** プロジェクトの毎セッションに適用されなければならない指示には、CLAUDE.md がまだ正しい道具。Skills はオンデマンドの移植可能な手順向け。

#### ワークフローに明示的なエントリポイントを与える

カスタムコマンドは、定義された手順のショートカットだ。現在の Claude Code では、明示的と自動の両方の呼び出しに skills が推奨形式だ。skill を `/skill-name` で直接呼ぶか、関連するとき Claude が自動的に読み込む。古い `.claude/commands/` ディレクトリ形式はまだ動くがレガシーの手順だ。明示的に呼んだときだけ走るワークフローが欲しいときは、frontmatter に `disable-model-invocation: true` を持つ skills を使う。

プラグインコマンドは自動的に名前空間化される。プラグインの名前が接頭辞になり、`payments` というプラグインの `run-tests` コマンドは `/payments:run-tests` として呼ばれる。だから2つのプラグインが両方とも `run-tests` コマンドを出荷しても衝突しない。作者はプラグイン名をインターフェースの一部として扱うべきだ。出荷するすべてのコマンドに接頭辞が付くから。プラグインの改名がすべてを改名することに注意。

#### セットアップをインストール可能にするパッケージ化レイヤー

プラグインは、skills・hooks・サブエージェント・MCP サーバーを1つのインストール可能な単位に束ねる。プラグインは marketplace を通じてパッケージ化し配布できる。marketplace は誰かが作って共有したプラグインのカタログだ。公式 Anthropic marketplace は Claude Code 起動時に自動的に利用可能で、`/plugin marketplace add <owner/repo>` のようなコマンドで GitHub リポジトリにホストされたサードパーティ marketplace を追加できる。チームメイトは1つの簡単なインストールコマンドで同じセットアップを得られる。プラグインは、手動セットアップの手順のページを、バージョン管理された監査可能なインストールに置き換える。プラグインはコンポーネントを次のように置く。

- Skills は skills ディレクトリに。
- Hooks・サブエージェント・設定はそれぞれの場所に。

プラグインマニフェストが束を記述し、インストールコマンドが対象のインストールにそれを繋ぎ込む。プラグインは個人またはエンタープライズ全体でダウンロードできる。

エンタープライズ管理者は managed settings を通じてプラグインを組織全体にデプロイできる。managed marketplace allowlist が、ユーザーが追加を許可される marketplace ソースを門番し、組織がプラグインがどこから来られるかを制御する。allowlist はユーザーが追加できるものを制限するが、marketplace を自動的に登録しない。ユーザーに add コマンドを自分で走らせずにすべてのユーザーに marketplace を押したいなら、allowlist 設定を managed settings の `extraKnownMarketplaces` と組み合わせる。優先順位はデプロイスコープから来る。managed settings は設定階層で user と project settings の上にあり、managed スコープでデプロイされたプラグインは優先し、ユーザーやプロジェクトファイルで上書きできない。正確な設定名は参照レイヤーで確認。

#### コスト・複雑性・リスク
- **コスト:** Skills は起動時にコンテキストコストを加えるが、プラグインはインストールと保守のオーバーヘッドを加える。問うべきは、セットアップコストを1回（プラグインインストールのように）払うか、繰り返し（すべての開発者が同じ手動手順を手で走らせるように）払うかだ。
- **複雑性:** skill に絶対パスを硬编码するプラグインは、作者には正しくインストールされ、他の全員には失敗する。skill や hook コマンドに焼き込まれたパスや環境の想定が、マシンをまたいで最も壊れやすいから。
- **リスク:** プラグインは、バンドルしたコンポーネントを毎インストールに運ぶ。作者がローカルで依存した deny ルールや hook は、束の一部として明示的にリストされない限り含まれないことを忘れてはならない。skills や hooks が束に含まれないガードレールに結びついているなら、その保護はチームメイトのマシンに持ち越されない。

## Screen 10 — Packaging Workflows — Watch Out（注意）· 3 min

### 自分のマシンにインストールされ、他の全員で失敗したプラグイン

#### 設定

綺麗にインストールされたプラグインは、束が正しく組み立てられたことを告げる。しかしプラグインが効果的に走ることは告げない。インストールと実行は別物だから。インストールはファイルを所定の場所にコピーする。実行は、それらのファイルが指すパスと変数を、走っているマシンに対して解決する。プラグイン作者が自分のマシンのレイアウトを skill に焼き込むと、インストールはどこでも成功するが、実行は作者自身のセットアップ以外どこでも失敗する。この隙間は作者には見えないものだから起きる。

#### 何が起きたか

開発者はデプロイワークフローの skill を構築し、プラグインとしてパッケージ化し、ローカルでテストした。ローカルテストは通り、プラグインは内部 marketplace を通じてチームに出て、すべてのチームメイトのインストールは成功した。しかしチームメイトが skill を走らせた瞬間、失敗した。

根本原因は skill の SKILL.md の、`/Users/alexmorgan/projects/deploy-utils/validate.sh` を指すコマンドにあった。そのディレクトリは作者のマシンにだけ存在し、他にはなかった。skill は作者のホームディレクトリへの絶対パスを運んだので、すべてのチームメイトの実行は、自分のシステムにあるか skill に含まれるファイルを探した。

同じプラグインの2つ目の skill は、作者が自分のシェルプロファイルに設定した環境変数 `DEPLOY_TOKEN` に依存し、プラグインの README はそれに一言も触れなかった。3人のチームメイトが、2つ目の失敗を欠けた変数にたどり着く前に2時間デバッグに費やした。

プラグインは作者のマシンをチームのマシンとして誤って扱い、それが破損を引き起こした。上の例の両方の失敗は同じ根本原因と同じ絶対パスを持つ。それは SKILL.md に平文であり、ファイルを読むレビュアーはそれを捉えられる。環境変数は危険になりうる。パッケージのどこも関連する依存を告知しないので、skill は変数を必要とするステップまでうまく走り、そのとき初めて失敗する。だから3人が2時間かけて直すことになりうる。

#### 注意すべきこと

skill・hook コマンド・プラグインコンポーネントのあらゆるパス参照は、プロジェクトルートに相対であるか、ベースパスに環境変数を使わなければならない。プロジェクトに保存されたスクリプトを参照するには `$CLAUDE_PROJECT_DIR` を、プラグイン自身にバンドルされたスクリプトには `${CLAUDE_PLUGIN_ROOT}` を使い、どのマシンが走らせても・セッションがどのディレクトリで始まってもパスが正しく解決するように。プラグインが依存するスクリプト・設定ファイル・その他アセットはすべて、プラグインにバンドルするか共有プロジェクトの場所に含め、すべてのチームメイトがインストール後に同じファイルにアクセスできるように。プラグインが必要とするすべての環境変数をドキュメント化し、インストール時に検証し、実行中でなく即座に欠けたものを表面に出す。それから配布前にクリーンなマシンでインストールをテストする。これがビルドマシンが隠している問題を捉える。

## Screen 12 — MCP Servers — Teaching（MCP サーバー）· 21 min

### MCP サーバーの構築と設定：トランスポート・スコープ・GitHub サーバー

前節では、プラグインを、skills・hooks・サブエージェント・MCP サーバーを1つのインストール可能な単位に束ねるパッケージ化レイヤーとして導入した。

この画面は、MCP サーバーの束が何か・どう構築するかをさらに説明する。MCP サーバーは、コードベースの外から Claude にツールを公開するレイヤーだ。MCP サーバーを構築するとき、最初の決定の1つは、適切なトランスポート機構を決め、サーバーのスコープを定義することだ。

#### MCP サーバーとは何か、ツールを直接繋ぐのと何が違うか？

ツールをアプリケーションに直接繋ぐとき、ツールのスキーマと機能の定義に責任を持つ。両方ともそのアプリケーションのコードにある。3つの異なるアプリケーションが同じ外部サービスへのアクセスを必要とするなら、それぞれが独自の統合を保守する。Model Context Protocol（MCP）は、ツール定義を個々のアプリケーションから切り離し、サーバーと呼ばれるプロセスに変える。

MCP サーバーは、MCP クライアントが使えるツール・リソース・プロンプトを公開するプロセスだ。Claude Code は組み込みの MCP クライアントを持つ。MCP サーバーに接続すると、Claude Code はそれが提供するツールを発見し、セッション中に呼べる。MCP サーバーを使えば、能力を一度構築し、それに接続するすべての MCP クライアントが、統合を再実装せずにアクセスする。

#### MCP サーバーはリソースとプロンプトも公開する

MCP サーバーはツール・リソース・プロンプトを公開する。ツールについては既に学んだ。モデルが呼べる行動だ。残り2つは、ツール呼び出しが必要なものをくれないケースを覆う。

**リソース（resource）**は、モデルがツールを呼んで取ってくる代わりに、クライアントがフェッチしてコンテキストに直接置く、サーバーが公開する読み取り専用データだ。クライアントはアドレスでリソースを要求し、サーバーがデータを返す。リソースには2つの形がある。**direct resource** はパラメータを取らないデータ（利用可能なドキュメントのリストなど）に固定アドレスを持つ。**templated resource** はアドレスにパラメータを置く（ドキュメント識別子を取るドキュメントアドレスなど）。ターンの最初から既知のデータをコンテキストにおきたいとき、リソースに手を伸ばす。ツール呼び出しで取りに行くより、直接リソースを引く方が安く予測可能なときに欲しい。リソースサポートは MCP クライアントで異なる。このパターンに頼る前に、クライアントがリソースをコンテキストに注入する仕組みを持つか確認する。

**プロンプト（prompt）**は、サーバーが公開する事前書き込みの指示テンプレートで、クライアントが各ユーザーに自分で書かせる代わりに、名前で検証済みプロンプトを呼べる。ユーザーは既に自分の言葉でほとんどのタスクをモデルに頼める。だからプロンプトは、特定の言葉遣いが必要なとき有用だ。慎重に構築した指示がユーザーが打つものより大幅に良い結果を生み、すべてのクライアントが同じ品質を得てほしいタスク。指示をサーバーにパッケージ化すれば、プロンプトは1箇所で保守され、サーバーが接続されているどこでも再利用される。

#### トランスポート：Claude Code がサーバーとどう話すか

トランスポートは、MCP クライアントと MCP サーバーの間の通信チャネルだ。正しいトランスポートは、サーバーがどこで走るかによる。

**stdio** は、サーバーをクライアントと同じマシンのローカルプロセスとして走らせる。クライアントはサーバーをサブプロセスとして起動し、標準入出力で通信する。ローカルツール・個人的なスクリプト・自分のマシンで走らせる開発サーバーに正しい選択だ。チームで共有したい、またはリモートでホストしたいサーバーには働かない。

**HTTP** は、ローカルで走らないすべてのサーバーに推奨されるトランスポート形式だ。標準の HTTP 接続で繋ぎ、異なるマシンにホストされたサーバーをサポートする。HTTP サーバーを登録するとき、URL を提供し、クライアントがネットワークで接続する。共有チームサーバーとホストされた統合は HTTP を使う。

**SSE（Server-Sent Events）**は、現在の HTTP トランスポートに先行する古い手段だ。HTTP トランスポートに取って代わられ、新しいサーバーにはもはや推奨されない。既存の設定やドキュメントで SSE に出会ったら、現在の推奨でなくレガシーの選択肢として扱う。

#### コンテキストコスト

接続した各 MCP サーバーは、前払いで読み込まれるとコンテキストウィンドウを占めるツール定義を寄与する。デフォルトで、Claude Code はこれらの定義を前払いで読み込む代わりに延期し、検索ステップで、タスクがそれらを呼ぶときだけ関連ツールを発見し読み込む。呼ばれたツールだけがコンテキストに入る。

オプトインモードは、ツール定義がコンテキストウィンドウの約10%に収まるとき前払いで読み込み、その上限を超えるときだけ延期する。いずれにしても、必要なサーバーだけを接続することが各リクエストをリーンに保つ。接続したすべてのサーバーが、モデルが考慮しなければならない定義のプールに追加するから。

#### プロンプトキャッシュ：再利用可能なリクエストに一度だけ払う

MCP サーバーで見たコンテキストコストの問題は、コストとウィンドウの両面を持つ。毎リクエストは入力をゼロから再処理し、前回のリクエストと同一だった部分を含む。つまり毎回再処理に払う。プロンプトキャッシュは、同じ安定した内容に2度払うのを止められる。

キャッシュは、リクエストの安定したプレフィックスになされた処理を保存し、後続リクエストが同じトークンを再処理する代わりに再利用できるようにする。最初のリクエストがプレフィックスをキャッシュに書き、後続リクエストは同じ地点まで同一の内容をコストの一部で送る。内容は exactly に合致しなければならない。キャッシュポイントの前の1文字の変更がそのキャッシュを無効にし、新規書き込みを強制する。だからキャッシュの最も有力な候補は、めったに変わらないリクエストの部分だ。長い system prompt・大きいツール定義セット・複数の問いをする参照ドキュメント。

キャッシュはキャッシュブレークポイントを印してオンにする。キャッシュをオンにするグローバル設定はない。Messages API では、キャッシュしたい最後のブロックに `cache_control` フィールド（type: `ephemeral`）を追加する。これはそのブロックまで（含む）をキャッシュする。最大4つのブレークポイントを置ける。リクエストはツール・system prompt・メッセージの固定順で処理されるので、ツールの後のブレークポイントはツール定義をキャッシュしつつメッセージを動的に保つ。

キャッシュには時間制限がある。デフォルトのキャッシュ寿命は最終読み取りから5分。ブレークポイントに `ttl: 1h` を設定すると、オプトインの1時間寿命が使える。5分のデフォルトは、リクエストが数分ごとに届くやり取りモデルに合う。各読み取りが時計をリセットするから。1時間の選択肢は、リクエスト間の隙間が長いワークロード（ステップ間で一時停止するエージェントなど）に合う。5分窓が次リクエスト前に切れる場所だ。窓が次リクエスト前に切れると、読み取りの利益なしに書き込みコストをまた払うことになる。キャッシュは最小トークン閾値（現在のほとんどのモデルで1,024トークン）を超えた場合にのみ適用されるので、短いプロンプトはブレークポイントを設定してもキャッシュされないことに注意。

#### 検索拡張生成：Claude がリクエストが必要な知識だけを引く方法

MCP サーバーで見たコンテキストコストの問題は、大量の参照資料が作るのと同じだ。モデルは毎リクエストでコンテキストウィンドウのすべてを読む。だから前もって読み込むドキュメントが多いほど、より多くのコンテキストが使われ、作業の余地が減る。検索拡張生成（retrieval-augmented generation・通常 RAG と短縮）が、これを解くパターンだ。すべてのドキュメントをコンテキストに読み込む代わりに、システムは資料をコンテキストウィンドウの外に保存し、現在のリクエストに最も関連する部分を見つけ、リクエスト時にその部分だけをモデルに供給する。モデルは全体のライブラリでなく、その検索されたスライスから答えを生成する。

RAG には2つの形がある。

**Classical RAG** は前もって重い仕事をする。誰かが問う前に、ソース資料をチャンクに分け、各チャンクを意味を数学的に捉える数値のセット（埋め込み・embedding と呼ぶ）に変換する。それらの数値はデータベースに保存される。ユーザーが問うたとき、システムは問いを同じ型の数値に変換し、最も似た数値を持つチャンクを見つける。図書館が開く前にすべての本を読み、各章の正確な要約カードを書いた司書のように、問いで来たときに正しいカードを即座に引ける。

**Agentic search** は前もったインデックスを完全に飛ばす。事前構築されたデータベースがない。代わりに、モデルが尋ねられた瞬間に必要なものを把握し、それを取りに行く。ライブソースを検索し、オンデマンドでドキュメントを読み、タスクが展開するにつれ結果を引く。問いで来たとき、事前準備されたカードを相談する代わりに、自分で答えを探しに行く研究者のように。

名前を知らずに agentic search に出会ったことがあるかもしれない。Claude Code では、多くの外部ツール（MCP サーバー）に接続しているとき、Claude はすべてのツール定義を前もって読み込まない。一度に保持するには多すぎるから。代わりに、現在のタスクに必要なツールだけを発見し読み込む。Claude.ai Projects はアップロードされたドキュメントで同じように働く。プロジェクトのナレッジベースがアクティブウィンドウに収まらなくなる大きさに成長すると、すべてを読み込む代わりに、各問いに最も関連するドキュメントセクションだけを表面に出す。

両アプローチは同じ根本のことをする。関連する資料のスライスを見つけ、そこから生成する。違いはタイミングだ。Classical RAG は事前に構築されたインデックスに対するマッチでスライスを見つけ、agentic search は必要の瞬間の検索で見つける。

検索に手を伸ばす前に理解する価値のある2つの性質。

- **スケールする。** ソース資料が成長しても、各リクエストのコストは平坦に保たれる。モデルが受け取るのはその問いに関連するスライスだけで、ライブラリ全体でないから。ナレッジベースは数千のドキュメントに成長でき、単一の問いは依然としてほぼ同じ量のテキストを引く。それが検索をスケールで働かせる。ソースは成長し続けても、リクエストはそれと共に成長しない。
- **見つけるもの次第。** モデルは受け取ったスライスを推論する。検索ステップが必要なドキュメントを見逃せば、モデルはそれを見ない。つまり資料の整理方法が重要。曖昧な名前（"notes_final_v3.pdf"）のファイルは、説明的な名前（"Q3 refund policy, updated August 2024"）より表面に出しにくい。関連するファイルを一緒にグループ化も役立つ。良い検索はよく整理されたソースから始まる。

#### 設定スコープ：誰がサーバーを読み込むか

スコープは、どのユーザーとプロジェクトがサーバーを読み込むかを決める。各スコープは異なる設定場所に対応する。

- **Local scope** は、現在のプロジェクトのパスの下の `~/.claude.json` にサーバー設定を保存する。現在作業しているプロジェクトにのみ適用され、チームメイトと共有されない。リポジトリに commit する準備ができていない特定のプロジェクトコンテキストに結びついたサーバーや、1つのプロジェクトでしか意味をなさないツールに適したスコープ。
- **User scope** は、個人の Claude 設定にサーバー設定を保存し、すべてのプロジェクトをまたいで利用可能にする。個人のまま。チームメイトには見えず、リポジトリには書かれない。すべてのプロジェクトで使う個人のユーティリティ（ローカルデータベースツールや、どのコードベースでも頼るスクリプト）に適したスコープ。
- **Project scope** は、リポジトリルートの `.mcp.json` ファイルにサーバー設定を書く。そのファイルがバージョン管理に commit されると、リポジトリを clone するすべての人が同じサーバーを自動的に得る。チーム全体がアクセスできるサーバーに適したスコープ。設定がコードと共に移動するから。留意すべきこと：project-scoped サーバーは各チームメイトのマシンから走る。stdio サーバーでは、commit された設定は起動コマンドを保存し、すべての clone が独自のローカルサブプロセスを立ち上げる。だから各チームメイトはランタイム（npx で起動するサーバーの Node など）をローカルにインストールする必要がある。
- **Enterprise scope** は、管理者が制御する中央管理された設定を通じてデプロイする。管理者は個別の設定ステップなしに、組織のすべてのユーザーにサーバーを押せる。共有された内部サービス・セキュリティツール・組織全体に存在しなければならず個々の開発者の設定に任せられないサーバーに適したスコープ。

#### サーバー全体でなく単一の MCP ツールを対象にするパーミッションプルール

サーバーに接続するとその完全なツールリストが公開されるが、エージェントにチェックなしにそれらすべてに届いてほしいことはまれだ。パーミッションモードの節のパーミッションレイヤーは MCP ツールに拡張され、ルールはサーバー全体でなく個々のツールを名指しできる。

MCP ツールは、パーミッションプルールでサーバー名とツール名で識別される。`mcp__server__tool`。`mcp__github__create_issue` の allow ルールは、その1つのツールをプロンプトなしで走らせ、GitHub サーバーの他のすべてのツールはまだプロンプトを出す。write 可能なツールの deny ルールはそれをブロックし、同じサーバーの読み取り専用ツールは利用可能に保つ。これが、広いサーバーに接続しつつ、エージェントができることの狭いスライスの中に留める方法だ。1つのツールの deny はサーバーの allow を上書きする。

API MCP connector はもう1つの有用な制御だ。API MCP connector 経由でサーバーに届く場合、`mcp_toolset` オブジェクトでツールごとに `enabled` フラグを設定できる。この enabled フラグで、サーバーを登録しつつ、モデルに見せたい特定のツールだけを公開できる。パーミッションプルールは公開されたツールが走ってよいかを決め、enabled フラグはモデルがツールを見るかを決める。前者はガバナンス統制、後者はコンテキストコストとスコープの統制。これらの統制は一緒に使われることが多い。公開前に、ドキュメントに対して正確なルール構文と connector の beta ヘッダーを常に確認する。

#### GitHub MCP サーバー：具体例でのトランスポート・スコープ・認証

GitHub MCP サーバーは、GitHub が保守するリモートサーバーで、プルリクエストのレビュー・issue のオープン・コード検索などリポジトリ管理のツールを公開する。接続プロセスを歩くことで、トランスポート・スコープ・認証が、誰か他の人が保守するサーバーでどう一緒に働くかが見える。

GitHub サーバーは、GitHub がリモートでホストするので HTTP トランスポートを使う。サーバー URL を提供して登録し、クライアントがネットワークで接続する。スコープでは、チーム全体が同じリポジトリツールにアクセスする必要があるときは project scope、自分だけがサーバーにアクセスする必要があるときは local scope を選ぶ。

GitHub MCP サーバーの認証は Personal Access Token を使う。GitHub でトークンを生成し、MCP 設定のリクエストヘッダーで Bearer トークンとして渡す。トークンは環境変数で供給し、設定ファイルで参照しなければならない。`.mcp.json` にインラインで commit してはならない。commit されたファイルに直接書かれたトークンはリポジトリ履歴に入り、後の commit でファイルを上書きしても取り除けないから。

OAuth は別の認証機構で、サービスがブラウザベースのサインインフローで個々のユーザーを認証するサーバーで使われる。Linear はこのパターンを使うサーバーの例だ。Linear MCP サーバーに初めて接続するとき、クライアントは Linear のサインインページにリダイレクトする。アクセスを承認すると、トークンが発行され自動的に保存される。クレデンシャルは何も手でコピー・管理されない。OAuth は、サービスの認可モデルがユーザー identity に結びつく統合に正しいパターンだ。

GitHub MCP は自分が生成して保存するサービスクレデンシャルを使う。Linear MCP はクレデンシャルを処理するサインインフローを開始する。両方ともリモート HTTP サーバーで、同じトランスポートとスコープの論理に従う。認証ステップだけが違う。

#### MCP セットアップの参照

| コンテキスト | トランスポート | スコープ | 設定場所 | シークレットの扱い |
|---|---|---|---|---|
| 個人のローカルツール（自分のマシンでのみ走る） | stdio | Local | `~/.claude.json`（プロジェクトごとのエントリ） | 環境変数のみ。設定ファイルには決して。 |
| 共有チームサーバー（全チームメイトが同じサービスに接続） | HTTP | Project（`.mcp.json`） | `.mcp.json` をリポジトリルートに commit | OAuth または環境変数。API キーは `.mcp.json` に決して commit しない。 |
| 個人の実験（共有の準備ができていない） | stdio または HTTP | Local | 個人の Claude 設定 | 環境変数のみ。 |
| 組織全体のデプロイ（管理者管理） | HTTP | Enterprise | Managed settings（管理者制御） | シークレットは管理者が管理。上書きを防ぐため設定をロック。 |

#### コスト・複雑性・リスク
- **コスト:** 接続した各 MCP サーバーはツール定義をコンテキストウィンドウに追加する。接続するサーバーが多いほど、毎リクエストが大きくなる。所与のタスクが必要なサーバーだけを読み込む。
- **複雑性:** トランスポートとスコープは独立の決定だが、相互作用する。stdio サーバーは1台のマシンでしか走らないので、共有のために project-scoped にはできない。スコープを選ぶ前に、サーバーがどこで走るかにトランスポートを合わせる。
- **リスク:** `.mcp.json` の中に API キーをバージョン管理に commit するのが、このセクションで最も一般的な失敗。キーは、後のローテーションでは露出を取り除くのに不十分なリポジトリ履歴に入る。シークレットは環境変数へ。設定ファイルはサーバーアドレスのみ。

- **Handles well（うまく扱える）:** 複数の Claude Code セッションをまたいで使い、チームと共有したい再利用可能な統合で、能力が別プロセスとして保守するのに十分安定しているもの。GitHub サーバーが素晴らしい例。
- **Adds cost or complexity（コスト・複雑さを加える）:** 環境シークレットを慎重に管理していないチームは注視すべき。MCP サーバーを追加すると、シークレットが誤扱われる場所の数が増える。リスクはリポジトリに commit される `.mcp.json` ファイルに集中する。
- **Use a different approach（別の手法を使う）:** ツールの論理が直接コードベースにあり、セッションやアプリケーションをまたいで再利用する必要のない単発タスク。1人のみが使う単一プロジェクトの統合には、サーバーを保守するより API 呼び出しでツールを直接繋ぐ方が単純かもしれない。

## Screen 13 — MCP Servers — Watch Out（注意）· 4 min

### 設定ファイルと共にリポジトリへ旅した API キー

#### 設定

サーバーは動いており、チームは共有セットアップを必要とし、認証方法の整理は引き渡し後にできるように感じた。その近道は、設定ファイルが commit された瞬間に、一時的な硬编码 API キーを共有クレデンシャル露出に変えた。

#### 何が起きたか

開発者はサービスアカウントの API キーを使ってデータウェアハウス MCP サーバーに接続した。セットアップ中にサーバーを素早く動かすため、キーは `.mcp.json` 設定ファイルに直接置かれた。計画は、チームにセットアップを共有する前に環境変数に移すことだった。

開発者はチームメイトがリポジトリを clone して同じサーバーに接続できるよう、`.mcp.json` をプロジェクトリポジトリに commit し、キーもそれと共に commit された。48時間以内に、3人のチームメイトがリポジトリを clone し、CI パイプラインが新規 clone を起動した。キーは4箇所にあった。ローカルマシン・リポジトリ履歴・3人のチームメイトのマシン・CI ランナーのファイルシステム。

これに気づいた後、開発者はキーを環境変数に移し、`.mcp.json` を更新し、修正したファイルを commit した。しかしキーはまだ commit 履歴にあり、サービスアカウントはローテーションされなければならなかった。ローテーションは同じキーで構成された2つの外部サービスを壊し、直すのに3時間かかった。

修正した `.mcp.json` はインライン値でなく環境変数参照を使う。

**Before（使わないこと）**
```json
{
  "type": "http",
  "url": "https://warehouse.internal/mcp",
  "headers": {
    "Authorization": "Bearer sk-abc123..."
  }
}
```

**After（正しい）**
```json
{
  "type": "http",
  "url": "https://warehouse.internal/mcp",
  "headers": {
    "Authorization": "Bearer ${WAREHOUSE_MCP_TOKEN}"
  }
}
```

#### 注意すべきこと

設定ファイルに commit された API キーは、リポジトリ履歴に commit される。後の commit でファイルを上書きしても、履歴からキーを取り除かない。現在のバージョンから取り除くだけ。commit されたファイルにインラインで書かれたクレデンシャルは、侵害されたとみなしローテーションしなければならない。正しいパターンは、値を環境変数に置き、設定ファイルでその変数を参照すること。

エージェントがクレデンシャル値を直接 commit されたファイルに書くのを防ぐには、2層を使う。1つ目、CLAUDE.md に、クレデンシャル値は `.mcp.json` に決してインラインで書いてはならないという慣習指示を追加する。これが毎セッションでルールをモデルに伝える。2つ目、その指示を、`.mcp.json` への書き込み・編集操作を調べ、インラインクレデンシャル値らしきパターンを検出したら exit code でブロックする PreToolUse hook で裏打ちする。CLAUDE.md の指示が意図を伝え、hook がモデルが何を決めるかにかかわらず決定論的にそれを強制する。これは永続的なプロジェクトコンテキストの節で扱った hook 対 指示の区別と同じ。CLAUDE.md の指示は、ファイルが成長したりコンテキストが移ったりすると一貫して従われなくなりうる。一方 hook は例外なく関連するすべてのツール呼び出しで発火する。

## Screen 15 — Enterprise Integration — Teaching（エンタープライズ統合）· 18 min

### Claude をエンタープライズシステムに接続し、安全に認証する

前の方で、MCP サーバーをどう構築し、トランスポートとスコープをどう設定するかを扱った。内部プロジェクトでチームのみが使うサーバーでは、GitHub の Personal Access Token の例が認証パターンを覆う。

この画面は、統合が規制環境で働かなければならないとき何が変わるかを覆う。プロトタイプが通常無視する、identity・シークレットの扱い・データ常駐の問いが、プロダクションデプロイで答えなければならない要件になる。

#### エンタープライズ統合は稼働するプロトタイプと何が違うか

Claude を内部サービスに接続するプロトタイプは、1つの問いに答える。接続は働くか？ プロダクションのエンタープライズ統合は、追加の問いに答えなければならない。モデルは誰として振る舞い、その identity は監査可能か？ どのデータにアクセスでき、そのデータは組織のどこを出るか？ 管理者は、個々の開発者が認証セットアップを変えられないよう設定をロックできるか？ アクセスはコンプライアンス監査を満たす形でログできるか？

これらの問いはエンタープライズソフトウェアに新しいものでない。規制データに触れるすべての外部システムに適用されるのと同じ identity・アクセス・コンプライアンス要件だ。それらを統合設計の一部として扱うことが、デモとデプロイ準備完了のものを分ける。

#### サービスタイプ別の認証パターン

正しい認証機構は、サービスがどこで走るかと、どんな identity モデルをサポートするかによる。

**ユーザー identity を持つリモートサービス**
OAuth を使う。MCP サーバーが認証が必要だと合図するため 401 Unauthorized を返す。クライアントがブラウザベースのサインインフローを開始する。ユーザーがアクセスを承認すると、トークンが発行され保存される。誰もシークレットを手でコピーしない。OAuth フローは、クラウドサービス・SaaS ツール・ユーザーの identity が認可モデルの一部であるすべての統合で期待されるパターン。このモジュールの前の Linear MCP サーバーがこのパターンを使う。一方 GitHub サーバーは、対照的に、ヘッダーとして渡される personal access token で認証する。

**サービス identity を持つリモートサービス**
環境変数で渡される API キーを使う。キーはサービスアカウントを識別する。キーは設定ファイルに決して commit してはならず、実行のポイントで環境に住む。Agent SDK を使う CI パイプラインでは、キーはコードに焼き込まれず、パイプラインランナーによってシークレットとして注入される。

**ファイルシステムアクセスを持つローカルサービス**
ネットワーク認証のない stdio トランスポート。セキュリティ境界はファイルシステムのパーミッションモデル。設定ファイルの deny ルールがガバナンスレイヤー。

#### 認証後のシークレット管理：保存・ローテーション・設定からの分離

シークレットそのものの管理が、安全な認証のもう半分だ。クレデンシャルは、それを参照する設定と共には決して旅しない。設定ファイルは変数参照だけを持ち、値は実行のポイントで注入される環境変数か管理されたシークレットストアに住む。サービスアカウントキーはファイルでなくシークレットマネージャーに保存し、スケジュールで、また疑わしい露出の直後にローテーションする。キーが漏れたらローテーションしなければならないが、commit されたコードに焼き込まれた値はローテーションできないことを忘れない。各クレデンシャルを、そのタスクが必要とする最も狭いアクセスにスコープし、侵害されたキーがその1つの統合が必要としたものにしか届かないように。

認証後のシークレット管理は、接続を確立することと別の問題。前出の MCP キー漏洩は、認証方法の悪い選択でなく、間違った場所に住み、広がった後に清掃できなかったクレデンシャルだった。3つの実践がそれを防ぎ、それぞれがクレデンシャルが露出する特定の方法に対処する。

1つ目は**分離（separation）**：クレデンシャルは、それを参照する設定と共には決して旅しない。設定ファイルは変数参照を持ち、値はファイルのないどこかに住む。これが漏洩したキーの失敗が破ったルール。重要な理由は機械的だ。設定ファイルは commit され・共有され・ clone される。インラインに書かれた値はそれらのコピーすべてに便乗し、commit された値は上書きでも取り除けない形でリポジトリ履歴に入る。値をファイルの外に保てば、ファイルは共有に安全なまま。

2つ目は**値の置き場所**。ファイルの外に出た値がどこに行くか。1台のマシンや1つのパイプライン実行にしか住まない値には、実行のポイントで注入される環境変数で十分。CI ランナーがシークレットとして設定し、設定が名前で読み、ディスクには何も書かれない。複数のサービスや人が必要な値には、シークレットストアが良い。シークレットストアはクレデンシャルを保持する管理サービスで、実行時に認可された呼び出し元に返し、誰が何を読んだかを記録する。値を一元化し、1回のローテーションですべての消費者を一度に更新し、各サービスが独自のファイルにクレデンシャルを保持するときに蓄積するコピーを取り除く。シークレットがローカルで短命なら環境変数、共有されるか監査されなければならないならシークレットストア。

3つ目は**ローテーション（rotation）**：スケジュールで、また疑わしい露出の直後に、クレデンシャルを新しいものに置き換える。ローテーションは漏れたキーに対する唯一の適切な対応。露出されたキーは再び秘密にできないから。新しいものを発行しなければならない。だからインラインクレデンシャルのパターンは高くつく。commit されたコードに焼き込まれた値は綺麗にローテーションできない。古い値が履歴に留まり、それに hardcode されたすべての消費者が変更で壊れるから。シークレットストアや環境変数から読まれるクレデンシャルは、それを使うコードに触れずにローテーションする。コードは値を名前で参照し、その背後の値が変わっても名前は変わらないから。

2つの習慣がローテーションを安くする。各クレデンシャルを、そのタスクが必要とする最も狭いアクセスにスコープし、漏れたキーがその統合が必要としたものにしか届かないように。各クレデンシャルを使うサービスの記録を保ち、ローテーションが消費者を表面に出さないように。

前出の漏洩したキーの失敗は間違いを特定した。commit されたファイルにインラインで書かれたクレデンシャル。これが起きるのを防ぐため、3つの実践を適用する。分離が値をファイルの外に保ち、シークレットストアか環境変数が値にファイルが共有しない家を与え、ローテーションは最初の2つが保たれたときにのみ回復を助けられる。

#### 規制業種が稼働する認証の上に加えるもの

金融サービスやヘルスケアの顧客は「認証は働くか？」以上の問いをする。データがどこで処理されるか・アクセスがどうログされるか・管理者が監査ウィンドウ中に開発者が auth セットアップを変えられないよう設定をロックできるか。

前節のエンタープライズ managed configuration が最後の問いに答える。管理者がデプロイしたサーバー設定で、個々のユーザーが上書きできないものは、auth セットアップが組織全体で一貫し、各開発者の設定ファイルが正しいことに依存しないことを意味する。

audit hooks がロギングの問いに答える。すべてのツール呼び出しとそのパラメータを監査ストアにログする PostToolUse hook が、コンプライアンスレビューが必要な記録を提供する。hook はモデルが何を決めるかにかかわらずすべての呼び出しで決定論的に発火し、ログはモデルが飛ばせるものではない。

データ常駐が処理の問いに答える。特定リージョンの HTTP エンドポイントで構成されたサーバーと、処理をそのリージョンに固定するプラットフォームデプロイの組み合わせが、データがどこに行くかの確認可能な答えをコンプライアンスレビュアーに与える。だからインフラ要件とプラットフォーム選択は、ビルド時でなく監査時に効く。

#### コードモダナイゼーション：このモジュールの全体をレガシー変更に適用する

コードモダナイゼーションは、このモジュールが覆うすべての有用なテストケースだ。各ツールが管理するよう設計されたリスクに集中するから。不慣れなレガシーコードベースへの大規模変更は、大きい blast radius・予測不能な依存・限られた可逆性を運ぶ。このモジュールのツールは、作業が始まる前に適用すれば、それらのリスクそれぞれに直接対処する。explore・plan・code のループがこの種の作業のコアワークフロー。plan モードは、変更に自信を構築するあいだ、エージェントを読み取り専用の explore フェーズに留める。提案された編集をレビューし、予想外のパスに触れるものを特定し、1つのファイルが変更される前に押し戻せる。hooks は最も機密なフェーズで特定パスへの編集を防ぐガードレールを強制する。CLAUDE.md は新しいターゲットパターンの規約を運ぶので、エージェントは周囲のコードに読むレガシーパターンに戻る代わりに、変更の全スコープをまたいで一貫してそれらを適用する。

高リスク作業の責任あるスコープ付けは、セッションが始まる前に3つの問いに答える。

- 何かが間違ったら blast radius は何か：変更されるコードにどのシステムが依存し、編集が間違ったら下流で何が壊れるか？
- 変更はどう監査されるか：すべてのツール呼び出しをログする PostToolUse hook はあるか、そのログはエージェントが触れたものをレビューする必要がある人を満たすか？
- 次のフェーズが始まる前に各フェーズを誰が承認するか？ plan モードは探索と実行の境界を強制するが、承認決定そのものは、作業が始まる前にあなたが定義しドキュメントするものだ。

これらの問いはモダナイゼーション作業に特有でない。任意の高リスクエージェントタスクに当てはまる。コードモダナイゼーションがそれらを明確に表面に出すのは、スコープが大きく・コードベースが不慣れで・間違えたときのコストが高いから。

#### 認証と統合のチェックリスト

| サービスタイプ | 認証方法 | シークレットの場所 | ログされるもの | 設定をロックできる人 |
|---|---|---|---|---|
| ユーザー identity 持つリモート（SaaS・クラウド） | OAuth | OAuth プロバイダーが発行しクライアントが保存するトークン | 監査ログへの PostToolUse hook | エンタープライズ managed settings による管理者 |
| サービス identity 持つリモート（内部 API） | 環境変数の API キー | 環境のみ。commit された設定には決して | 監査ログへの PostToolUse hook | エンタープライズ managed settings による管理者 |
| ローカル（ファイルシステム・ローカル DB） | ファイルシステムパーミッション | クレデンシャル不要。deny ルールがパスアクセスを強制 | 監査ログへの PostToolUse hook | エンタープライズ managed settings の deny ルール |

#### コスト・複雑性・リスク
- **コスト:** OAuth フローはユーザー・サービスごとに1回のセットアップステップを加える。API キー管理はシークレットローテーションプロセスを必要とし、PostToolUse hook による監査ロギングは毎ツール呼び出しに小さなオーバーヘッドを加える。
- **複雑性:** 規制環境はプロトタイプに現れない要件を加える。スコープ中にそれらを特定するのが、統合をスケジュール通りに保つ規律。
- **リスク:** リスクはプロトタイプがプロダクションに向かうとき集中する。硬编码クレデンシャルを使い・監査ログがなく・中央でロックできないシステムは、規制顧客のセキュリティレビューを通らない。直しは難しくないが、レビューの前に注意が必要。

- **Handles well（うまく扱える）:** 規制顧客が気にするデータに触れる統合で、同じツールが既にエンタープライズ managed settings と audit hooks をサポートするもの。セキュリティ要件を前もってスコープするのはほとんどオーバーヘッドを加えず、最終レビューで統合が止まるのを防ぐ。
- **Adds cost or complexity（コスト・複雑さを加える）:** OAuth フローやエンタープライズシークレット管理に不慣れなチーム。これらのパターンはほとんどの規制組織でセキュリティや IT チームとの調整を必要とし、タイムラインはそれを考慮する必要がある。
- **Use a different approach（別の手法を使う）:** プロダクションデータを決して見ないプロトタイプや概念実証。完全なエンタープライズ統合チェックリストはデモ専用統合には不要だが、シークレットに環境変数の習慣を適用するのはコストゼロで良い実践。

## Screen 16 — Enterprise Integration — Watch Out（注意）· 3 min

### ステージングで働きプロダクションで失敗した OAuth 接続

#### 設定

OAuth 接続はステージングでエンドツーエンドで働いたので、プロダクションに移すのは日常的なカットオーバーに感じた。チームが見逃したのは、OAuth redirect URI がホストごとに登録され、しばしば環境ごとに治められること。だからステージングでの成功は、プロダクションホストがサインインフローを完了するのを認可されたことを意味しなかった。

#### 欠けたステップを表面に出した会話

次のやり取りは、MCP 統合がプロダクションで失敗した後のデプロイ後レビューで起きた。統合はすべてのステージングテストに通っていた。

> **Security reviewer:** "Every production sign-in attempt through the MCP connection is failing. The error is a redirect URI mismatch. Where was the OAuth app registered?"
> **Developer:** "I registered it for staging.mycompany.com during development. We moved to production last week. The connection worked all through staging."
> **Security reviewer:** "That's the issue. The OAuth provider only accepts redirect URIs you've explicitly registered, and production.mycompany.com is not on the allowed list. Every sign-in attempt hits the check, fails the URI match, and loops back to the sign-in screen."
> **Developer:** "So I just need to add the production URI to the app registration?"
> **Security reviewer:** "Yes, and before you do, check whether your staging app registration should be a separate app from production. Most enterprise customers require separate OAuth app registrations for each environment as part of their security policy, so using the same app registration across environments is the second issue I'd flag."

開発者はステージングで OAuth フローをエンドツーエンドでテストし、働くことを確認した。つまりプロダクション失敗はコードの欠陥でない。代わりに、ホストごと・環境ごとに適用される設定ステップであり、開発者はプロダクションでそれをするのを知らなかった。

#### 注意すべきこと

OAuth redirect URI はホストごとに登録される。だから稼働するステージングの OAuth 接続は、プロダクション接続が構成されていることを意味しない。OAuth 認証された MCP 統合を新しい環境に移す前に、新しいホストの redirect URI を OAuth アプリ登録に追加する。規制エンタープライズ顧客には、ステージングとプロダクションで別々の OAuth アプリ登録が必要か確認する。登録ステップをデプロイチェックリストに含め、最初のプロダクションサインイン試行で発見されないように。

## Screen 20 — Recap: Key Takeaways（ふりかえり：7つの要点）· 6 min

### 7つの要点

セクションごとに1つの要点で、モジュールをまとめる。

1. **パーミッションモードは速度の決定でなく、リスクの決定だ。**
Claude Code はすべてにプロンプトを出すものから何にも出さないものまでのモードを与える。パーミッションモードは、プロンプトを減らしたい好みでなく、作業と環境のリスクプロファイルに合わせるべきだ。ライブコードベースに対する開発者ワークステーションでの bypass モードは、エージェントとファイルの間のすべてのチェックポイントを取り除く。触れてはならないパスの deny ルールを project または enterprise レベルで設定すれば、モード単独では覆えない隙間を覆う。

2. **AI コードレビューは、適用すべき評決でなく、トリアージすべき一連の所見を与える。**
レビュアーが目の前の diff から証明できる所見（null チェックの欠落や未クローズのリソースなど）を信頼し、それが引用する行で確認する。実行時の振る舞いや別システムについての主張は、テストすべき仮説として扱う。レビュアーはそれを証明する証拠なしにその主張をしたからだ。所見が元に戻しにくい行動に変わるポイントに人間のゲートを置き、レビュアーがさもなければ推測しなければならない規約を与えて精度を上げる。

3. **skill は移植可能だが、「どこでも走る」は設計するものだ。**
同じ SKILL.md は Claude Code・Messages API・Agent SDK で走るが、それぞれが違って読み込みサンドボックス化する。Claude Code はファイルシステム発見、API は beta ヘッダーと code execution container、SDK は settingSources。明確な description にスコープされ、ローカル環境の想定がない skill は綺麗に移植する。書かれたターミナルを想定するものはしない。すべてのランタイムで、サブエージェントはクリーンに始まる。skills を自動的に事前読み込みしない。

4. **永続的なコンテキストには、各関心に応じた正しいメカニズムが必要だ。**
CLAUDE.md はセッション永続のプロジェクト記憶だが、サイズで薄まる。ルールファイルは指針を適用される場所にスコープする。hooks は確率論でなく決定論的にガードレールを強制する。サブエージェントは探索作業をメインコンテキストの外に保つ。これら4つのメカニズムはそれぞれ違う問題を解くので、すべてを CLAUDE.md に押し込むと、保守しやすく無視しやすい単一ファイルを生む。

5. **共有可能なセットアップには移植可能なコンポーネントが必要だ。**
作者のホームディレクトリへの絶対パスを参照するプラグインは、1台のマシンにインストールされ、他のすべてで失敗する。共有される skills・hooks・プラグインコンポーネントはプロジェクトルートに相対なパスを参照しなければならず、環境変数の要件はドキュメント化またはインストール時に検証しなければならない。配布前にクリーンなマシンからインストールをテストする。

6. **トランスポートとスコープは、依存する帰結を持つ独立の決定だ。**
stdio は自分のマシンで走るサーバー向け。HTTP はリモートでホストされる、または複数の開発者がアクセスするあらゆるもの向け。Local スコープはサーバーを個人のままにし、project スコープは `.mcp.json` 経由でリポジトリと共有する。組み合わせはデプロイの意図に合わなければならない。共有チームサーバーは HTTP トランスポートと project または enterprise スコープを必要とする。`.mcp.json` の stdio サーバーは、共有可能に見えるがそうでない構成だ。

7. **エンタープライズ統合は、デプロイ前にセキュリティ要件を特定することを必要とする。**
規制顧客は identity・データ常駐・アクセスロギング・設定制御について問う。答えは、ユーザー identity サービスには OAuth、サービスクレデンシャルには環境変数、監査ロギングには PostToolUse hooks、設定ロックにはエンタープライズ managed settings から来る。これらはどれも実装は難しくないが、プロダクションデプロイがセキュリティレビューに失敗した後の後付けはすべて難しい。

## Screen 20B — Glossary: Key Terms（用語集）· 3 min

### このモジュールの主要用語

- **Claude Agent SDK**
Claude Code がターミナルで走らせるのと同じエージェントループを公開するプログラム可能なインターフェース。開発者がコードからループを呼び・パーミッションモードと利用可能なツールを設定し・対話セッションなしでタスクを走らせることを許す。ターミナルで適用されるのと同じパーミッションモデルと deny ルールが SDK で適用される。

- **CLAUDE.md**
Claude Code プロジェクトのルートに置かれた Markdown ファイル。内容は毎セッションの開始時にコンテキストウィンドウの前に追加される。すべてのセッションをまたいで無条件に適用すべき、普遍的なプロジェクト制約・規約・コマンドを保持する。約200〜300行を超えて成長するファイルは、内容の重みで重要なルールを薄めるリスクがある。

- **Hook**
Claude Code の実行のライフサイクルイベント（PreToolUse・PostToolUse・UserPromptSubmit・Stop）に束縛されたコマンド。CLAUDE.md の指示と異なり、hooks はモデルが何を決めるかにかかわらず構成されたイベントで決定論的に走る。PreToolUse hook は exit code 2 でツール呼び出しを実行前にブロックできる。

- **MCP（Model Context Protocol）**
Claude Code のような MCP クライアントが、ツール・リソース・プロンプトを公開する MCP サーバーに接続できるようにする、オープンな通信レイヤー。プロトコルはクライアントがサーバーのツールをどう発見し呼ぶかを定義する。MCP を使うと、ツールの定義と保守が個々のアプリケーションコードから、任意の MCP クライアントが接続できる再利用可能なサーバーに移る。

- **MCP transport**
MCP クライアントと MCP サーバーの間の通信チャネル。stdio はサーバーをクライアントと同じマシンのローカルサブプロセスとして走らせる。HTTP はネットワーク越しにリモートホストされたサーバーに接続する。トランスポートの選択が、サーバーがどこで走れるか・誰が接続できるかを決める。

- **Permission mode**
Claude Code の設定で、ツール呼び出しを実行する前にエージェントがどれくらい頻繁に確認を求めて止まるかを制御する。モードは default（ほぼすべての行動の前にプロンプト）から bypass モード（プロンプトなし）まで。deny ルールはあらゆるモードを上書きする。enterprise settings レベルの deny ルールは、いかなる個人の設定でもバイパスできない。

- **Plugin**
marketplace を通じて配布される、Claude Code コンポーネント（skills・hooks・サブエージェント・MCP サーバー設定）のバージョン管理された束。プラグインをインストールすると、受け手は1ステップで作者と同じセットアップを得る。エンタープライズ管理者は managed settings を通じてプラグインを組織全体にデプロイできる。

- **Rules instruction file（ルール指示ファイル）**
Claude Code で、指針を特定のパスや条件にスコープするファイル。毎セッション無条件に読み込まれる CLAUDE.md と異なり、ルールファイルは Claude Code がその管理するディレクトリで作業しているときだけ起動する。パス固有の指針をメインのプロジェクト記憶ファイルの外に保つために使う。

- **Subagent（サブエージェント）**
委譲されたタスクを扱うために Claude Code が立ち上げる別個の実行コンテキスト。サブエージェントはメインの会話のコンテキストや蓄積されたファイルを継承せず、クリーンに始まり、タスクを行い、要約だけを返す。探索や調査作業にサブエージェントを使うと、再利用されない内容でメインのセッションコンテキストが埋まるのを防ぐ。
