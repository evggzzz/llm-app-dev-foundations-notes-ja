## Screen 1 — Orientation（オリエンテーション）· 2 min

### このモジュールを終えるとできること

Claude を使ってコードを書くことと、Claude を使うコードを書くことは違う。

これまで、プロンプトを入力し、応答を読み、必要に応じて調整するという対話的な使い方をしてきたはずだ。このモジュールは、そうした基本の先にあるすべて、すなわちツールスキーマ・コンテキスト管理・エージェントループを扱う。Claude の出力を形作る能力の上に積み上げる。エンジニアとして、Claude をプログラム的に統合し、出力の扱いを信頼できるものにし、堅牢なプロダクション解决方案をデプロイする責任は、あなたにある。

このモジュールの各トピックは、開発中は見過ごされやすいが、開発が進んでからは特定と解決に多大な時間と手間がかかる、特定の失敗モードを扱う。こうした失敗モードを識別し回避する方法を学べば、Claude を開発プロセスに効率的に組み込める。

このモジュールを終えると、次ができるようになる。

1. system prompt・XML タグ・few-shot 例・出力制約（output constraint）を使って本番運用に耐えるプロンプトを書き、初回結果が外れたときにプロンプトがなぜ低調なのかを診断する。
2. extended thinking をいつ有効にするか決め、effort 設定を調整し、ツール使用ターンをまたいで thinking ブロックを正しく扱う。
3. Claude が正しく選択するツールスキーマを定義・実装し、ツール使用ループを構築し、マルチターンのメッセージブロックを扱い、1回のツール呼び出しと複数の並列呼び出しを使い分ける。
4. ストリーミング応答を消費し、ストリームイベントを完全なコンテンツブロックに組み立て、途中で中断されたストリームから綺麗に復旧する。
5. コンテキストウィンドウの管理・圧縮（compacting）・タスク間での履歴クリア・サブエージェントの引き継ぎを含むコンテキストエンジニアリングの技法を適用し、マルチターンのエージェントセッションを、タスクの連続性を失わずに予算内に収める。
6. ワークフローとエージェントのパターンを使い分け、ツールとコンテキストを動くループに繋ぎ、デプロイ制約に合う繋ぎ方を選び、取り消せない操作には Human-in-the-loop（HITL）のチェックポイントを置いて、プロダクションエージェントを構築する。
7. 永続ストレージのパターンと適切なメモリスコープの選択でセッションをまたいでエージェントメモリを管理し、ターンをまたいでエージェントの状態を保ちつつ、コンテキストコストを膨らませない。
8. 正しいメッセージブロック構造で画像と PDF を Claude に送り、再利用可能なアセットには Files API を使い、大量のワークロードは Message Batches API で非同期に完了させる。

このモジュールは、Claude を使ってプロトタイプを、実使用に耐える完全なプロダクションシステムに仕上げる準備のできた Developer 向けだ。実践的で、コードファーストで、パターン志向であることを前提とする。プログラミングの基礎は教えないし、チャットウィンドウで気軽に Claude を使う話でもない。モデルを巡る工学の意思決定、すなわちプロンプトの構成・ツールの定義・ストリーミングの安全な扱い・コンテキストとメモリの管理、そしてデプロイ後に信頼でき・安く・制御可能であり続けるエージェントループの構築を教える。

### このモジュールにおける「ビルド」

このモジュールのすべては、繰り返し現れる1つの工学的な問題を軸に組み立てられている。開発中はうまく動いていた Claude 統合が、プロダクションで持ちこたえなければならない、という問題だ。

開発中は、プロンプトは堅牢に見え、ツール呼び出しは働き、セッションは短く、テスト入力は扱いやすかった。ところがプロダクションでは、同じシステムが、より長いセッション・より大きいツール出力・中断されるストリーム・より厳しいコストとレイテンシの制約・ターンをまたぐメモリ・取り消せない操作に生き残らなければならない。このモジュールは、どの実装上の決定がどのプロダクションの失敗を防ぐかを教える。プロンプトの構成・ツールの定義・ストリーミングの扱い・コンテキストの管理・メモリスコープの選択、そしてエージェントの安全な繋ぎ方を、失敗が顕在化する前に済ませておく。

## Screen 2 — Prompting Craft — Teaching（プロンプトの工夫）· 20 min

### System prompts、XML、few-shot、出力制約

対話で1回うまくいったプロンプトは、本番で未テストの入力に当たると、たいてい壊れる。直し方は言葉を足すことではなく、プロンプトに欠けている構成要素を特定し、それを1つ追加することだ。この画面では、失敗した出力の読み方を振り返り、修正をもたらす4つの技法を順に見る。

### Claude に信頼できる出力の形を与える4つの技法

初回の応答が外れたとき、プロンプトに言葉を足してもう1回、という衝動がよく起きる。しかし、それは問題を切り分けにくくするだけで、まず直らない。言い換えは「どう言うか」を変えるだけで、プロンプトに欠けている構成要素を増やすわけではない。例えば、Claude が指示と入力データの境界を越えているなら、言葉を明確にしても直らない。出力フォーマットが漂い続けるなら、「please format this correctly」と言っても直らない。

失敗モードが、4つの技法のどれが欠けているかを教える。まずプロンプトがどう失敗しているかを診断し、その失敗に効く特定の技法を追加する。

| 何が起きたか | プロンプトに欠けているもの | なぜこの技法が効くか |
|---|---|---|
| 結果が違う形で戻る。ラベルを期待したのに文が来る、JSON を期待したのに散文が来る。 | 出力制約（output constraint）。プロンプトが応答の形・フィールド名・終わり方を一度も指定していない。 | 出力制約は、内容とは独立に応答の形を制御する。これがないと、Claude は下流のパーサーが受け付けない、もっともらしいテキストを返す。 |
| 内容が外れる。スコープが漂う、トーンが変わる、聞いたより広い問いに答える。会話が進むほど悪化する。 | system prompt、あるいはより具体的な system prompt。振る舞いの契約が、ターンをまたいで保つには曖昧すぎた。 | system prompt は、ユーザーターンにかかわらずすべての応答に適用するルールを定める。具体性が足りないと、会話が続くなかで役割・スコープ・フォーマットを留めるものが何もない。 |
| タスクは合っているが、構造がでっち上げられる。何をすべきかは理解し、聞いてもいない形で出力する。 | few-shot 例。Claude は記述だけから正確な構造を推論できない。 | few-shot 例は、記述ではなくパターンを示す。正しい入出力のペア1つで、Claude に合わせるべき正確な形を与える。それは書かれた指示では決まらないことが多い。 |
| テストした入力では綺麗だが、変種で壊れる。エッジケース・珍しいフィールド・想定外の入力。 | 変種を覆う制約。プロンプトは正常系を扱うだけで、パーサーが壊すケースにルールがない。 | プロンプトは狭い入力セットで検証された。制約の中で変種を名指しするか、その変種を覆う例を追加すれば、テスト入力が曝露しなかった隙間が閉じる。 |

### 違う形で戻る分類プロンプトの診断

ルールは単純だ。失敗を名指しし、それに合う技法を1つ追加し、もう1回走らせる。それでも失敗するなら、もう1度診断する。プロンプトが毎回長くなっているなら、それは診断を飛ばして言葉を足している合図だ。

以下のパターンは、表の1行目を実演する。下流コードが受け付けない形で、正しい内容が戻るプロンプトだ。分類器はタスクを理解し、正しいカテゴリを返す。だが答えの形が実行ごとに変わり、それを消費するルーターが壊れる。欠けているのは出力制約で、修正は、もう2つの技法を引いて、ラベルセットを固定しフォーマットを示すことだ。解説は、問題を起こす素のプロンプトから、それを解く制約付き版へと進む。

### 実例：分類プロンプトの before と after

開発者は、サポートチケットを billing・technical・escalation の3カテゴリに分類させたい。最初のプロンプトは、出力に制約を持たない素の指示だ。

```
System: "You are a support classifier. Classify the ticket."

User: <ticket>I was charged twice for the same month.</ticket>
```

Claude は実行によって "Billing"、"billing"、さらには "This looks like a billing issue." のような完全な文を返す。下流のルーターは固定のラベルセットのいずれかを期待し、不一致で壊れる。

これを前の表に当てはめると、1行目の状況だ。出力がパーサーが受け付けない形で戻る。だから欠けているのは出力制約。その制約を追加すると、もう2つの技法も一緒に引っ張ってくる。ラベルセットを固定しフォーマットを示すのは、書かれた指示よりその方がうまくいくからだ。few-shot 例で返すべき正確なラベルと大文字小文字を示し、XML タグでその例を指示と切り離して、Claude が例をタスクの一部として読まないようにする。

```
System: "You are a support classifier. Classify each ticket into exactly one of: BILLING, TECHNICAL, ESCALATION. Return only the label. No other text."

<sample_input>My account shows two charges for April.</sample_input>
<ideal_output>BILLING</ideal_output>

<sample_input>The API keeps returning a 429 error.</sample_input>
<ideal_output>TECHNICAL</ideal_output>

User: <ticket>I was charged twice for the same month.</ticket>
```

ここで3つの技法が、それぞれ別の働きをする。system prompt が出力の契約を定める。固定セットから1つだけ、他は何もない。XML タグが各例の終わりと次の始まりを印し、Claude が例を指示の一部として読まないようにする。few-shot のペアが、記述ではなく正確な大文字小文字とフォーマットを示す。一緒になると、プログラム的にルーティングできるほど一貫した結果が出る。

下の表は、4つの技法をどう積むか、どこでプロンプトを単純化すべきか、そして追加の前に診断すべき場所を示す。

| やり方 | いつ |
|---|---|
| 4つすべてを積む | 明確に定義された出力契約に対し、4つすべてを積む。フォーマットが明確に決まっていて、例で覆えるエッジケースのあるタスク。 |
| プロンプトを単純化する | 1つしか要らない単純なタスクに、4つすべてを足すこと。要約プロンプトに few-shot 例も出力スキーマも要らない。 |
| 追加の前に診断する | 反復ごとに正確になるのではなく長くなっていくプロンプト。5回書き直しても出力がまだ違うなら、言葉を足す前に失敗タイプを診断する。 |

### 各技法をいつ使うか

4つの各技法と、それぞれの適所を見ていく。

**System Prompts（システムプロンプト）**
system prompt は、セッション全体の振る舞いの契約を担う。1回書き、永続する指示レイヤーとして扱う。Claude の役割・出力フォーマット・会話間で変えてはならないルールを定義する。

**XML Tags（XML タグ）**
XML タグは、プロンプトが入力と指示を混ぜるときに使う。提供したドキュメントを使って Claude にコードをデバッグさせるプロンプトが良い例だ。タグなしだと、コードとドキュメントが Claude には同じに見える。

`<my_code>` や `<docs>` のような説明的なタグ名で囲めば、境界は曖昧でなくなる。公式の XML タグ名を使う必要はない。内容に合う説明的な名前が最もよく働く。

**Few-shot Examples（few-shot 例）**
few-shot 例が有用なのは、語るのではなく見せるからだ。求める正確なフォーマットを記述しようとする代わりに、正しい入出力のペアを1つ与え、Claude にパターンを推論させる。使うときは、`<sample_input>` と `<ideal_output>` のように一貫した XML 構造で例を囲み、例とプロンプトの境界を明確にする。ゼロから書かずに、スコアの最も高い評価出力から例を拾える。

**Output Constraints（出力制約）**
出力制約は、Claude の応答がパーサーに届く前の最後の防衛線だ。必要なものを正確に指定する。フィールド名・型・長さの上限・前置きを含めるか・データがないときどうするか。フォーマットを機械可読にしたい場合は、構造化出力の機能を使う。

### 反復ループ：再プロンプトの前に診断する

初回の応答が外れたとき、プロンプトに言葉を足してもう1回、という衝動がよく起きる。その衝動は問題を診断しにくくすることがほとんどで、まず直らない。

代わりに、まず問題を診断し、その発見に基づいて再プロンプトする。失敗タイプが、どの技法が欠けているかを教える。

- 違うフォーマット：出力制約の欠落が原因。プロンプトが結果の形を一度も指定していない。
- 違う内容やスコープの漂い：system prompt の具体性不足が原因。振る舞いの契約が会話をまたいで保つには曖昧すぎた。
- タスクは正しいが構造が幻覚：few-shot 例が必要なときに起きる。Claude は記述だけから正確な構造を推論できない。
- 単純な入力では良好だがエッジケースで壊れる：プロンプトは正常系を扱うだけで、パーサーが壊す変種を覆う制約がない。

修正は構造的で、言い回しの問題ではない。Claude が指示とコンテンツの境界を無視しているなら、言葉を明確にしても直らない。出力フォーマットが漂い続けるなら、「please format this correctly」と言っても直らない。いずれの場合も、4つの技法のどれが欠けているかを特定し、それを追加する。

### 出力制御をプロンプトから API へ：構造化出力

ここまでは、プロンプトに指示を書き込み、Claude が従ってくれることを期待して出力を形作ってきた。それはたいていうまくいく。しかしプロンプトは「依頼」なので、モデルは余分な1文・違うフィールド名・malformed な JSON を返し、下流のパーサーを壊すことがまだある。

Claude API には、プロダクションコード向けにその隙間をなくす別の仕組みがある。**structured outputs（構造化出力）**で、言葉で形を頼む代わりに、API に JSON スキーマを渡し、生成時にモデルがそれに合致する出力だけを出すよう制約する。この技法は **constrained decoding（制約付き復号）**だ。Claude が各トークンを生成するとき、API は出力をスキーマに対して有効に保つトークンだけを許す。だからスキーマに違反する応答は、そもそも生成されない。

構造化出力は、実パイプラインで現れる2つの状況を覆う。それぞれモデルが返すものの別の部分を制約し、単独でも、同じリクエストで併用しても使える。

**JSON outputs** は最終応答を制約する。`output_config.format` パラメータに `type: json_schema` とスキーマを設定すると、Claude は毎回スキーマに合致する有効な JSON を応答テキストで返す。モデル自身が、サポートチケットからのフィールド抽出や API 応答のフォーマットなど、コードが消費する構造化ペイロードを生成する場面で使う。さもなければ各呼び出しの周りに書く parse-and-retry のコードをなくせる。

**Strict tool use** は、Claude がツールに渡す入力を制約する。ツール定義に `strict: true` を設定すると、コードが動く前に、Claude がそのツールに送る引数が入力スキーマに対して検証される。malformed なツール引数が関数をクラッシュさせたり誤操作を引き起こしたりするエージェントループで使う。コードが受け取る呼び出しが、定義した契約に既に適合していることを保証できる。

これがプロンプトでなくプロダクションコードに属する理由は、テストしなかった入力での信頼性だ。プロンプトレベルの「JSON のみを返せ」という指示は、試したケースでは保ち、試さなかったエッジケースで外れる。それは前に見た分類の例が歩んだ失敗そのものだ。スキーマ制約は外れない。API が、モデルに指示を覚え込ませるのではなく、すべてのトークンで執行するからだ。これにより、出力の正しさは、事後に検証するものから、事前に API が排除するものへと移る。

生成の制約にはコストがあり、プロダクションで選ぶ開発者は、どこでもデフォルトで有効にするのではなく、コストを秤にかける必要がある。考慮すべきコストを以下に挙げる。

- 新しいスキーマの最初の要求は遅い。API は出力を制約する前にスキーマを文法へコンパイルし、そのコンパイルが最初の呼び出しにレイテンシを加える。コンパイルされた文法は最終使用から24時間キャッシュされる。だから安定したスキーマへの定常トラフィックは1回だけコストを払うが、スキーマを絶えず変えるワークロードは繰り返し払う。
- 入力トークン数が増える。構造化出力がオンのとき、API は期待されるフォーマットを記述する system prompt を追加し、その注入されたプロンプトは他の入力トークンと同じように課金される。呼び出しあたりの増加は小さいが、ボリュームでコストを見積もるときには知っておく価値がある。
- 保証されたスキーマは、保証された成功ではない。それでもスキーマに合わない出力を返すケースが2つある。1つは refusal（安全性の理由でモデルが辞退し、応答が `stop_reason: refusal` を持つ）。もう1つは truncation（応答が `max_tokens` 上限に達し、構造の途中で止まり、`stop_reason: max_tokens` となる）。コードは、すべての応答がパースできると仮定するのではなく、やはり `stop_reason` をチェックする。
- メッセージの prefilling とは組み合わせられない。JSON outputs と assistant メッセージの prefilling は両立しない。だから、Claude の応答を始めるパターンと、応答全体をスキーマに制約するパターンは、同じリクエストでは動かない。タスクに合う方を選ぶ。

## Screen 3 — Prompting Craft — Watch Out（注意）· 5 min

### よくならず、長くなる一方だったプロンプト

#### 設定

プロンプトは本番準備ができたように見えても、静かな失敗を生むことがある。エッジケースがフィールドを欠落させたり制約を無視させたりする。それは制約が十分に正確に指定されていなかったときによく起きる。

#### 6回の改訂、毎回前より長く

前の例で使ったプロンプトを再掲する。開発者はサポートチケットを billing・technical・escalation の3カテゴリに分類させたい。最初のプロンプトは素の指示だ。

```
System: "You are a support classifier. Classify the ticket."
User: <ticket>I was charged twice for the same month.</ticket>
```

下のトレースは、分類プロンプトを反復する開発者を示す。各パスは言葉を足すが、出力は漂い続ける。このパターンは、開発者が制約の指定を意識せずにプロンプトに足し続けると現れる。

| パス | 追加したもの | 出力の振る舞い |
|---|---|---|
| 1 | "Classify this ticket as billing, technical, or escalation." | 完全な文を返す。"This appears to be a billing issue." パーサーが壊れる。 |
| 2 | "Be concise." と "Use only the category name." を追加 | "Billing" と大文字で返すことも、"billing" と小文字で返すこともある。ルーターが大文字小文字の不一致で壊れる。 |
| 3 | 各カテゴリを詳しく説明する3段落を追加 | 単純なチケットでは正しい。曖昧なチケットでは、単一ラベルでなく 'billing/technical' を返す。パーサーがスラッシュで壊れる。 |
| 4 | "Never return two categories." と "If ambiguous, choose the most likely one." を追加 | チケットの80%で動く。妥当的に2カテゴリに当てはまるチケットで失敗する（例：請求されたが機能も止まった）。ここではラベルでなく完全な説明を返す。 |
| 5 | エッジケースに関する段落をさらに2つと、正確であれという念押しを追加 | 冗長なプロンプトが冗長な出力を生む。1呼び出しあたり2000文字超。長く焦点のぼやけたプロンプトは、長く焦点のぼやけた出力を作りがちだからだ。モデルは入力に合わせて応答の長さとスタイルを調整する。出力長のせいでレイテンシが大きく上がったが、精度は上がっていない。 |
| 6 | すべての指示を JSON スキーマと、正確な入出力ペアを示す few-shot 例2つに置き換え | 毎回 `{"category": "billing"}` を返す。パーサーが動く。レイテンシが下がる。曖昧なチケットでの精度がパス4に並ぶ。 |

この6パスで、2つのことが別々に間違った。別々に識別する価値がある。パス4は診断の失敗だ。開発者は違う問題を特定し、制約でなく記述を足し、出力は壊れたままだった。パス5は工学の失敗だ。プロンプトがレイテンシの後退を引き起こすほど冗長になり、モデルは入力に合わせて応答長を調整し、1呼び出しあたり2000文字超を生成した。精度の向上はない。両方の直し方は同じ構造的な一手だ。出力制約と few-shot 例2つ。しかし、これらが2つの違う失敗だと認識することが重要である。2つ目は1つ目が解決済みでも現れうるからだ。出力制約と few-shot 例を適用したプロンプトを示す。

```
System: "You are a support classifier. Classify each ticket into exactly one of: BILLING, TECHNICAL, ESCALATION. Return only the label. No other text."

<sample_input>My account shows two charges for April.</sample_input>
<ideal_output>BILLING</ideal_output>

<sample_input>The API keeps returning a 429 error.</sample_input>
<ideal_output>TECHNICAL</ideal_output>

User: <ticket>I was charged twice for the same month.</ticket>
```

#### 注意すべきこと

すべてのパスがプロンプトを長くし、どれも欠けていた出力制約を追加しなかった。開発者は反復ごとにより正確に問題を記述した。しかし、Claude が求めているのは請求チケットがどう見えるかの詳しい記述ではなく、許容される応答がすべて大文字の1語だけだということだ。

直しは2行だ。正確なフォーマットを指定する出力制約と、曖昧なケースを覆う few-shot 例。6パスのトレースは、早く認識すべきパターンだ。連続する3回の再プロンプトが効いていないなら、言葉を足すのをやめ、どの技法が欠けているかを診断する。

## Screen 5 — Extended Thinking — Teaching（拡張思考）· 12 min

### Extended Thinking：推論をオンにし、effort を調整し、正しく読み戻す

プロンプトの技法は、Claude が何を生成するかを形作る。extended thinking は、Claude が答える前にどれだけの仕事をするかを形作る。オンにすると、モデルはまず段階的な推論を書き出し、それから最終の答えを返す。あなたの仕事は、その追加の仕事がコストに見合うときを決め、モデルが送り返してくる推論を扱うことだ。

#### extended thinking がすること

extended thinking をオンにすると、モデルは応答する前に「考えを声に出す」。その推論は、API 応答の中に固有の **thinking block** として戻り、実際の答えを保持するブロックの直前に置かれる。最新モデルでは、thinking block の内容はデフォルトで省かれる。読めるようにするには、display setting で読める要約をリクエストする。

現在のモデルでは、推論は適応的（adaptive）だ。デフォルトでオンでない場合に `thinking` パラメータで有効にし、モデルが各リクエストにどれだけ推論が必要かを決める。深さは、固定のトークン予算ではなく **effort setting** で調整する。古い `budget_tokens` 制御は非推奨で、最新のモデル世代では 400 エラーを返す。

その推論は無償ではない。thinking tokens は output tokens と同じコストがかかる。だから単純なタスクを高い effort で走らせれば、不要な精度にお金を払うことになる。ここでの選択は、すでにしてきたものと同じだ。道具をタスクに合わせる。extended thinking をデフォルトで手伸ばししない。必要なところに戦略的に適用する。

#### extended thinking をいつ使うか

| タスクの形状 | extended thinking の判断 | 理由 |
|---|---|---|
| モデルが同時に複数の制約を保つ必要がある多段推論。数学の導出、複数ホップの論理問題、依存する一連の行動の計画。 | 有効にする。effort レベルを問題の深さに合わせる。 | 推論のパスこそ、モデルがさもなければ飛ばす依存関係を処理する場面だ。 |
| 機械的・参照系のタスク。分類、フォーマット変換、フィールド抽出、短い事実の応答。 | オフのままにする。 | extended thinking は答えを良くしない。不要なものにより多くのトークンを払うことになる。出力制約を持つ素のプロンプトが正しい道具だ。 |
| モデルが複数のツール呼び出しをまたいで計画するエージェントループ。 | 有効にし、呼び出しごとでなく計画ステップに予算を割く。 | 計画の前の推論は、下流での誤ったツール選択を減らす。後述の carry-back rule に注意。これはすべてのツール使用ループに当てはまる。 |

#### carry-back rule：thinking block は変更せず API に戻す

extended thinking がオンで、会話がツールを使うとき、飛ばせないルールが1つある。受け取ったすべての thinking block は、次のターンで、届いたときと exactly に API に戻さなければならない。各ブロックは **signature** を持ち、推論が改竄されていないことを確認する。それを編集・要約・削除すると、signature が合わなくなり、API はリクエストを拒否する。

redacted thinking block も同じように扱う。内容は暗号化され、人間が読むことを想定しないが、それでも手つかずで戻さなければならない。

これは構造的な要件で、あなたが選べるプロンプトの選択ではない。最も一般的な失敗は、コンテキストを節約するために thinking block を剥ぎ取ることだ。それは次のリクエストを壊す。蓄積した推論でコンテキストが積み上がることの方が本当の懸念なら、直しはこのモジュールで後述するコンテキストエンジニアリングの作業だ。

#### 先のポインタ

このレッスンは推論を有効にし、effort setting を調整する。モデル選択は扱わない。推論を有効にするかどうかとは別の、どのモデルを実行するかの選択は、これより前の MSO Foundations モジュールで教える。

#### まとめ

- **Handles well（うまく扱える）:** 間違った答えが高くつき、追加トークンが精度を買う、難しい推論・計画タスク。
- **Adds cost or complexity（コスト・複雑さを加える）:** ツール使用ループでの carry-back 要件と、これから調整しなければならない effort setting。
- **Use a different approach（別の手法を使う）:** 分類・抽出・フォーマットのタスクでは、よく制約されたプロンプトの方が安く、同じく正確。

## Screen 7 — Tool-use and Schema Design — Teaching（ツール使用とスキーマ設計）· 20 min

### Claude が正しく選ぶツールスキーマ：定義・ループ・呼び出しパターン

ここまでは、Claude が何を生成するかを形作る仕事だった。リクエストを枠付け、例を与え、欲しい出力に合う技法を選ぶ。ツール使用では、もう良い答えへとことばを導くのではない。Claude に一連の行動を渡し、正しいものを選ぶことを信頼する。その選択は、スキーマに何を書いたかによってほぼ完全に決まる。

#### ツール使用ループの仕組み

ツール使用で最も一般的な誤解は、Claude がツールを実行するというものだ。実際には、Claude はツール定義を読み、状況に合うものを決め、必要な入力とともにアプリケーションに何を呼ぶかを伝える。アプリケーションがツールを実行し、結果を得て、それを送り返す。そして Claude がその結果を使って続ける。

このやり取りをプロダクションで無視してはならない。アプリケーションが戻りを正しく扱わないと、Claude は求めたデータを受け取れず、ループが壊れる。Claude が担うものとコードが担うものの境界こそ、ツール使用のバグが最も住み着く場所だ。ツール使用を正しく実装するためのシーケンスを以下に示す。

各ステップをクリックすると何が起きるか見える。

1. Define schema
2. Send message
3. tool_use block
4. Execute tool
5. Return result
6. Claude continues

**Define schema**: name・description・input_schema を持つスキーマを定義する。Claude はこれを読んで、ツールを呼ぶか・いつ呼ぶかを決める。

ループは自動ではなく、4番目のステップを自分で完了させる必要があることに注意。見落としが組繙的なら、直しはスキーマ定義のステップにある。

#### ツール使用会話のメッセージブロック構造

ツール使用の会話は、平文ではなく構造化ブロックで組み立てられる。各 assistant ターンと user ターンはブロックのリストで、4つのブロック型がツール使用セッションの仕事を担う。**text block** は Claude の散文応答を運ぶ。**tool_use block** はツール呼び出しを運び、ツール名・一意の ID・入力引数を含む。**tool_result block** は、コードがツールを実行した後に返したものを運ぶ。**thinking block** は Claude の内部推論を運び、extended thinking が有効なときだけ現れる。

API はこれらのブロック間に特定のペアリングを強制する。assistant ターンのすべての tool_use block は、直後の user ターンで、一致する ID を持つ tool_result block で応答されなければならない。ID が合わない、結果が欠けている、ターンの順序が狂っていると、リクエストはバリデーションに失敗する。これはプロンプトの調整で直せるものではない。構造的で、コードが毎リクエストで正しくこのシーケンスを生成しなければならない。

| ブロック型 | 役割 | 内容 | 重要なルール |
|---|---|---|---|
| text block | Assistant/Claude | Claude の散文出力 | Claude は同じターンで tool_use block と並んで text block を返すことがある。そのときコードは、そのターンを会話履歴に追加する際、text block を含む content 配列全体を保たなければならない。text block を落とすと、後続ターンで Claude が依存するコンテキストが壊れる。 |
| tool_use block | Assistant/Claude | ツール名・一意の ID・関数に渡したい入力引数 | すべての tool_use block は、直後の user ターンで tool_result block で応答されなければならない。tool_result は同じ ID を持つ。そのペアリングがないと、API は次のリクエストを拒否する。 |
| tool_result block | User | 一致する tool_use ID・結果の内容・ツール呼び出し失敗時に true にする任意の is_error フラグ | tool_use_id の値は、元の tool_use block と exactly に一致しなければならない。Claude はこの ID で各結果をそれを生んだ呼び出しに結び戻す。1つの assistant ターンが複数のツール呼び出しを発行し、結果が違う順で届くときに重要だ。 |
| thinking block | Assistant（extended thinking のみ）/Claude | Claude の内部推論。extended thinking が有効なときだけ見える | このブロックは後続ターンで変更せず API に戻さなければならない。signature が推論の改変を検証する。編集や要約は signature を壊し、API はメッセージを拒否する。redacted thinking block も同じルールに従う。内容は暗号化され人間が読めなくても、受け取ったまま戻す。 |

重要な不変条件は、assistant ターンのすべての tool_use block が、直後の user ターンで対応する tool_result block を持つことだ。欠けた tool_result block や、直後でなく後のターンに現れる tool_result block は、API のバリデーションエラーを引き起こす。

#### スキーマの解剖：Claude がツール選択の決定のために読むもの

ツールスキーマは name・description・input_schema の3部からなる。description が、Claude がツールを正しく選ぶかを決める。

- **Name**: 具体的であるべき短い識別子。例えば get_account_balance は get_data より Claude に有用だ。
- **Description**: Claude がツールが必要かを決めるために読む、重要な部分。description は常に2部、いつ使うかといつ使わないかで書くべきだ。
  - 「use this to find information」という description は、誤選択を引き起こす。Claude はそれを、何かを取得する他のツールと区別できないからだ。
  - 「use this to retrieve the current balance for a specific account ID and do not use this for transaction history」という description は、Claude に排除条件を与え、適切に説明的だ。
- **input_schema**: JSON Schema を使って、ツール関数が受け付ける入力（パラメータ）を定義する。
  - Claude がツールを正しく呼ぶために必要なパラメータは required とマークすべきだ。
  - ツールがなくても動けるパラメータは optional とマークできる。ツール間でパラメータの型が重なることが、誤ツール呼び出しの最も一般的な原因だ。

#### 決定表：スキーマ設計の選択

スキーマは、Claude がどのツールを呼ぶか・どんな引数を渡すか・応答に十分な情報を持つかを決めるために読むものだ。曖昧で・説明不足で・必須フィールドを欠くスキーマは、構文的には正しく見えても、誤ったツールを選び・malformed な入力を渡し・不必要にループするツール呼び出しを生む。以下の5つの決定が、実装が実条件下で予測可能に振る舞うかを決める。

| 決定 | どう扱うか | なぜ重要か |
|---|---|---|
| サブタスクの依存 | あるツールの出力が次を食べるとき、呼び出しは逐次に走らなければならない。2番目の呼び出しは最初の結果が戻るまで構築できないからだ。サブタスクが互いに独立なら、Claude が1ターンで複数の tool_use block を発行し、コードがそれらを並行実行するようツールセットを構築できる。 | スキーマの設計を変える唯一の決定だ。現在の Claude モデルは、呼び出しが独立なら並列呼び出しをデフォルトにする。本当の依存がある場合は、次の呼び出しを構築する前に最初の結果が利用できるよう、別ターンとしてモデリングする。必要なら `disable_parallel_tool_use` で1ターン1ツール呼び出しを強制する。 |
| 必須フィールド | 呼び出しがそれなしで成り立たないときだけ required とマークする。input schema の required 配列に置く。 | すべてを required にすると、Claude は埋める根拠のないフィールドの値をでっち上げる。required 配列は、どの入力が譲れないかを Claude に伝える手段だ。 |
| 任意フィールド | デフォルトが妥当なパラメータや、不在に意味があるパラメータには任意フィールドを使う。required 配列から外し、関数シグネチャでデフォルトを与える。 | 任意フィールドにより、Claude は持っていない情報を推測する代わりに省略できる。フィールドが任意なのに required とマークされていると、毎回の呼び出しで値をでっち上げ、悪い入力を引き起こす。 |
| description の長さ | ツールごとに3〜4文で、何をするか・Claude がいつ手伸ばしすべきか・何を返すかを書く。フォーマットが重要な場合は有効な入力の例を含める。 | description が短すぎると、Claude はあなたのツールを他から区別するシグナルが足りず、推測する。長すぎると、トリガー条件が決定時に Claude が参照しない詳細の下に埋もれる。 |
| 重なるパラメータの型 | 2つのツールが同じパラメータの形を受けるとき、各 description に、ツールが意図するドメインやトリガーを名指しする曖昧さ回避の言葉を加える。 | Claude は名前と description でルーティングし、パラメータの型は二次シグナルだ。シグネチャが同一だと、ルーティングは description だけに崩れ、似た響きの description は区別できなくなる。 |

#### 実例：誤ツール選択を起こすスキーマとその修正

これは、ツール使用実装で見られる一般的なパターンに基づく例だ。ツール名・description・テスト結果は、選択の曖昧さ回避の原則を示すために構築したもので、特定のプロダクションシステムから引いたものではない。

開発者は search_knowledge_base と get_cached_result の2ツールを登録する。ツール名は異なるが、Claude のツール選択は description に重みを置く。description が重なると、名前だけでは曖昧さを解消するのに十分ではない。両方とも "use this to find information" で始まる description を持つ。排除条件がないと、開発テスト中、曖昧な入力で Claude は頻繁に誤ったツールを選んだ。

問題は、選択の決定が行われる地点で、両方の description が Claude には同じに見えることだ。直しは、description ごとに追加の1文を加えることだ。

```
search_knowledge_base: "Use this to search the knowledge base when the user asks a question that requires looking up current information. Do not use this if the result of a prior search in this session already covers the question."

get_cached_result: "Use this to retrieve a result that was already fetched during this session. Only use this if search_knowledge_base was called earlier in this conversation for the same query."
```

排除条件が、見た目が同じ2つの選択肢ではなく、決定ルールを Claude に与える。これらの条件は、毎リクエストで完全な会話履歴が渡されることに依存する。前のターンが切り詰められたり落とされたりすると、Claude はそれらを評価できず、排除ロジックは黙って失敗する。

登録するツールを1つ増やすごとに、Claude が推論すべき表面積が増える。だからこの規律は、元のツールが distinct なときにだけ報われる。

- **Handles well（うまく扱える）:** description が具体的で排除条件が述べられているとき、Claude を確実に正しいツールへルーティングする。
- **Poor fit（不向き）:** 似たことをする2つのツールで、切り離すためにますます長い description が必要な場合。その時点で、型パラメータを持つ1つのツールに統合する。

#### 誰かが既にツールを書いているとき：手書きスキーマの代替としての MCP

前の節はすべて、あなたがツールスキーマを自分で書く（name・description・input_schema・Claude が tool_use block を発行したときに実行される関数）と想定している。多くの統合では、その必要はない。**Model Context Protocol（MCP）**は、ツール定義と実行をアプリケーションコードから外し、専用サーバーに移す、標準化された通信レイヤーだ。到達したいサービス向けの MCP サーバーが存在すれば、自分で統合を構築する代わりに、その MCP サーバーに直接接続できる。

GitHub 統合を具体例に取る。GitHub はリポジトリ・プルリクエスト・issue・プロジェクト等を公開する。このモジュールのツールスキーマ手法で完全な統合を構築するには、その機能の各部品ごとにスキーマと実行関数を書き、GitHub の API が進化するにつれて保守しなければならない。GitHub 向けの MCP サーバーは、それを既に済ませている。だからアプリケーションはサーバーに接続し、利用可能なツールの完全なリストを受け取り、Claude は、すでに使ってきたのと同じ description ベースのルーティングでそれらから選ぶ。根本の仕組みは同じだ。変わるのは、誰が書き、誰がツール定義を所有するかだ。

#### MCP がツール使用ループにどう合うか

このモジュールで前に構築したループは、MCP を導入しても変わらない。Claude は tool_use block を発行し、アプリケーションはツールを実行して tool_result を返し、メッセージブロックのペアリングルールは引き続き適用される。違いはセットアップステップだ。自分が書いたスキーマを登録する代わりに、MCP クライアントが MCP サーバーに ListToolsRequest を送り、完全なツールリストを受け取り、その定義を Claude に渡す。Claude の視点からは、それらのツールは手で書いたものと区別がつかない。

1つ実用的な含意を書いておく。MCP サーバーは、ツールが現在のターンで使われていなくても、ツール定義をコンテキストウィンドウに追加する。複数のサーバーを一度に接続すると、最初のメッセージが届く前に、ツール定義自体が予算を消費する。このモジュールで前述したスキーマ設計の規律はここでも当てはまる。アクティブに使っているサーバーだけを登録し、同じセッションで複数サーバーを接続する場合は、ウィンドウ上限に対してコンテキストコストを確認する。

API の MCP Connector を使う場合、tools 配列の `mcp_toolset` オブジェクトで読み込みコストを制御する。mcp_toolset は `default_config` ブロックを持ち、サーバー上のすべてのツールに適用される。また、ツール名でキー付けした `configs` で個別ツールを上書きできる。コンテキストコストに関わる2つの設定がある。

- **`defer_loading` boolean**（default_config または configs のツールごとのエントリ内）は、ツール定義の読み込みをモデルがそれを必要とするまで遅らせる。ツールリストが大きいサーバーを接続するとき、前払いのコンテキストコストを減らす。
- **`enabled` boolean** は個別ツールをオン/オフする。サーバーを登録しつつ、モデルに見せたいツールだけを公開できる。MCP Connector は、リクエストに `mcp-client-2025-11-20` beta ヘッダーを設定することを要求する。

そのヘッダーがないと、mcp_toolset の設定はここに書いたようには適用されない。

この段階で知っておく価値のあるもう1つは、クライアントが実際にサーバーとどう話すかだ。MCP は2つのトランスポートのいずれかで動く。どちらを使うかはサーバーがどこにあるかによる。ローカルサーバーは **stdio** を使い、アプリケーションはサーバーをサブプロセスとして生成し、標準入出力で通信する。リモートサーバーは **Streamable HTTP** を使い、アプリケーションは HTTP 経由でネットワーク接続し、クライアントからサーバーへのメッセージには POST を、サーバー起点のメッセージには任意の GET ベースの SSE ストリームを使う。SSE のみの古いトランスポートが存在するが非推奨で、新しい統合は Streamable HTTP を使うべきだ。API で Anthropic の MCP connector を使う場合、1つ制約を示しておく。connector がサポートするのは HTTP で公開されたサーバーだけで、stdio サーバーは SDK 経由で自分で MCP クライアント接続を管理する必要がある。接続が確立しツール定義を受け取れば、アプリケーションコードは両トランスポートを同じように扱う。

- **Use MCP when（MCP を使うとき）:** 必要なサービス向けに、よく保守された MCP サーバーが既に存在する（必要な特定の操作を覆い、サービスの現在の API に対して活発に保守されていることを確認）。自分でそれらのスキーマを書き・所有することは、追加の能力なしに実装のオーバーヘッドを加える。Claude API の MCP Connector はリモートサーバーだけをサポートすることに注意。ローカルの stdio サーバーは Claude Desktop か Claude Code をクライアントとして必要とし、API 経由では直接接続できない。
- **Write schemas manually when（手書きするのは）:** どの MCP サーバーもユースケースを覆わないとき、または、汎用サーバーが提供しない、ツールのスコープと description の品質への精密な制御が必要なとき。スコープ制御のために手書きにデフォルトする前に、API MCP Connector が MCPToolset 設定経由でサーバーごとに特定ツールの allowlisting/denylisting をサポートすることに注意。description の品質には手書きがまだ正当化されるが、スコープでは常にそうとは限らない。
- **Use both when（両方を使うのは）:** 幅のために MCP サーバーに接続し、このモジュールで前述した description 調整の規律を、アクティブにルーティングしている特定のツールに適用する。MCP と手書きスキーマは相互に排他的ではない。サーバーが網羅性を与え、description が重要なところで精度を与える。MCPToolset 経由のツール allowlisting を適用して、description 調整を重ねる前に Claude が推論する表面積を限る。ツールセットを狭めることと description を鋭くすることは2つの別のレバーで、両方を使うべきだ。

## Screen 8 — Tool-use and Schema Design — Watch Out（注意）· 5 min

### Claude を誤ったツールへ送った description

#### 設定

スキーマは正しく見えても失敗することがある。型付きパラメータと正常系のテスト通過は、構造が有効であることを教えるが、入力が2つの重なる description の境界付近にあるとき、Claude がツール間で確実に選べるかは教えない。それが初期テストが見逃し、プロダクションで最も表面化しそうな失敗モードだ。

開発者はコードレビューに入って3時間、内部チャネルの会話をデバッグセッションに貼り付ける。これは、開発者のデバッグ会話の一般的なパターンに基づく複合的なやり取りだ。対話は、description の重なりが名指しされる診断の瞬間を示すために構築したもので、特定のコードレビューから書き起こしたものではない。

朝から誤ったツール選択をデバッグし続けていた開発者の後に起きた、以下のやり取りを見る。シニア開発者が1つ問い、問題全体を再枠付けする。

> **Developer:** "Why does Claude keep calling search_docs when the answer is already in the context? I've re-run this four times, and it keeps going to the wrong tool."
> **Senior Developer:** "What does the description for search_docs say?"
> **Developer:** "'Use this to find information about the product.'"
> **Senior Developer:** "And what does get_context_summary say?"
> **Developer:** "'Use this to retrieve relevant information from the current session.'"
> **Senior Developer:** "Those descriptions are the same thing from Claude's perspective. Both say, 'find information.' One of them needs to say when not to call it."
> **Developer:** "So, I need to add an exclusion?"
> **Senior Developer:** "Right. Try using search_docs 'when the user asks a question that requires looking up content not already present in this conversation. Do not call this if the answer is available in the current session context.' Then get_context_summary handles the in-context case. You will want to tighten get_context_summary's description the same way: add 'Only use this if the answer is already present in the current session. Do not use this to look up new information.' Both tools need the boundary, not just one."
> **Developer:** "That's two sentences."
> **Senior Developer:** "Right. One to say when to use it, one to say when not to. That's the whole fix."

#### 注意すべきこと

Claude は、完全な会話のコンテキストで、登録されたすべての description を推論してツールを選ぶ。2つの description が似ているとき、その推論には区別する信頼できるシグナルがない。だから Claude は、あなたが意図した区別に対応しないかもしれない小さな表面の差に基づいて選ぶ。

失敗が重なる description のとき、直しは一貫している。ツールをいつ呼ばないかを名指しする1文を加え、Claude に決定境界を与える。排除条件を加えても description を綺麗に分けられないなら、ツールを型パラメータを持つ1つに統合する必要があるかもしれない。

## Screen 10 — Streaming Responses — Teaching（ストリーミング応答）· 16 min

### 状態を壊さずに部分出力を扱う

ここまでのすべてのリクエストは、応答全体が届くのを待ってから処理してきた。それは、応答が長くなるか、ユーザーが空白の画面を見つめて座っているまでは問題ない。ストリーミングは、応答を小片で、モデルが生成するのに応じて送る。体感は速くなるが、コードに新しい仕事を与える。一連の出力に基づいて最終コンテンツを自分で組み立て、その系列が途中で止まった場合に備えなければならない。

#### ストリーミングが応答について変えること

非ストリームのリクエストでは、API は完全に形作られたすべてのコンテンツブロックを持つ1つの完全なメッセージを渡す。ストリームのリクエストでは、API は代わりに、メッセージが構築される様子を記述する一連のイベントを送る。コードはその系列を聞き、ブロックを再組み立てする。最終的に得られるメッセージは非ストリーム呼び出しが返すものと同じだ。違いは、小片を自分で組み立てなければならないことと、メッセージが完成する前にイベントが止まったらどうするかを自分が決めることだ。

何が起きていないかを知っておくと役立つ。モデルが何か生きたオブジェクトを開いたままにしているわけではない。各イベントは、単一の変更を記述する独立した小さなメッセージだ。ブロックが始まった、テキストや入力が追加された、ブロックが終わった、メッセージ全体が終わった。ハンドラは各イベントを受け取り、それまで構築してきた部分状態に適用する。

#### イベント系列と、各イベントでハンドラがすること

| イベント | 何を合図するか | ハンドラがすること |
|---|---|---|
| `message_start` | 新しいメッセージが始まる。空の content と初期 usage を持つメッセージの殻を運ぶ。 | ブロックを集める空の content 配列を用意する。 |
| `content_block_start` | 新しいコンテンツブロックが開く。その型（text・tool_use・thinking）と index を伴う。 | その index に、名指しされたブロック型のスロットを作る。tool_use block は名前と id で開くが、入力はまだない。 |
| `content_block_delta` | 1つのブロックの増分：テキストの断片、ツール呼び出しの JSON 入力の断片、thinking の断片。 | その index のブロックに断片を追加する。ツール呼び出しの入力は、複数の delta に広がる partial JSON 文字列として届く。ブロックが閉じるまでパースできない。 |
| `content_block_stop` | この index のブロックが完成した。 | ブロックを確定する。tool_use block では、これが蓄積された JSON 入力がパースできるほど完全になった最初の瞬間だ。 |
| `message_delta` | メッセージへのトップレベルの変更：stop_reason と最終 usage カウント。 | stop_reason を記録する。モデルが終わったのか、別の理由で止まったかを教える。 |
| `message_stop` | ストリームが完成した。 | 組み立てた content 配列が、完成したメッセージになる。ここからは、非ストリーム応答と全く同じように扱う。 |

#### 状態が壊れるのを防ぐルール：partial block に作用しない

tool_use block が注目すべきものだ。その入力は、多くの content_block_delta イベントに広がる partial JSON 文字列として現れ、content_block_stop がブロックを閉じるまで有効な JSON ではない。コードがブロックが閉じる前に入力をパースしたりツールを実行したりすると、malformed な JSON で詰まるか、引数が半分欠けたまま実行する。だからルールは単純だ。delta を集め、そのブロックの content_block_stop の後でのみ作用する。

同じ規律は、ストリームされた assistant ターンを会話履歴に追加するときにも当てはまる。追加するのは message_stop の後、すべてのブロックが完全に組み立てられてからにする。途中で切れたストリームから作ったターンは不完全で、半分構築された tool_use block が履歴に入ると、ツール使用のペアリングルールが次のリクエストを拒否する。

#### ストリームが途中で止まったとき

ストリームは時々途中で失敗する。ネットワーク接続の切断・タイムアウト・クライアントの切断が、message_stop が届く前にイベント系列を終わらせる。本当に痛い失敗は、それまで集めたものを完全であるかのように扱うことだ。ユーザーに見せた partial な text block は単なる見栄えの問題だが、履歴に書き込まれた partial な tool_use block は、次のターンを壊す構造的な問題だ。

意図的に完了を追跡する。ターンが使えるのは message_stop が届いた後だけだ。それまでは、蓄積したものを暫定として扱う。
中断したストリームでは、partial な assistant ターンを履歴に保存する代わりに捨て、リクエストをリトライする。半分構築されたターンを確定することこそ、次のリクエストを壊す。
ループを続ける前に、message_delta の stop_reason をチェックする。stop_reason が tool_use なら、組み立てたツール呼び出しは実行 ready。他の値なら、ツールのパスではなく別のパスにいる。

- **Handles well（うまく扱える）:** 長い応答やユーザー向けインターフェースで、生成されるのに応じて出力を示せば空白画面の待ちが消える。
- **Adds cost or complexity（コスト・複雑さを加える）:** ブロックを自分で組み立てる。partial block に作用してはならない。ストリーム途中の中断を明示的に扱わなければならない。
- **Use a different approach（別の手法を使う）:** 誰も出力を待っていない短い応答やバックエンドジョブでは、非ストリーム呼び出しの方が単純で、partial 状態のリスクを完全に除く。

## Screen 11 — Streaming Responses — Watch Out（注意）· 5 min

### history に半分書きかけのツール呼び出しを残したストリーム

#### 設定

ストリーム応答は画面では問題なく見え、次のリクエストを壊すことがある。テキストは描画され、ユーザーは答えを見た。ハンドラはターンを履歴に追加した。ハンドラが捉えなかったのは、ストリームがブロックの途中で落ちたことだ。だから格納した tool_use 呼び出しは入力を半分欠いていた。次のリクエストはバリデーションに失敗し、エラーは原因のストリームではなく次のターンを指す。

#### ポストモーテム：落ちたストリームの後、partial な tool_use block が履歴に確定された

あるエージェントは、オペレーターが応答の生成をリアルタイムで見られるようストリーミングを使っていた。ハンドラは content_block_delta イベントを蓄積し、読み取りループが終わったときに assistant ターンを履歴に追加した。速いローカル接続でのテストでは、ストリームは常に最後まで走り、ループは常に message_stop で終わり、格納されたターンは常に完全だった。

プロダクションでは、ネットワークの瞬断が、tool_use block が開き JSON 入力の一部を受け取った後、content_block_stop の前に、あるストリームを終わらせた。読み取りループはいつもと同じように終わり、ハンドラはターンを追加した。入力文字列が切り詰められた JSON の tool_use block を含む assistant ターンだ。オペレーターは partial な答えを見てリトライした。リトライリクエストはその壊れたターンを履歴に含み、API は malformed な tool_use block を指すバリデーションエラーで拒否した。

チームは、エラーがリトライリクエストで表面化したため、スキーマとリトライロジックを半日調べた。しかし実際の原因は上流にあった。ハンドラが「読み取りループが終わった」を「メッセージが完成した」と同義に扱ったことだ。これらは同じではない。

#### 注意すべきこと

ストリームが終わることは、メッセージが完成することと同じではない。message_stop だけがメッセージが完全であることを意味する。ハンドラが読み取りループを抜けるたびにターンを確定すると、中断したストリームが半分構築されたブロックを履歴に書き込み、失敗は原因のリクエストでなく次のリクエストに現れる。履歴への追加を message_stop で gate し、中断時には partial ターンを捨て、最後の完全なターンからリトライする。ツール使用エラーがリトライで現れたら、スキーマに触れる前に、前のターンがストリームから組み立てられたかを確認する。

## Screen 13 — Context Engineering — Teaching（コンテキストエンジニアリング）· 16 min

### モデル選択と、マルチターンセッションの予算内維持

早期に1つ選ぶ。どのモデルがワークロードを走らせるか。Claude ファミリーはコスト・レイテンシ・能力のトレードオフの幅を覆う。だから選んだモデルが、その後のすべての決定が動く範囲の価格と速度の床を決める。

モデルが決まったら、次の制約はコンテキストウィンドウだ。プロンプト・これまでの会話・すべてのツール結果を含む、モデルが一度に取り込めるテキストの全幅。Claude が返すすべてのツール結果はコンテキストウィンドウに追加され、セッションの残りそこに留まる。単発のプロンプトでは見えない。10や20のツール呼び出しを走らせる多段エージェントセッションでは、ウィンドウはすぐに埋まる。一度埋まると、エージェントは compact する（詳細を失う）か、タスクが終わる前に止まる。

だから、どのエージェントワークフローでも問うべきは、コンテキストウィンドウに何を入れるか・何を要約として戻すか・何を一切入れないかを前もって決めたか、だ。その選択の組がコンテキストエンジニアリングだ。

#### モデル選択：Sonnet で始め、意識的に動く

Claude モデルファミリーは現在4つの階層にまたがる。Fable・Opus・Sonnet・Haiku。それぞれ異なるコスト・レイテンシ・能力のトレードオフに最適化されている。Sonnet はほとんどのプロダクションワークロードのバランスの取れたデフォルトだ。Haiku は、能力範囲に収まるタスクでスピードとコスト効率のために作られた。Opus は Sonnet の範囲を超える要求の高い作業を扱う。Fable は Anthropic の最も能力の高いモデルで、複雑な推論・高度なコーディング・研究の統合・成熟したエージェントワークフローなど、最大の知性が優先される最も要求の大きいタスク向けだ。ビルド時に platform.claude.com/docs で現在のラインナップとモデル識別子を確認すること。

デフォルトの出発点は Sonnet だ。eval セットが Sonnet では品質基準に届かないと告げたときだけ Opus に上げる。eval セットが、単にコストを節約するためではなく、タスクで品質の後退が許容できると告げたときだけ Haiku に下げる。モデルを変える決定は常に計量的な決定であるべきだ。

#### コンテキストウィンドウは無償の資源ではない

コンテキストウィンドウを、Claude がワーキングメモリに保持できる空間と考える。送るすべてのメッセージ・返すすべてのツール結果・注入するすべてのドキュメント・Claude が生成するすべての応答が、そのウィンドウの空間を占める。リクエストがすでにコンテキストウィンドウより大きければ、Messages API は生成前にバリデーションエラーで拒否する。リクエストが収まっても生成が途中で天井に達すれば、現在のモデルは `model_context_window_exceeded` という stop reason で、それまで生成した出力を返す。どちらの経路も、最も古いコンテキストを黙って切り詰めない。セッションをウィンドウ上限を超えて続けたいなら、アプリケーションが次のリクエストを出す前に履歴を切り詰めるか要約することで、自分で管理しなければならない。

開発時は、テスト入力が小さくセッションが短いので、ウィンドウが埋まることはまれだ。プロダクションでは、ツール出力はテストフィクスチャより3〜5倍長く、セッションはより多くのターンを走り、ウィンドウは50ターン目でなく8ターン目で埋まる。つまり開発より早く埋まる。これに計画しないコストは、プロダクションの停止だ。

#### 予算内に留まる4つの戦略

前の節は、状態を生きたコンテキストウィンドウの外に出す論を展開した。その理由は予算だ。ウィンドウ内のすべてのトークンは、入力でお金がかかり、応答にレイテンシを加える。長いセッションは両方を複利で増やす。以下の4つの戦略は、その予算を管理する具体的な方法で、それぞれ会話の異なる形状に向く。

| 戦略 | すること | いつ適用するか | 失う連続性 |
|---|---|---|---|
| **Pruning（刈り込み）** | 前のメッセージに戻り、そこから続き、その後に来た会話を取り除く。 | Claude が生産的でない道に入った後や、次のタスクに役立たないデバッグのやり取りが積もった後。 | 巻き戻し点より後の仕事が消える。Claude がその区間で何か有用なことを学んでいたら、学び直さなければならない。 |
| **Compaction（圧縮）**（Claude Code の `/compact`・API の server-side compaction（ベータ・プラットフォームが代行・クライアント側の手動要約が代替） | 会話履歴を、Claude が学んだ鍵となる情報を保ちつつ凝縮版に要約する。要約は元のターンより少ないトークンで済む。 | セッションがコンテキスト天井に近づいているが、Claude が蓄積した知識で同じ機能への作業を続けたいとき。 | 要約で詳細が失われる可能性がある。要約に捕捉されなかったものは、以後 Claude に利用できない。 |
| **Clearing（クリア）**（Claude Code の `/clear`・API の新規セッション） | 空のコンテキストで新しい会話を始める。前のセッションから何も持ち越さない。 | 次のタスクが現在のものと完全に違い、前のコンテキストが偏りや混乱を招くだけのとき。 | すべてのセッションコンテキストが消える。Claude がセッションをまたいで覚える必要のあるものは、CLAUDE.md ファイルのような永続的な場所に置かなければならない。 |
| **Subagent Handoffs（サブエージェント引き継ぎ）** | 必要なタスク記述と system prompt だけを持つ、独立したコンテキストウィンドウでサブエージェントを生成する。サブエージェントが仕事をし、要約を返す。 | サブタスクが委譲できるほど自己完結しているとき。特に、過程がメインコンテキストを散らかすが答えは短い、探索作業。 | サブエージェントがどう結論に至ったかの可視性。中間ステップはサブエージェントのコンテキストと共に捨てられる。 |

#### もう2つのレバー：プロンプトキャッシュとトークンカウント

上の4つの戦略は、コンテキストウィンドウに入るものを管理する。2つの API 機能は、既にそこにあるものに払う額を減らす。

**プロンプトキャッシュ（prompt caching）**は、リクエストの安定したプレフィックスに対する処理を保存し、後続リクエストが同じトークンを再処理する代わりに再利用できるようにする。最初のリクエストがプレフィックスをキャッシュに書く。そこまで同じ内容を送る後続リクエストは、元のコストの一部を払う。最も有力な候補は、ターンをまたいでめったに変わらないリクエストの部分だ。長い system prompt・大きいツール定義セット・繰り返し問う参照ドキュメント。キャッシュは、キャッシュしたい最後のブロックに `cache_control` フィールド（type: `ephemeral`）でキャッシュブレークポイントを印して有効にする。最大4つのブレークポイントを置ける。安定した system prompt とツールスキーマを持つマルチターンセッションでは、それらのプレフィックスを一度キャッシュし、ターンをまたいで再利用するのが、利用可能な最もレバレッジの高いコスト削減だ。

**トークンカウント（token counting）**は、リクエストが失敗した後でなく、出る前にコンテキストの圧力を測らせる。`count_tokens` エンドポイントは messages 呼び出しと同じリクエストボディを取り、推論を実行せずにトークン数を返す。開発中に、コンテキスト予算の想定がテストフィクスチャでなく実際のツール出力に対して成り立つかを検証するために、またプロダクションで、ウィンドウを超えるリクエストをエラーにする前に gate するために使う。

#### RAG の経路が壊れうる3箇所

その経路が間違う場所は3つある。チャンキング・埋め込みのマッチ・プロンプトへの組み立てだ。

**チャンキング（chunking）**は、検索可能なコンテキストの単位を何にするかを決める。小さく分けすぎると、単一チャンクは有用であるための周辺コンテキストを欠く。大きく分けすぎると、1チャンクが無関係なテキストでマッチを薄める。文単位・節単位のチャンキングに少しの重なりを持たせるのが妥当なデフォルトだ。重なりが重要なのは、境界をまたぐ事実がさもなければ切り離され、検索しにくくなるからだ。
**埋め込みのマッチ（embedding match）**は、どのチャンクが返るかを決める。類似度検索を使うので、意味的に近い内容を検索する。それが常に、必要な正確な用語を含むものとは限らない。特定の識別子の問い合わせは、より意味的に似た結果が上位に来れば、関連チャンクを見逃す。だから辞書的マッチが意味的マッチと並行して走ることがある。
**組み立て（assembly）**では、検索されたチャンクが、プロンプトが期待する構造でモデルに届かなければならない。さもなければ、モデルは検索されたテキストからでなく記憶から答える。

**fetch-once** の経路は、推論できるシステムを与える。ある問い合わせにどのチャンクが検索されたかを検査し、その検索を直接テストできる。コストはインフラだ。構築し・保存し・コーパスの変化に合わせて同期し・在り処で保護しなければならないインデックス。**search-across-rounds** の経路は、そのインデックスとそれに伴う陳腐化を取り除く。モデルが問い合わせ時に現在のファイルを読むからだ。ただし1問い合わせあたりより多くのトークンと時間を費やし、検査しにくいプロセスになるコストで。安定した参照コーパスを単純な参照で問うなら、インデックスは所有する価値がある。変化するコーパスや多段の問いなら、1問い合わせのコストが高くても反復検索の方が通常は単純なシステムだ。
単一エージェントのエージェント的検索が検索インデックスを上回る報告された性能向上は、バージョン固定の数値だ。このモジュールの数字を頼りにするのでなく、ビルド時に参照レイヤーに対して確認すること。

では、最も一般的な2つの戦略、compaction と subagent handoffs を少し詳しく見る。

#### compaction の適用：何が保たれるかは要約器の書き方次第

Claude Code で `/compact` を使うと、ツールが要約に何を含めるかを決める。API では、ドキュメントされた主要戦略は server-side compaction（ベータ）で、リクエストで設定するとプラットフォームが会話を要約する。API セッションで手動 compaction を実装するときは、要約器のプロンプトを自分で書く。そのプロンプトが、後続ターンでエージェントが何を知るかを決める。

| 要約器プロンプト | 結果 |
|---|---|
| 「summarize the conversation so far」と言う | 一般的な要約を生成するが、タスクに不可欠な状態を落とす可能性がある。どのファイルが変更されたか・分岐点で何が決まったか・どんなエラーに遭い解決したか。 |
| 「summarize the conversation, preserving all file paths modified, all decisions made, and any errors encountered and their resolutions」と言う | エージェントが使える要約を生成する。 |

これはエッジケースではない。要約器の指定不足によるタスク必須状態の喪失は、マルチセッションのエージェント失敗の最も一般的な原因の1つだ。

#### サブエージェント引き継ぎ：長期間タスクの管理

タスクが単一のコンテキストウィンドウには大きすぎるとき、ウィンドウを大きくするのは解決にならない。解決は、タスクを分解し、各サブエージェントに必要なコンテキストだけを渡すことだ。サブエージェントは、スコープされたタスクと必要最小限のコンテキスト、直接関連する前のステップの結果、タスクを完了するために必要なツール、明確な終了条件を受け取る。親エージェントが結果を集める。このパターンはターンあたりのコストを低く保ち、長期間タスクを扱えるようにする。

compaction や pruning と同様、サブエージェント引き継ぎは実装のオーバーヘッドを加える。だから、コンテキストコストが本当の制約のところだけに適用する。単純な単発プロンプトや短いワークフローにはこれ要らない。

- **Handles well（うまく扱える）:** トークン予算を超え、分解を必要とする多段エージェントセッション。プロダクションの修正として後から当てるのでなく、アーキテクチャ段階で設計するのが最善。
- **Use a different approach（別の手法を使う）:** ウィンドウ上限に近づかないパイプライン。管理のオーバーヘッドを加える前に、モデルのコンテキスト上限に対する実際のトークン使用量を測る。

#### 先のポインタ

ここまでの戦略は、コンテキスト予算が圧力下にあると知っており、それを管理するツールを選んでいると想定する。ここでの重要点は、セッションが壊れるまで圧力の存在を知らないことにある。ワークロードは開発でテストをすべて通り、プロダクションで1つの理由で失敗しうる。ツール出力がより大きくなり、セッションがより長くなり、20ターンを綺麗に収めていたコンテキストウィンドウが8ターン目で埋まる。次の画面は、それがどう起きるかを、テストフィクスチャでうまく動いたエージェントが、実ドキュメントが流れ始めると天井に当たったポストモーテムで歩く。

## Screen 14 — Context Engineering — Watch Out（注意）· 5 min

### 開発ではうまく動き、プロダクションで天井に当たったセッション

#### 設定

ツール出力は、プロンプトやファイル読み取りと同じようにコンテキストを消費する。コンテキストウィンドウは、所与のターンで Claude が見る必要のあるすべてを保持する固定の予算だ。system prompt・会話履歴・これまでに蓄積したすべてのツール呼び出しとツール結果。ツール出力が短いとき、各ターンはその累計に少量を加え、予算は長く持つ。ツール出力が大きくなると、各ターンは同じ累計により多くを加え、予算はより早く減る。ウィンドウ自体は変わっていない。変わったのは、各ターンが今それのどれだけを使うかだ。開発で20ターンを綺麗に扱うセッションが、プロダクションでまさにこの理由で8ターン目で失敗し始める。

#### ポストモーテム：コンテキスト予算がプロダクションのツール出力に対して一度も測られなかった

あるエージェントは、40k コンテキストウィンドウのトークン予算の下で売上レシートを処理するために作られた。モデルの天井でなく、エージェントのコンテキストに対するコスト管理としてチームが設定した上限だ。モデル自体ははるかに余裕があった。現在の Claude API モデルは少なくとも 200k トークンのコンテキストウィンドウを持ち、最新のフラグシップモデル、Fable を含む、はデフォルトで 1M トークンを提供する。だから 40k という数値は、チームが課した意図的な予算で、モデルが彼らに強いた上限ではない。開発は20枚のレシートのテストフィクスチャセットを使い、それぞれがほぼ 800 トークンのツール結果を返した。完全な20ターンセッションは約18,000トークンを消費し、チームの 40k トークン予算上限に余裕で収まった。

プロダクションでは、レシートには取引記録ややり取りなど補助ドキュメントが含まれた。平均ツール出力は呼び出しあたり約3,200トークンに膨らんだ。ツール出力だけで8ターンで約25,600トークンになり、そこに system prompt・user メッセージ・assistant メッセージが加わると、累計はチームの 40k 予算上限に達した。エージェントは分析を完了する前に、8ターン目でその上限に当たった。失敗はツール選択の劣化のように見えた。エージェントが誤ったツールを選び、不完全な分析を返し始めたからだ。しかし根本の原因は違った。system prompt と初期指示が、使った後に一度も刈り込まれなかった蓄積ツール出力に押し出され、エージェントはもはや最初の指針を含まないコンテキストウィンドウで決定を下していた。

| | 開発 | プロダクション |
|---|---|---|
| 利用可能なコンテキストウィンドウ | 200k 標準・現在の Opus/Sonnet は 1M | 200k 標準・現在の Opus/Sonnet は 1M |
| チームの予算上限 | 40k トークン | 40k トークン |
| 平均ツール出力 | 呼び出しあたり約800トークン | 呼び出しあたり約3,200トークン |
| ウィンドウが埋まるまでのターン | セッションは上限に達さず完了 | 8ターン目で上限到達 |
| 観察された症状 | なし。セッションは綺麗に完了 | 8ターン目から誤ったツール選択と不完全な出力 |
| 根本原因の特定 | 該当なし | トークン使用量監査・デプロイから2日後 |
| 修正 | 該当なし | ツール出力を使った後に刈り込み、上限に達する前に compaction を予防的に適用 |

#### 注意すべきこと

開発のテストフィクスチャは、プロダクションのデータより短かった。これは、フィクスチャセットに対して作られたほぼすべてのエージェントに当てはまる。直しは、エージェントを出荷する前に、ターゲットデータの中で見つけられる最も大きい入力に対して、ツール結果の実際のトークンコストを測ることだ。

コンテキストオーバーフローの症状は、出力が似ているため、しばしばツール選択の失敗と読み間違えられる。固定数のターンの後にツール選択が劣化するのを見たら、スキーマをデバッグし始める前に、コンテキストウィンドウが埋まっているかを確認する。

## Screen 16 — Agent Construction — Teaching（エージェント構築）· 22 min

### プロダクションエージェントの構築：ループ・ワイヤリングパス・オーケストレーション・HITL

エージェントは、コンテキストを管理し定義されたゴールを持つ、多段のツール使用ループだ。各部品（ツールスキーマやコンテキスト管理）は既に構築した。この画面はそれらを稼働するシステムに繋ぎ、どのトピックも単独では扱わない層を加える。

部品が複数ターンをまたいで一緒に走ると、単独テストが捉えない新しい失敗モードが現れる。単発テストで働いたルーティング決定が複利で響き始める。コンテキストは予想より早く埋まる。前の結果に依存するステップが、前のツール呼び出しの構造が誤っていたため、間違った入力を得る。すべてのエージェント構築に先立つべき問いは、これだ。この問題はエージェントを必要とするか？

エージェントは、より単純なパターンより、調整のオーバーヘッド・拡大したコンテキストコスト・より大きい失敗の表面積を伴う。その問いに意識的に答えることが、最初の設計決定だ。

#### Workflow or agent：最初の1行を書く前にこの決定をする

エージェント開発で最も致命的な失敗は、最初に誤ったパターンを選ぶことだ。ワークフローとエージェントは異なる問題を解く。ワークフローで十分なところにエージェントを使うと、能力を加えずに振る舞いの複雑さを加える。エージェントが必要なところにワークフローを使うと、ユーザー入力が予定の経路から外れるたびに壊れるシステムを生む。

| ワークフローを選ぶのは… | エージェントを選ぶのは… |
|---|---|
| 正確なステップをコードで列挙できる。 | ゴールとツールは指定できるが、正確な経路はできない。 |
| エラーのコストが現実で、ステップレベルのガードレールが重要。 | 仕事を通る経路を前もって列挙できない。 |
| 標準ツールでの可観測性が必要。 | 非決定性が許容され、エージェントの可能な行動が登録されたツールセットで制約される。 |
| 入力が既知のセットにうまく制約されている。 | ユーザー入力が内容・構造で予測不能に変わる。 |
| タスクの毎回の実行が同じ順序に従う。 | タスクが利用可能なツールの創造的な順序付けを必要とする。 |

#### エージェントがパターン。ワイヤリングパスは実装の選択

タスクがエージェントを必要すると決めたら、パターンも決まっている。ツールを呼び・コンテキストを管理し・ゴールが満たされるまで走るループだ。単一エージェントシステムでは、そのパターンは3つすべてのワイヤリングパスで一定だ。プランナー・エクゼキューター・エバリュエーターが別々のエージェントとして構造化アーティファクトを通じて引き継ぐマルチエージェントアーキテクチャは、ループ自体を超えた追加の設計決定を導入する。それらのパターンはこのトラックの後で扱う。そのパターンはどう構築するかで変わらない。変わるのは、ループのどれだけを自分で書き、どれだけをライブラリやホステッドサービスに任せるかだ。

3つのワイヤリングパスがあり、どれだけのインフラを所有するかのスペクトラム上にある。Messages API に直接ループを書けば、完全な制御と完全な責任を得る。Agent SDK を使えば、同じループを自分のプロセス内で走らせ、ツール実行・コンテキスト管理・反復構造を既に構築済みのものとして受け取る。あるいは Claude Managed Agents（現在パブリックベータ）を使い、Anthropic がループとサンドボックスを走らせ、アプリケーションはイベントを流入させ結果を戻す。ループこそ変わらないものだから、以下の節はループ自体を教える。選ぶパスが、その周りの部品を誰が保守するかを決める。

#### ワイヤリングパス：誰がループを走らせ、何を引き受けるか

3つのパスは1つの変数で違う。エージェントのランタイムをどれだけ所有するか。表は、引き渡すインフラが多い順に上から下へ並ぶ。デプロイとコンプライアンスの制約で選ぶ。プロトタイプが最も速いという理由だけで選ぶ誘惑に負けない。

**1. Raw Messages API ループ**
- **誰がループを走らせるか:** あなたのコードが毎回の反復を走らせる。リクエストを送り、tool-use ブロックを読み、ツールを実行し、結果を自分で追加する。
- **何を所有するか:** 完全なループ・ツール実行・コンテキスト管理・リトライ・終了条件。何も提供されない。
- **これを選ぶのは:** 各ステップの完全な制御が必要なとき、ライブラリが収容しない制約があるとき、または抽象を加える前にループがどう働くかを自分で学んでいるとき。
- **commit 前に確認すること:** 保守コストはあなたのもの。SDK が無償でくれる振る舞い（コンテキスト管理や並列ツール処理を含む）すべてが、あなたが書きテストするコードになる。

**2. Agent SDK**
- **誰がループを走らせるか:** SDK が自分のプロセス内でループを走らせる。反復しコンテキストを管理し、あなたのコードはエージェントが呼ぶツールをまだ実行する。
- **何を所有するか:** ツール実行と周囲のアプリケーション。SDK がループ構造・コンテキスト管理・ツール登録を提供する。
- **これを選ぶのは:** Claude Code を支えるループ・コンテキスト処理・ツールの足場を再構築せずに欲しく、Python や TypeScript で自分の環境でエージェントを走らせたいとき。
- **commit 前に確認すること:** CLAUDE.md や skills のようなファイルシステムベースの機能が Agent SDK で読み込まれるかは、`settingSources` configuration で制御される。デフォルトに頼らない。常に `settingSources` を意図するソースに明示的に設定する（例：Claude Code CLI の挙動に合わせるなら `["user", "project", "local"]`、プログラムで渡したものだけで完全に孤立して走らせるなら `[]`）。ビルド時に Agent SDK リファレンスで現在のデフォルト挙動を確認すること。

**3. Claude Managed Agents**
- **誰がループを走らせるか:** Anthropic がループとサンドボックスを走らせる。アプリケーションはユーザーイベントを送り、server-sent events で結果を戻す。
- **何を所有するか:** アプリケーション層とエージェント定義。モデル・system prompt・ツール・MCP サーバー・skills を一度定義し、セッションをまたいで ID でエージェントを参照する。
- **これを選ぶのは:** 分・時間で測る長時間実行が必要なとき、マネージドサンドボックスが欲しいとき、またはループ・サンドボックス・ツール実行層を一切構築したくないとき。AWS 上の Claude Platform でも一部機能差で利用可能。commit 前にデプロイ面に対して機能の同等性を確認すること。
- **commit 前に確認すること:** セッションはステートフルでサーバー側に保存される。つまり現在 Zero Data Retention や HIPAA の Business Associate Agreement の対象ではない。（platform.claude.com の Anthropic API データ保持ドキュメントを参照・公開時に確認。）
- 現在パブリックベータで、すべてのエンドポイントが `managed-agents-2026-04-01` beta ヘッダーを必要とし、振る舞いはリリース間で洗練される可能性がある。マイグレーション計画を伴って構築する。

#### Claude Managed Agents：いつ使うか

上の表は Managed Agents を第3のパスとして挙げた。一部のワークロードでは正しいデフォルトなので、その選択を具体化する。

核心の違い：raw ループや Agent SDK では、あなたのコードが反復を走らせる。各リクエストを送り、tool-use ブロックを読み、ツールを実行し、結果を追加する。Managed Agents では、Anthropic がループとサンドボックスを代行する。アプリケーションはエージェントを一度定義し（モデル・system prompt・ツール・MCP サーバー・skills）、ID で参照し、ユーザーイベントを送り、server-sent events で結果を戻す。

| カテゴリ | 所有をやめるもの | 代わりに引き受けるもの |
|---|---|---|
| 実行とインフラ | 反復ループ・実行サンドボックス・ループ内のリトライ・ツール実行ランタイム。Anthropic がすべてサーバー側で走らせる。 | バージョン管理された API リソースとして管理されるエージェント定義と、イベントを送りストリーム結果を消費するアプリケーション層。 |
| セッション持続と状態 | 長時間実行の管理。セッションは、自分のプロセスがループを開いたままにせず、分・時間走れる。 | サーバー側のセッション状態。セッションはステートフルで Anthropic に保存され、そのデータ取り扱いポリシーと制約に従う（下の制約注記を参照）。 |
| サンドボックスライフサイクル | ツール実行のためのサンドボックスのプロビジョニングと解体。 | 自分の環境でなく、マネージドサンドボックスの利用可能なツールとその実行モデルへの依存。 |

**Managed Agents を選ぶのは:**
- タスクが長く走る。分・時間で測る実行は自分のプロセスで開いたままにするには不格好で、マネージドループはまさにそのために作られている。
- マネージドサンドボックスが欲しい。ツール呼び出しのために実行環境を構築・保護するのであれば、Managed Agents は大きいインフラをあなたの皿から取り除く。
- ループ・サンドボックス・ツール実行層を一切構築したくなく、エージェントを API リソースとして定義する用意がある。

**規制ワークのためにそれを決める制約:** Managed Agent セッションはステートフルでサーバー側に保存される。その保存こそ、これらのセッションが現在 Zero Data Retention や HIPAA BAA の対象ではない理由だ。だからワークロードが PHI を運ぶか ZDR 要件下にあれば、運用にどれだけ合致してもこの経路は排除され、代わりに Agent SDK や、カバーされた構成の raw ルートに流れる。支配的な制約が、都合が口を出す前にパスを決める。

よくある進展は、Agent SDK でローカルにプロトタイプし、プロダクションでは Managed Agents に移すことだ。核心のエージェント定義は概念的に持ち越せる。変わるのは形式だ。Agent SDK はコードレベル・ファイルシステム設定を使い、Managed Agents はエージェントをバージョン管理された API リソースとして定義する。直接のエクスポートでなく、再表現のステップを想定する。

- **Handles well（うまく扱える）:** 長時間実行エージェントと、サンドボックスやループを自分で構築・保護したくないワークロード。
- **Adds cost or complexity（コスト・複雑さを加える）:** サーバー側のステートフルセッション、agent-as-resource の定義形式、リリース間で変わりうるベータ面。
- **Use a different approach（別の手法を使う）:** PHI や ZDR ワークロード、または完全なインプロセス制御が必要な場合は、Agent SDK か、カバーされた構成の raw ループに留まる。

#### ループのワイヤリング：すべてのパスに共通する4ステップ

下の4ステップは、どのパスで構築しても稼働するエージェントループを定義する。Messages API に対してループを書くときは、4つすべてを自分で実装する。Agent SDK を使うときは、ツールの登録・system prompt の設定・ループの反復の構造を提供し、コードはまだツール実行を扱う。ステップは同じ。違うのは、どれだけ書き・どれだけ継承するかだ。

1. **ツールの登録（Register tools）:** 各ツールは同じスキーマ構造に従う。SDK はエージェントに対して登録し、Claude は何が利用可能かを知る。
2. **system prompt の設定（Set the system prompt）:** エージェントのタスクにスコープする。広い system prompt はより広く・より信頼性の低いツールルーティングを生む。特定のタスクとそのために利用可能なツールを名指しする system prompt が、より一貫した振る舞いを生む。
3. **ツール使用ループの処理（Handle the tool-use loop）:** 自分で反復するにせよ SDK が反復するにせよ、コードが実行を扱う。Claude が発行するすべてのツール呼び出しはコードが実行し、tool-result ブロックで返さなければならない。
4. **終了条件の定義（Define exit conditions）:** エージェントループは停止条件を受け取るまで走る。明示的な終了条件がないと、エージェントはタスクが必要とする以上にツール呼び出しを要求し続ける。done が done を意味するときを定義すべきだ。

#### ループワイヤリングのチェックリスト：パスに関わらず確認するもの

| # | 項目 | 確認すること |
|---|---|---|
| 1 | ツール登録 | エージェントが必要とする可能性のあるすべてのツールが登録リストにある。system prompt に未登録のツールが参照されていない。 |
| 2 | system prompt のスコープ | system prompt がタスクと利用可能なツールを名指しする。エージェントが持たないツールを記述しない。スコープの指針を必要とする、エージェントが持つツールを省略しない。 |
| 3 | ツール使用ループの実装 | コードが Claude が発行するすべての tool-use ブロックを扱い、次の assistant ターンまでにそれぞれに tool-result ブロックを返す。単一の assistant ターンからのすべての tool-use ブロックは一緒に解決されなければならない。 |
| 4 | HITL 挿入点の定義 | ループの少なくとも1点に human-in-the-loop チェックがある。どこに挿入するかは下の節を参照。 |
| 5 | 終了条件の定義 | ループに、Claude が自発的に止まることに依存しない明確な停止基準がある。 |

#### Human-in-the-loop（HITL）：挿入点とそれぞれの適所

human-in-the-loop チェックポイントは、エージェントの実行を一時停止し、先に進む前に人間のレビューステップに流す。どこに挿入するかを決める問いは、これだ。このステップが人間のチェックなしで走ったら、最悪の結果は何か？

| 挿入点 | チェックのトリガ | 対処するリスクレベル |
|---|---|---|
| 破壊的ツール呼び出しの前 | エージェントが write・delete・send 操作を実行しようとしている。 | 高：誤った呼び出しを元に戻せない、不可逆な操作。 |
| 計画ステップの後 | エージェントが計画を生成し、実行を始めようとしている。 | 中：すべてのステップが正しく実行されても誤った結果を生む、誤った計画。 |
| 予想外の出力時 | ツール結果がエラーフラグ・空の結果・予想範囲外の値を含む。 | 変動：リトライロジックだけでは解決しない失敗モードを捉える。 |

#### ツールオーケストレーション：過剰ツールと不足ツール

エージェントのルーティング挙動は2つで形作られる。ツールがどう記述されるかと、いくつ登録されるか。重なる description のツールが多すぎると、気まぐれなルーティングを生む。ツールが少なすぎると、エージェントは経路を幻覚するか不完全な結果を返すしかない。

プロダクションエージェントでは、過剰ツール（over-tooling）がより一般的な問題だ。チームは「念のため」と必要になるかもしれないすべてのツールを登録し、ツール表面が大きくなるにつれて Claude の選択品質が劣化するのを発見する。タスクに必要な最小セットで始め、能力の特定の隙間が確認されたときだけツールを追加する。

**エージェントが正しい選択のとき:**
- ゴール指向のタスクで、正確な経路を前もって列挙できない。ワークフローでは数十の条件分岐を必要とする、変動する入力の処理。

**エージェントを使うときに引き受けるもの:**
- エージェントは振る舞いの複雑さを加える。タスクを通る経路は、コードの明示的な分岐ロジックからでなく、蓄積コンテキストに対するモデルの推論から生じる。可観測性は標準の運用ログでなく、トランスクリプトレベルのツールを必要とする。

**ワークフローを選ぶとき:**
- ステップをコードで列挙できるなら、ワークフローを使う。エージェントは進展の最後のステップだ。問題を解く最も単純なパターンで始める。単一の API 呼び出し、次にワークフロー、次にエージェント。そして、より単純なパターンがタスクが必要とする変動性を扱えないときだけ上げる。

#### 規制データの制約は、ワイヤリングを書く前に配送ルートとクレデンシャルを決める

データが特定の制約で取り扱われる必要がある場合（弁護士・依頼人特権、HIPAA、GDPR、FedRAMP、社内データ常駐ポリシーなど）、その制約が、プロンプト・ツール・メモリについての設計選択を1つする前に、コードがどのエンドポイントを呼ぶか・どのクレデンシャルを運ぶか・ログがどこに落ちるかを決める。開発者として通常サーフェスを選ばないが、特定のエンドポイントを狙い・クレデンシャルを付け・リージョンを設定し・ログを出すコードを書く。支配的な制約を最初に名指しする。誤ったクライアント設定は、エージェントをワイヤリングした後で元に戻すより、最初に正しく設定する方がはるかに安い。下の5つの制約は、プロダクションで最も当たりそうなケースを覆う。

| 制約 | コードで排除されがちなもの | コードレビューを通るもの |
|---|---|---|
| 弁護士・依頼人特権 | 事務所がエンドツーエンドで監査できない、コンシューマグレードの Claude.ai サーフェスからの呼び出し。プロンプトや system メッセージの構造にかかわらず、特権文書の内容を、事務所が特権資料向けに承認していないエンドポイントに送るコードパス。 | 事務所の自分のアプリケーション内からの直接 API または SDK 呼び出し。SSO で認証し、完全なリクエスト・レスポンスロギングを伴う事務所承認の LLM ゲートウェイを通す。Anthropic のネイティブな Compliance Conversation の内容（プロンプト・レスポンス・ツール呼び出しペイロード）は直接 API トラフィックではデフォルトで Anthropic に捕捉されないので、組織はアプリケーション層で会話ロギングを実装し、承認されたログ宛先に流さなければならない。ツール呼び出しとツール結果は監査されたパス内に留まる。最終のログ設計を Anthropic のアカウントチームと確認。 |
| HIPAA（PHI 扱い） | 保護された健康情報を、使用中の特定の構成に対する BAA の対象でないエンドポイントや配送ルートに送るコード。これには、同じ BAA の下でスコープされていない、コードが書き込むあらゆるロギングや保持パスが含まれる。 | BAA 対象の構成での直接 API または SDK 呼び出し。Anthropic ファーストパーティ API アクセスの BAA カバレッジは Anthropic と手配し、Anthropic が独自の HIPAA 対応組織をプロビジョニングし、その独自の側で機能制限を強制する。対象構成を Anthropic のアカウントチームと確認。別案は AWS Bedrock や GCP Vertex 経由のクラウド媒介ルートで、パートナーの既存の HIPAA 対応クラウドアカウント上で。注：BAA は Console・Workbench・ベータ機能・コンシューマプランを覆わない。すべての API 機能が BAA の下で対象になるわけではなく、構成前に Anthropic の Implementation Guide で現在の機能適格リストを確認。 |
| GDPR とデータ常駐 | モデル実行のリージョンをコードで固定できない、またはリクエストが承認された地理的境界外のリージョンから提供されうる配送ルート。リージョンを指定せずにグローバルエンドポイントにデフォルトするのが、ここで壊れる一般的なパターン。 | Bedrock や Vertex のようなクラウド媒介ルートで、クライアント設定でリージョンを対象の管轄区に固定。直接の Anthropic API は別ケースで、現在 EU データ常駐を提供しないので、EU データ常駐要件のパートナーは API を直接呼ぶ代わりに Bedrock や Vertex を通すべき。 |

この表は、エンドポイント選択とクレデンシャル設定を直接決める制約を覆う。SOC 2 はここでは対象外だ。SOC 2 はコードがどのエンドポイントを呼ぶかでなく、システムがどう構築・運用されるかを治め、Module 4 で他のセキュリティ姿勢と監査要件と共に扱う。

## Screen 17 — Agent Construction — Watch Out（注意）· 5 min

### プロダクションのファイルを編集してしまったエージェント

#### 設定

エージェントはテスト環境が寛容だからエンドツーエンドで動くが、プロダクションは違う。同じツール・同じループ・同じ system prompt を持つが、HITL チェックポイントがない。テストがそれを必要とするユースケースを一度も表面化させなかったからだ。

#### スクラッチディレクトリでテストされたファイル編集エージェント、顧客環境にデプロイ

開発者は、設定ファイルを読み・変更し・書けるエージェントを構築した。system prompt は read_file・write_file・validate_config の3ツールを与えた。エージェントのループは単純だった。各 write の後、validate_config を再実行し、設定がまだバリデーションに失敗すれば、編集を調整してもう一度書き、止まる前に最大10回反復する。エージェントはターゲット設定のコピーを持つスクラッチディレクトリでテストされた。すべてのテストケースで正しく動き、典型的には2〜3回の反復で有効な設定に収束した。

顧客環境にデプロイされると、エージェントは設定パラメータが範囲外であることを正しく特定した。修正を提案し、write_file を呼び、validate_config を再実行し、pass を得た。ループは1回の反復で綺麗に終了した。設計通りだ。10回反復の上限には、必要なかったから一度も達しなかった。ループ設計は正しかった。しかし終了条件が問題だった。

エージェントが修正したパラメータは、顧客のアプリケーションが依存するレートリミットだった。validate_config は値がスキーマの許容範囲内であることをチェックし、今それは範囲内だった。validate_config がチェックせず、チェックするよう設計されたこともなかったのは、下流システムが古い値に依存しているかどうかだった。write の数分以内に、顧客のアプリケーションは、想定していないレートでリクエストが絞られたため、失敗し始めた。

エージェントのループは開発者が求めたことを正確にした。編集し、検証し、バリデーションが pass したら終了した。失敗はループになかった。失敗は、ループの終了条件（validate_config が pass を返す）がエージェントが編集しているファイルにスコープされており、「このファイルでバリデーションが通った」と「顧客環境に write が commit された」の間にチェックポイントがなかったことだ。欠けていたのはループ設計のチェックポイントだ。最初の write_file 呼び出しがライブの顧客設定に届く前に、提案された変更を人間のレビューのために一時停止し表面に出すこと。実践には、これはループが「提案された変更の準備完了」と「write が commit 済み」の間の明示的な分岐を必要とすることを意味する。テストがそれを必要とするケースを一度も生まなかったので、開発者が決して追加しなかった状態だ。

#### 注意すべきこと

このインシデントが示すパターンは、設計中に一度も聞かれなかったパーミッションの問いだ。エージェントはタスクがファイル編集を伴うので write アクセスを持ち、タスク自体は正当だった。チームが見逃したのは、変更を提案するエージェントと、それを commit するエージェントの間の隙間だ。使い捨てのテスト環境では、その隙間は決して表面化しない。「誤った write」が触れるものはテストでは何も意味を持たないからだ。しかしプロダクションは違う。

決して聞かれなかった設計の問い：「write_file が人間のチェックなしで走ったら、最悪の結果は何か？」その答えが、ツールが実行される前に human-in-the-loop チェックポイントが必要かを決める。

ツールがプロダクションで不可逆な行動を取れるなら、それが走る前にチェックポイントが必要だ。その制約を設計中、ツール表面をスコープしているときに登録する。最初のインシデントの後にではない。

def run_agent_loop(user_request):

## Screen 19 — Agent Memory — Teaching（エージェントメモリ）· 8 min

### セッションをまたぐ状態に適切なスコープを選ぶ

前の画面のエージェントは、単一セッション内で正しく動く。できないのは、そのセッションが終わったとき何も覚えていないことだ。メモリスコープは、次のセッションの開始時にエージェントが何を知るべきかと、その知識を前に運ぶのにどれだけかかるかを決める。

#### メモリのパターンと、それぞれの適所

メモリスコープを超えて、ブループリントはこの目標の下にいくつかのエージェント設計パターンをまとめる。それぞれはこのモジュールの前に構築済みだ。モデルがツールを呼び・結果を読み・続くツール使用ループは、ツール使用とエージェント構築のクラスタからのコアパターンだ。多段タスク分解はゴールを順序付きサブタスクに分け、計画と実行の分離（planning-and-execution）は計画を決めることとそれを実行することを分ける。それは計画ステップの後の human-in-the-loop チェックが守るのと同じ分離だ。次に扱うメモリスコープは、ループが終わった後にどの状態が生き残るかを決めるパターンだ。

メモリスコープは、新しいセッションが始まるときエージェントが何を知るかを決める。誤った選択には2つの失敗モードがあり、それらは逆の方向に引く。

- コンテキスト内に状態が多すぎると、すべての API 呼び出しを膨らませる。モデルが毎ターン完全な会話を再読するからで、請求はセッション長で増大する。
- 永続ストレージに状態が少なすぎると、セッションをまたぐメモリをエージェントから奪う。書き留められないものは会話が終わった瞬間に消えるからだ。

| スコープ | 何が残るか | コスト | いつ使うか | 失うもの |
|---|---|---|---|---|
| **In-context memory（コンテキスト内メモリ）** | 状態はアクティブな会話にあり、単一セッション内のターンをまたぐ。 | 取得オーバーヘッドゼロ。会話が伸びるほどトークンコストが膨らむ。 | 短いセッションで、エージェントが必要な状態がすべてコンテキストウィンドウに収まり、再起動をまたぐものがないとき。 | セッションが終わればすべて。clear コマンドや新規セッションが状態を拭う。 |
| **External storage（外部ストレージ）** | 状態はデータベースに書かれ、セッション開始時やオンデマンドで読み戻される。 | 各データベース呼び出しが取得レイテンシを加え、読み書きロジックの工学作業を引き受ける。 | セッションをまたいで生き残る状態、ユーザー間で移動する状態、複数エージェントインスタンスで共有される状態。 | 永続化側では何も。コストは毎呼び出しのレイテンシと継続的な実装複雑性として現れる。 |
| **Summarized memory（要約メモリ）** | 前の会話の凝縮版が生成され、次のセッションの開始時に注入される。 | 完全な履歴を再再生するよりセッションあたりのトークンコストは低いが、要約ステップで元にあった詳細を落とす。 | 完全な履歴が会話が終わる前にコンテキスト予算を超える、長期間の会話エージェント。 | 要約器が保たなかった詳細。エージェントは要約プロンプトが残すと選んだものしか見ない。 |
| **No persistent memory（stateless・ステートレス）** | 何も。各セッションは独立。 | 取得や保存するものが何もないので、オーバーヘッドは皆無。 | 終わって閉じるタスク実行エージェントや、すべてのセッションが設計上完全に独立なパイプライン。 | 以前のコンテキストすべて。フォローアップが前のセッションの何かに依存するなら、エージェントはそれに届く手段を持たない。 |

#### エージェント設計時にメモリスコープを選ぶ

エージェントが前のやり取りをどう覚えるかの選択は、プロダクションのリファクタでなく設計フェーズに属する。複数日にわたり同じユーザーを助けるエージェントは、セッション間で状態を運ぶ必要がある。つまり、要約か完全な履歴をモデルのコンテキストウィンドウの外に保存し、次のセッションが読み戻せるようにする。単一のジョブを受け取り・完了し・閉じるエージェントには、思い出す前のセッションがないので、ステートレスで走る。

デフォルトの経路は最初は妥当に見える。完全な会話履歴を messages 配列に保存し、毎 API 呼び出しで送る。プロトタイプは動く。しばらく動き続ける。トラブルはもっと先で始まる。トークンコストが追加ターンごとに増大し、コンテキストウィンドウが埋まるにつれてレイテンシが上がり、やがて長いセッションが硬い上限に当たりエージェントが応答を止める。その時点でリファクタが必要になる。会話状態を生きたコンテキストから引き出し、外部ストレージに置き、各ターンが必要なものだけを加える。リファクタ自体は機械的で、数100行のコードとチームが既に持つデータベースだ。コストがかかるのはタイミングだ。その作業はプロダクションの圧力の下で、通常は既に動いている期限と共に起きる。メモリを再構築する1時間は、エージェントが次にすべきことに充てられない1時間だ。設計フェーズで決めるのは安く、リファクタのときに決めるのはより高い。

- **Handles well（うまく扱える）:** メモリスコープが設計時にタスクに合う。エージェントがセッションをまたいでスレッドを続けるときは外部ストレージ。各ジョブが自己完結のときはステートレス。セッションが短く再起動を必要としないときはコンテキスト内。
- **Adds cost or complexity（コスト・複雑さを加える）:** 外部ストレージは取得レイテンシとそれに伴う読み書きロジックを加える。要約メモリはよく指定された要約器プロンプトに依存し、それがないとタスク必須の状態が毎圧縮で落とされる。どちらのアプローチも無償でないので、コストを秤にかけて賢く選ぶ。
- **Use a different approach（別の手法を使う）:** ウィンドウが十分大きいという前提で、すべての状態をコンテキスト内に持つこと。完全なコンテキストが毎 API 呼び出しで送られるため、トークンコストは追加ターンごとに増大する。キャッシュや compaction なしでは、長いセッションは、チームが初期ターンだけ測るときに想定するより速くコストを蓄積する。commit 前に、ウィンドウ上限に対する実際のセッショントークン使用量を測る。

#### Skills：毎セッションを膨らませず、オンデマンドで読み込む再利用可能な指示セット

上のメモリスコープ表は、エージェントがセッションをまたいで状態をどう運ぶかを覆う。関連するが別の問題がある。特定の種類のタスクの扱い方を Claude に一度教え、すべてのセッションに注入するコストを払わずに、タスクをまたいで繰り返し可能な指示をどう運ぶか。そのパターンが **Skill** だ。再利用可能な markdown ファイルで、要求がその description に合致すると Claude が自動的に読み込む。指示は必要になるまでディスクにあり、すべての会話に常駐しない。

Skill は、識別されたディレクトリ内の `SKILL.md` ファイルに置かれる。ファイルは2部からなる。name と description を持つ frontmatter ブロックと、その下の指示。description がマッチング基準だ。要求を送ると、Claude は利用可能なすべての Skill の name と description を読み、メッセージと比較し、合致したときだけ完全な指示を読み込む。指示が現在の要求に関連しないなら、コンテキストウィンドウに入ることはない。

これが上の表のメモリパターンとの重要な対比だ。コンテキスト内メモリは常に存在し、毎ターンで増える。CLAUDE.md の挙動は Claude Code をどこで走らせるかによる。Claude Code CLI では、CLAUDE.md ファイルは実行中のタスクにかかわらず毎セッションに読み込まれる。Agent SDK では、CLAUDE.md を含むファイルシステム設定が読み込まれるかは `settingSources` configuration で制御される。デフォルトに頼らない。意図するソースに明示的に設定し、ビルド時に Agent SDK リファレンスで現在のデフォルト挙動を確認する。一方 Skill は、両環境で、タスクがそれを求めるときだけ読み込まれる。すべてのセッションでなく特定の繰り返しタスクに適用される指示セットには、Skill はどの替代よりも低オーバーヘッドなパターンだ。

#### Skills vs. CLAUDE.md vs. コンテキスト内指示：適切なパターンの選択

| パターン | いつ読み込まれるか | コンテキストコスト | 最適な用途 |
|---|---|---|---|
| **Skill（SKILL.md）** | 要求が skill の description に合致したときオンデマンド | 低。起動時に name と description だけが読み込まれ、完全な内容は合致時だけ | 不要なセッションを膨らませたくない、タスク固有の専門知識。例：ドメイン固有の出力フォーマット、専門的なレビューチェックリスト、すべてのやり取りでなくタスクの一部に適用されるワークフロー。 |
| **CLAUDE.md** | 毎セッション・無条件 | タスクにかかわらずセッションあたり固定のオーバーヘッド | すべてに適用される、常時オンのプロジェクト標準。例：チームが標準化したコーディング規約、プロジェクトが要求する出力フォーマットルール、コードベースの全タスクをまたぐ制約。 |
| **コンンテキスト内指示** | そのセッション内の毎ターンで存在 | セッション長で増大。セッション終了で生き残らない | 完全な履歴がウィンドウに収まり、何も永続する必要のない短いセッション。例：単発の探索作業、単一の会話にスコープされたタスク。 |

#### 現在の可用性：Messages API 上の Skills

Skills は今日 Messages API で利用可能だが、統合はベータで、構成は Claude Code や Agent SDK の経路と同じでない。API リクエストに2つの beta ヘッダーが必要だ。`code-execution-2025-08-25` と `skills-2025-10-02`。この方法で呼ばれた Skills は、呼び出し元アプリケーションの環境でなく、code execution container の中で走る。これには、Skill が依存できるツールとファイルシステムアクセスへの含意がある。

beta ヘッダーはバージョン管理され、機能が一般提供に向かうにつれて変わる。プロダクションでこの構成に対して構築する前に、現在の Anthropic API ドキュメントでヘッダー値、機能が一般提供に達したか、code execution container がまだランタイムパスかを確認する。

1つの重要な制約：サブエージェントは親セッションから Skills を自動的に継承しない。サブエージェントにタスクを委譲すると、それはクリーンなコンテキストで始まる。Skills と会話履歴は持ち越されないが、サブエージェントは親セッションからパーミッションコンテキストを継承することに注意。パーミッションスコープは委譲でリセットされない。サブエージェントが Skill を必要とするなら、サブエージェントの構成に明示的にリストしなければならない。これはエージェント設計時に意味を持つ。特定の指示に依存するタスクを行うサブエージェントをワイヤリングするなら、その指示は親から持ち越されると想定するのでなく、サブエージェントに対して登録されなければならない。

## Screen 24 — Multimodal and Batch Ingestion — Teaching（マルチモーダルとバッチ取り込み）· 13 min

### 画像・PDF・大量処理

ここまでは、ターン間で Claude が何を覚えるかを管理してきた。マルチモーダル取り込みは、問いを「何を送るか」に移す。すべての画像と PDF は、Claude がプロンプトの1文字を読む前にコンテキスト予算を消費する。それがリクエストの構成と1回に収まるものを変える。このトピックの後半は、同じ問題の逆の端を扱う。処理する入力が数千あるとき、一度に1リクエストを送り各応答を待つことは意味をなさなくなる。Batch API は、アプリケーションをブロックせずにそのボリュームを扱う手段だ。

#### 画像のトークンコスト：commit 前に計算する

画像はコンテキスト予算の点で無償ではない。Claude は画像をパッチで見る。画像の28×28ピクセルブロックごとが1つの視覚トークンで、画像は ⌈width / 28⌉ × ⌈height / 28⌉ の視覚トークンを消費する。1,000×1,000ピクセルの画像は ⌈1000/28⌉ × ⌈1000/28⌉ = 36 × 36 パッチ、約1,296視覚トークン。その割合だと、高解像度スクリーンショット10枚が、詳細な system prompt と同じ量のコンテキストを消費する。各モデルはまた、長辺上限と視覚トークン上限として表現される最大ネイティブ画像解像度を持ち、これらの上限はモデル階層で異なる。最新モデルは標準階層より大幅に大きい画像を受け付ける。どちらかの上限より大きい画像は処理前にダウンスケールされ、公式はスケール後の寸法で走る。ビルド時に Vision ページ（Resolution and token cost）で現在の階層別上限を確認すること。上限はモデル世代間で変わっており、また変わる。

計算は設計時に意味を持つ。画像を処理するパイプラインを構築するなら、取り込みコードを書く前に、典型的なプロダクション画像のトークンコストをモデルのコンテキスト上限に対して測る。予算超過パイプラインの直しは、10分の画像リサイズステップであることが多い。デプロイ後に発見すれば、もっと時間がかかる。

#### 画像の送り方3つ：それぞれの適所

**Inline base64**
- **仕組み:** 画像バイトを base64 文字列にエンコードし、データをメッセージブロックに直接含める。
- **オーバーヘッド:** 完全にエンコードされたペイロードが毎リクエストで流れ、リクエストサイズを膨らませ、大きい画像ではレイテンシに響く。
- **いつ使うか:** アップロードステップが見返りなく複雑さを加える、単発画像に最適。同じ画像を繰り返し送るとコストが掛け算になるので、再利用が見込まれるなら別の方法へ。

**URL reference**
- **仕組み:** source ブロックに公開到達可能な URL を渡し、Claude がリクエスト時に画像を取得する。
- **オーバーヘッド:** ペイロードはリクエストに流れないが、URL が安定・公開・Claude が取得を試みるときに到達可能でなければならない依存を引き受ける。
- **いつ使うか:** 画像が既に自分が管理する安定した公開 URL でホストされているときに最適。認証の裏・短い有効期限で署名されたもの・リクエスト実行時に到達可能と保証できないものには避ける。

**Files API**
- **仕組み:** 別の API 呼び出しで一度ファイルをアップロードし、file_id を受け取り、以降の任意のメッセージでその ID を参照する。
- **オーバーヘッド:** アップロードは一回限りのコスト。以降のリクエストはバイトでなく ID を運ぶので、そこから先のペイロードオーバーヘッドはほぼゼロに落ちる。現在ベータで、Bedrock や Vertex AI では利用不可。デプロイ先での可用性を確認。
- **いつ使うか:** 同じ画像や PDF が複数リクエストにまたがって現れるとき、またはアセットが大きくて再送がリクエストサイズを支配するときに最適。また、アセット管理を推論呼び出しから切り離したいときの最も綺麗な選択で、会話ターンをまたいで現れる画像に適する。file_id は履歴が伸びてもペイロードの重さを持たないから。

#### PDF の送信：document ブロック

PDF の場合、ブロック型は image でなく document だ。source 構造は画像と同じパターンに従い、base64・URL・Files API の file_id のいずれかになる。document ブロックに必須の name フィールドはない。ブロックは読めるドキュメント名の任意の title フィールドと、追加メタデータの任意の context フィールドを受け付けるが、PDF を送るのにどちらも必須でない。他のすべての仕組み（トークンコストの考慮や Files API の再利用を含む）は同じように当てはまる。

```json
{
  "type": "document",
  "source": {
    "type": "base64",
    "media_type": "application/pdf",
    "data": "<base64-encoded-pdf-bytes>"
  },
  "title": "contract_review.pdf"
}
```

#### マルチモーダル入力へのプロンプト技法の適用

最初のセクションのプロンプト技法は、画像・PDF 分析にも同じように当てはまる。素の "describe this image" プロンプトは、素のテキストプロンプトと同じ理由で浅い出力を生む。Claude が狙うターゲット構造を持たないからだ。

違いは、画像がテキストにはできない曖昧さを運ぶことだ。重なるオブジェクト・深度と空間関係・部分的な隠蔽を含む。視覚分析のプロンプトは、Claude が各タイプの曖昧さをどう扱うかを名指しすべきだ。「オブジェクトが重なるなら、それぞれを別々に記述し、重なりを注記せよ」は、テキストのみのプロンプトが決して必要としない、具体的な制約だ。

#### Message Batches API：大量の非同期処理

同じプロンプトパターンを数百・数千の入力に対して走らせる必要があるとき、同期 API は誤ったモデルだ。各同期呼び出しは完了までブロックする。スケールでは、アプリケーションはスレッドを焼くか、レートリミットに対して数千の同時接続を走らせるかになる。

Message Batches API は、単一のバッチ呼び出しで最大100,000リクエストまたは256 MB（先に来た方）を受け付ける。バッチを送信し、batch_id を受け取り、完了をポーリングする。バッチが終わると結果をダウンロードする。バッチリクエストのトークン単価は同期より低い。

トレードオフはレイテンシだ。バッチ処理は非決定的で、最大24時間（多くはもっと速い）かかる。このパターンは、リアルタイムのユーザーやり取りでなく、オフラインパイプライン・評価実行・データ処理ジョブに向く。

| ユースケース | 正しい API パターン | 理由 |
|---|---|---|
| ユーザーが写真をアップロードし、即時分類を期待 | 同期 API | リアルタイム応答が必要。バッチのレイテンシは対話利用に容認できない。 |
| 毎夜のパイプラインが5,000の顧客レコードを分類 | Message Batches API | レイテンシは制約でない。バッチのコスト削減と非同期処理の両方が価値がある。 |
| 評価実行が2,000例に対して新プロンプトをテスト | Message Batches API | リアルタイム要件のないオフラインタスク。バッチが正しいパターン。 |
| チャットボットがユーザーメッセージに返信を生成 | 同期 API | ユーザーが待っている。バッチは容認できない遅延を導入。 |

#### マルチモーダルとバッチが合うとき、合わないとき

この組み合わせは、同じアセットを再利用し数千の入力にまたがって構造化出力が必要なオフラインワークロードに合う。固定タクソノミに対して画像を分類する毎夜のパイプラインが教科書的なケースだ。Files API が冗長なアップロードを除き、Batches API がレイテンシを吸収し、構造化出力技法が結果を機械可読に保つ。

2つの失敗モードが適合を壊す。

1つ目はレイテンシの誤読。画像を伴うユーザー向けフローでバッチに手を伸ばすと、テストを通りプロダクションで失敗するシステムを生む。ユーザーが待っており、バッチは待ってくれないから。
2つ目はコンテキストコストの過小評価。画像と PDF は Claude がテキストを処理する前に予算を消費する。だからリクエストごとに複数の大きい画像を読み込むパイプラインは、スケールでトークン上限を突破する。構築前にプロダクションスケールの入力でトークンコストを測る。

## Screen 25 — Multimodal and Batch Ingestion — Watch Out（注意）· 4 min

### 実はバッチでなかったバッチジョブ

#### 設定

ジョブをチャンクに分けて1つずつ処理するのはバッチではない。手順を増やした直列化だ。Message Batches API が大量ワークロードのために存在するのはまさに、同期 API に対して入力をループすると、入力リストをどう切ろうとボリュームが本物になった瞬間にレートリミットに当たるからだ。

#### 毎夜のジョブがレートリミットに当たり続けた内部チャネルの会話

開発者は同じ毎夜の分類ジョブを3晩再実行し、毎回ほぼ同じ地点でレートリミットエラーに当たり続けている。シニア開発者が1つ問い、実際の問題を表面に出す。

> **Developer:** "My nightly job keeps hitting rate limits. I've already split it into smaller chunks. What else can I do?"
> **Senior Developer:** "How are you submitting them?"
> **Developer:** "I'm looping over the list and calling the API for each item."
> **Senior Developer:** "That is not batching. That is serial calls against the synchronous endpoint. Splitting the list into chunks does not change what the API sees: it still sees one request per item, back to back."
> **Developer:** "So the rate limit is firing because I am making thousands of synchronous calls?"
> **Senior Developer:** "Right. The Message Batch API takes up to 100,000 requests or 256 MB per batch in a single batch call, returns a batch_id, and processes them asynchronously. You poll for completion, which means your code repeatedly checks the status of the batch on a schedule until the API tells you it's done. The per-token cost is lower than synchronous, and the rate limit does not fire because you are not making thousands of individual requests."
> **Developer:** "And the tradeoff?"
> **Senior Developer:** "Latency is non-deterministic. Batch processing can take hours. If this were a real-time user interaction, it would be the wrong tool. However, this is perfect for a nightly classification run."

#### 注意すべきこと

リストをチャンクして同期 API をループするのは、バッチでなくても、バッチのように感じる。チャンクなし版と同じ数の API 呼び出しを生み、同じレートリミットに当たる。Message Batches API は、より小さいバッチサイズでなく、別の送信モデルだ。ワークロードが大ボリュームでオフラインのときは常に使い、反対側にユーザーが待っているときだけ同期 API に手を伸ばす。結果は、リクエストが送信された順でなく、任意の順で戻る。各リクエストの `custom_id` フィールドで、結果を入力にマッチさせる。

## Screen 27 — Recap: Eight Takeaways（ふりかえり：8つの要点）· 3 min

### 8つの要点、可能化目標ごとに1つ

1. **プロンプトが失敗したとき、失敗タイプが欠けている技法を教える。**
違う形で出力されるなら出力制約の欠落、ターンをまたぐドリフトなら system prompt の指定不足、構造の幻覚なら few-shot 例の不在を指す。指示を言い換えてもう1回という衝動はまず効かない。それらの失敗はどれも言い回しの問題でないから。まず失敗タイプを診断し、それに効く技法を追加する。プロンプトレベルの指示では不十分で、未テストの入力がまだパーサーを壊すときは、structured outputs で出力制御を API に移す。JSON outputs はスキーマに対して最終応答を制約し、strict tool use は Claude がツールに渡す引数を検証する。初回呼び出しのコンパイルレイテンシと追加入力トークンのコストと引き換えに。

2. **プロンプトを調整する前に、推論の深さをタスクに合わせる。**
推論のパスが答えを変える場合だけ推論を有効にし、毎呼び出しで上げるのでなく effort setting を問題に合わせて調整する。thinking block は変更せず API に戻すことを忘れない。さもないと次のリクエストが失敗する。推論を有効にするかどうかとは別の、どのモデルを実行するかの選択は、これより前の MSO Foundations モジュールで教える。

3. **ストリームが終わることは、メッセージが完成することではない。**
ストリーミングは、partial なイベントから自分で応答を組み立てるコストで、体感レイテンシを買う。ブロックが閉じた後でのみ作用し、message_stop の後でのみターンを履歴に commit し、中断したストリームでは partial ターンを捨ててリトライする。認識すべき失敗モードは、スキーマでなく、落ちたストリームの半分構築されたブロックに遡る、リトライ時のツール使用エラーだ。

4. **誤ったツール選択はすべてスキーマに遡り、ほとんどは description に遡る。**
Claude は description フィールドを読んでユーザーの要求に合致させることでツールを選ぶ。つまり、両方とも "use this to find information" と言う2つのツールは、入力スキーマが全く似ていなくても、Claude 側からは区別できない。誤ツールのバグの多くを解く1文は排除条件だ。すべての description にある、ツールをいつ呼ばないかを名指しする行で、最初の誤呼び出しがログに現れた後でなく設計時にスキーマに書き込む。誰かが既にツールを書いているとき、MCP はすべてのスキーマを手で書く代わりに、保守されたサーバーに接続させる。ただし接続した各サーバーは、ツールが使われるかにかかわらずツール定義をコンテキストウィンドウに追加する。だから意識的に接続し、読み込みコストを制御する。

5. **コンテキストは固定の予算で、ツール出力はループ内の他の何より早くそれを使う。**
プロダクションのツール出力は開発のフィクスチャより3〜5倍長い。だからテストで50ターンを綺麗に保つセッションが、出荷されると8ターン目で天井に当たる。Pruning・compaction・サブエージェント引き継ぎはそれぞれ違う方法で余地を買い戻し、どれを適用するかは、まだ前の状態が必要かによる。ツール選択が固定ターン数の後に劣化し始めたら、スキーマでなくウィンドウが最初に見る場所だ。

6. **workflow-or-agent の決定が後に続くすべてのコストを決め、人間のチェックポイントは設計に属する。**
正確なステップをコードで書けるならワークフローが正しい呼び出しで、ゴールとツールは指定できるがその間の経路は指定できないならエージェントが正しい呼び出しだ。どちらの方向の誤選択もプロダクションでのみ表面化する。ワークフローで済むところにエージェントを使うと、コンテキストコストとトランスクリプトに住む振る舞いを加える。エージェントが必要なところにワークフローを使うと、入力が経路を外れた最初の瞬間に壊れる。ツールが不可逆な行動を取れるなら、human-in-the-loop チェックポイントは最初の write が顧客環境に届いた後でなく、ループをワイヤリングする前に入れる。

7. **メモリスコープは実装しやすさでなく、セッションの形状で決まる。**
コンテキスト内メモリは最も書きやすいパターンで、だからこそ、プロダクションのセッションが開発の長い連続セッションより短く多数であることが分かったとき、最も早く失敗するパターンでもある。外部ストレージはレイテンシを加えるが状態はセッションをまたぐ。要約メモリはコストを削るが要約器プロンプトが保たなかったものはすべて失う。ステートレスは完了して閉じるジョブに正しい。プロダクションの圧力の下でコンテキスト内から外部へのリファクタは約1時間。設計時に同じ選択を意識的にするのは約20分。タスクをまたいで繰り返し可能な指示を運ぶことは、状態を運ぶこととは別の問題で、そのパターンが Skill だ。毎セッションに注入される指示でなく、description に合致させて Claude がオンデマンドで読み込む markdown ファイル。

8. **取り込みコードを書く前にマルチモーダル入力のコストを計算し、API をワークロードに合わせる。**
画像は ⌈width / 28⌉ × ⌈height / 28⌉ の視覚トークンを消費し、画像あたりの天井はモデル階層で異なる。最新モデルでの高解像度のオリジナルは、テストセットのサムネイルのコストの何倍もかかる。だから公式は手元の入力でなく、プロダクションで予想する最も大きい入力に対して走る必要がある。Inline base64 は単発画像に合い、Files API はリクエストをまたいで再利用されるアセットに合い、Message Batches API は非決定的なレイテンシと引き換えに低いトークン単価でオフライン作業を扱う。避ける価値のある失敗は、同期 API をループで呼び、それをバッチとして扱うこと。

## Screen 28 — Glossary: Key Terms（用語集：主要用語）· 3 min

### このモジュールの主要用語

アルファベット順。

- **Claude Agent SDK**
`@anthropic-ai/claude-agent-sdk`（TypeScript）／`claude-agent-sdk`（Python）として配布される、マネージドなエージェントランタイム。Claude Code を支えるのと同じエージェントループ（反復・ツール実行・観察・終了）へのプログラム的アクセスをパートナーに与え、ターミナルで Claude Code を走らせる代わりに、パートナーが自分の製品の中にエージェントを埋め込めるようにする。API の上の薄い便利ラッパーでエージェントループを走らせない Anthropic SDK とは別物。

- **Context Window（コンテキストウィンドウ）**
モデルが1回のリクエストで処理できるトークンの総量。system prompt・会話履歴・ツール定義・ツール結果・モデル自身の出力を含む。累計が上限に達すると、新しい内容を追加する前に、以前の内容を削除するか要約しなければならない。

- **Function signature（関数シグネチャ）**
関数の宣言を意味するプログラミング用語。関数の名前と、受け付けるパラメータのリスト（名前・型・デフォルト値を含む）。

- **HITL**
Human-in-the-loop。結果的な行動が取られる前に、自動化プロセスに人間のレビューまたは承認ステップを挿入すること。

- **Refactor（リファクタ）**
外から見た振る舞いを変えずに、コードの内部構造を変えること。より綺麗に・速く・テストしやすく・拡張しやすくするため再編成・改名・実装の書き直しをするが、システムの他の部分が見る振る舞いは同じまま。

- **SOC 2**
Service Organization Control 2。SaaS ベンダーやクラウドサービスプロバイダーが、自社のセキュリティ慣行が認知された基準を満たすことを示すよう求められたときに最もよく引用される標準。顧客データをどう取り扱うかを評価するため、米国公認会計士協会（AICPA）が開発した監査フレームワーク。

- **State（状態）**
エージェントがターンをまたいで運ぶ情報。これまでの会話・ユーザーが求めたもの・前のツール呼び出しの結果。

- **Stop_reason**
モデルが生成を止めた理由をコードに伝える、API 応答のフィールド。エージェントループに最も関連する2つの値は、`end_turn`（Claude が終わり、これ以上の行動を要求していない）と `tool_use`（Claude が1つ以上の tool_use block を発行し、続行前に結果を待っている）。

- **Subagent（サブエージェント）**
オーケストレーティングエージェントが、個別のサブタスクを扱うために立ち上げる別のエージェントインスタンス。サブエージェントは親セッションから会話履歴・skills・コンテキストを継承せず、それぞれがクリーンに始まり、必要な指示とツールで明示的に構成されなければならない。結果はオーケストレーターに戻り、より大きなタスクに組み込まれる。

- **Token（トークン）**
Claude がテキストを測り処理するために使う単位。トークンあたりの平均文字数は対象モデルのトークナイザに依存し、モデル世代で異なる。文字数／トークンの目安はモデル依存とみなし、ビルド時に現在のトークナイザの振る舞いを確認する。トークンはコンテキストウィンドウのすべて（プロンプト・応答・ツールスキーマ・ツール結果）で消費される。価格とコンテキスト予算の計算の両方の基礎。

- **Tool_use_block**
Claude が関数を呼びたいときに assistant が返すコンテンツブロック。ツール名・一意の ID・コードに渡したい入力引数を含む。すべての tool_use block は、直後の user ターンで、同じ ID を exactly に保った一致する tool_result block で応答されなければならない。
