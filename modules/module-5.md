## Screen 1 — Orientation（オリエンテーション）· 2 min

### このモジュールを終えるとできること

ビルドをうまくドキュメント化すれば、次のエンゲージメントはゼロからでなくそこから構築する。

最後の3つのモジュールで、プロダクションエージェントを構築し・正しいパーミッションとコンテキスト統制で Claude Code に繋ぎ・セキュリティレビューを通る MCP 接続を設定した。それぞれは稼働するビルドだった。

このモジュールは、ビルドが稼働した後に何が起きるかを覆う。次のエンゲージメントでゼロから再構築するか、一度パッケージ化して次のチームがただ設定するようにするか。2つ目の経路が、繰り返される再構築でなく新しい仕事にあなたの時間を解放する。

このモジュールを終えると、次ができるようになる。

1. 稼働するソリューションを再利用可能なアクセラレータとしてパッケージ化する。パラメータ化されたエージェントテンプレート・構成可能な MCP サーバー・移植可能な eval スイートのいずれであれ、次のエンゲージメントが完全に再構築するでなくアセットを設定するように。
2. ツール・パターン・修正を、ドキュメント化されたチャネルを通じてコントリビュートし、メンテナーが受け入れられるよう準備し、プライベートなアセットを共有インフラに変える。
3. Claude ワークロードが、first-party API・Amazon Bedrock・Google Vertex AI・サードパーティプラットフォームをまたぐどこで走るかを選び、モデルやプロンプトの変更がプロダクションを静かに壊さないよう、出荷するものをバージョン化する。
4. それらのプラットフォームをレイテンシ・コンプライアンス・コストで比較し、チームが手を伸ばしたデフォルトでなく、調達とセキュリティチームが署名できる選択にする。
5. 複数の Claude デプロイを1つのワークフローに調整するアプリケーションを構築し、データと identity の境界がセキュリティやコンプライアンスレビューの下で保たれるようスコープする。

このモジュールは、稼働するビルドを持ち、今それを長持ちさせなければならない Developer 向けだ。実践的で、コードファーストで、パターン志向で、前のモジュールはそれを仮定し構築した。この時点で、プロダクションエージェントを書き・Claude Code で構成し・セキュリティレビューを通る MCP 接続を繋ぎ・eval ですべてを証明できる。このモジュールはそのどれも教え直さない。コードが正しく走る瞬間に拾い上げ、より難しい問いをする。他の誰かがそれを再利用できるか・メンテナーが受け入れられるか・モデル更新を生き残れるか・セキュリティや調達チームがそれが走る場所に署名できるか。ここでの作業は、コードを書くことより、完成したコードを再利用可能・デプロイ可能・書かなかった人に擁護可能にする決定についてだ。

### このモジュールにおける「ビルド」

このモジュールのすべては、1つの繰り返す隙間にかかっている。稼働するビルドは、再利用・レビュー・デプロイを生き残るビルドではない。開発では、テンプレートは走り・コントリビューションはあなたの問題を解き・モデルは応答し・プラットフォームは構築しやすく・各プラットフォームは独自のテストに通った。そのどれも完成状態ではない。同じテンプレートは、あなたと一度も話さなかったチームのために構成されなければならない。同じコントリビューションは、何も再構築しなければならないメンテナーによって検証可能でなければならない。同じモデルは、上流の変更が驚きでなく決定になるよう pin されなければならない。同じプラットフォームは、顧客の常駐とコンプライアンスレビューをクリアしなければならない。同じ接続されたプラットフォームは、監査の下で信頼境界を保たなければならない。このモジュールの各トピックは、同じ教訓の違う版だ。コードが働き始めるポイントが、このモジュールの作業が始まる場所だ。あなたが思うより多くのこれらの決定は、顧客の既存のクラウド・コンプライアンス姿勢・レビュープロセスによって駆動される。

## Screen 2 — Packaging for Reuse — Teaching（再利用のためのパッケージ化）· 16 min

### 稼働するビルドをパッケージ化し、次のエンゲージメントがアセットから始めるようにする

前のモジュールを、走るビルドで終えた。エージェントループ・構成された MCP サーバー・プロンプトが働くことを証明する eval。チームで最も時間がかかり高価なのは、次の顧客のために同じものを再構築するために費やされるエンジニアリング時間だ。

#### アクセラレータがすること：再利用可能な部品を保ち、残りを切り離す

アクセラレータは、将来のエンゲージメントが空のリポジトリでなく稼働する基盤から始めるようパッケージされたソリューションだ。ブループリントの言葉では、これは再利用のためのパッケージ化だ。エンゲージメント固有のコードを再利用可能なコアから切り離し、残りをパラメータ化する。稼働するビルドを取り、顧客固有の部品を切り離し、ドキュメント化されたデフォルトを持つパラメータとして公開する。アセットは完全に書き直されるのでなく構成される。ビルドが新鮮な間に再利用のためにパッケージ化するのは、なぜ値が hardcode されたかを知る人がいなくなった数か月後に意図を再構築するより安い。

#### ほとんどの再利用可能な作業は異なるアセットタイプに分かれ、それぞれ違ってパッケージ化する

ほとんどの再利用可能な作業は、このモジュール全体で使われる3つのカテゴリの1つに分かれる。テンプレート・構成可能なサーバー・移植可能な eval。各タイプは異なる種類の作業を持ち、独自の方法でパッケージ化されなければならない。誤ったタイプに手を伸ばすと、アセットを適用しにくいまま再利用可能に見せることができる。

| アセットタイプ | 何を束ねるか | 正しいパッケージ化が必要なもの |
|---|---|---|
| **Agent Template** | 稼働するエージェントからの system prompt・ツールスキーマ・ループ構造。 | ドメイン固有の値をドキュメント化されたデフォルトで構成に引き出し、新しいチームがループを編集するのでなく値を設定するように。 |
| **MCP Server Package** | サーバーが公開するツールと、その入力とインストールチームが制御するスコープ。 | 各ツール入力をドキュメント化し、インストールチームがスコープを設定できるようにし、サーバーがコード編集なしで新しい環境にインストールされるように。 |
| **Eval Suite** | アセットが働くことを証明する採点されたテストセットとジャッジルーブリック。 | データセットとルーブリックを一緒に出荷し、新しいチームが自分のコンテキストで走らせ、そこでアセットがまだ働くことを確認できるように。同じ eval スイートはデプロイでのゲートとしても働く。新しいモデルバージョンをプロダクションに昇格させるとき、バージョンがライブになる前に pin された基準スコアに対して走らせる。 |

エージェントをテンプレートでなく緩いスクリプトのセットとして出荷するのが、誤ったアプローチの最も一般的な版だ。スクリプトは走るので再利用可能に見えるが、顧客固有のすべての値は異なるファイルに埋もれ、次のチームは1つのアセットを構成するでなくコピーして分岐させる。

#### コードと前提の両方をドキュメント化する

コードは振る舞いを記述する。ドキュメンテーションは、ソースを読んで確実に推論できない将来のビルダーが覆う。アセットが環境についてする前提・期待する入力・既に扱う失敗モード・まだ働くかを定義する eval。これがないと、次のチームはアセットをブラックボックスとして扱い、再構築する。

#### 監査ログをパッケージの一部として束ねる

規制顧客のレビュアーは、アセットがどのデータに触れるか・どの identity で振る舞うか・どんなログを残すかを尋ねる。これらのないアクセラレータはデモに通り、最初のセキュリティレビューで止まる。監査ログをパッケージの一部として扱う。

#### パッケージ化チェックリスト

パッケージ化する間、このチェックリストをビルドの隣に置く。各列はアセットごとに1回決める決定だ。

| アセットタイプ | 何をパラメータ化するか | 何をドキュメント化するか | 監査のために何を束ねるか |
|---|---|---|---|
| **Agent template** | 顧客ごとに変わるすべての値：プロンプト・パス・スコープ・参照によるクレデンシャル・閾値。 | 環境の前提・期待する入力・扱った失敗モード・働くことを定義する eval。 | 触れたデータ・振る舞った identity・アセットがしたことのログ。 |
| **MCP Server** | スコープ・参照によるクレデンシャル・顧客/環境ごとのパス。 | ツールごとの期待する入力・スコープ境界・扱った失敗モード。 | 触れたデータ・振る舞った identity・アセットがしたことのログ。 |
| **Eval Suite** | 顧客/環境ごとに変わる閾値とデータセットパス。 | ルーブリックの論理・スコアが何を意味するか・アセットが pin される基準。 | 触れたデータ・振る舞った identity・アセットがしたことのログ。 |

- **Handles well（うまく扱える）:** ビルドが新鮮な間にパラメータ化することが、1つの納品を、次のエンゲージメントが数時間で構成するアセットに変える。
- **Adds cost or complexity（コスト・複雑さを加える）:** 一般化可能なものを顧客固有のものから切り離し、前提をドキュメント化することが、最初のビルドに実時間を加える。
- **Use a different approach（別の手法を使う）:** 顧客が決して再利用しない単発には、パッケージ化のオーバーヘッドは割に合わない。ビルドを出荷して次へ進む。

## Screen 3 — Packaging for Reuse — Watch Out（注意）· 2 min

### 速く出荷され、再利用できなかったテンプレート

#### 設定

hardcoding は速く出荷し、期限の下で作業していたので、デモを作る値を hardcode した。テンプレートは働いた。だから次のチームが再利用を試みるまで、誰もそれを見なかった。

これは、再利用の試みが失敗した後にチームが書くようなポストモーテムで、誰かが間違いとラベル付けする前に失敗の形を見せる。

#### 何が起きたか

あるチームは、顧客エンゲージメントのためにエージェントテンプレートを構築し、期日通りに出荷した。期限に間に合わせるため、顧客固有の値はコードに直接入った。リポジトリパス・モデル名・レビュー閾値・その顧客のドメインに固有の少数のプロンプト断片。テンプレートは走り、エンゲージメントは閉じ、ビルドは再利用可能とラベル付けされて共有リポジトリに入った。

数か月後、2つ目のチームが似たエンゲージメントのためにそれを拾った。彼らはそれを構成できなかった。構成するものが何もなかったから。変更が必要なすべての値は、2つ目のチームがファイル全体を読まなければ見えないループに焼き込まれていた。どの値が顧客固有で、どの値が load-bearing かを言うドキュメントがなかった。束ねられた eval もなかったので、編集に推測した後でも、テンプレートが新しいコンテキストでまだ働くことを確認するものが何もなかった。彼らはゼロから書き直さなければならなかった。

#### なぜ壊れたか

ビルドは、再利用できる瞬間でなく走った瞬間に完成したものとして扱われた。hardcoding は期限の下では妥当な呼び出しで、決して再訪されなかった。稼働するテンプレートは再利用できないことを告げない。コストは、2つ目のチームが、パッケージ化が防ぐはずだった再構築に払い、テンプレートが行き止まりだと発見して失った時間と共に、初めて現れた。

#### 注意すべきこと

走るテンプレートは再利用のためにパッケージ化されていない。これらは異なる完成状態だ。警告サインは3つのものの不在だ。顧客固有の値が属する場所にパラメータがない・前提を記述するドキュメンテーションがない・アセットが異なるコンテキストでまだ働くことを証明する束ねられた eval がない。ビルドが新鮮な間にアセットをパッケージ化する。何が顧客固有かの知識は、それを持っていた人がいなくなった後に最も再構築に高くつく。

## Screen 5 — Contributing Back — Teaching（コントリビュート）· 12 min

### アセットをプライベートな再利用から、メンテナーが受け入れる共有インフラへ

あなたは既に、アセットを共有可能にする作業のほとんどをした。自分のチームが再利用するためにパッケージ化したとき、パラメータを引き出し・前提を書き下し・eval を束ねた。パラメータは、アセットが書き直されるでなく構成できることを示す。ドキュメント化された前提は、アセットが期待する環境をメンテナーに告げる。束ねられた eval は、それがまだ働くことを確認する方法を与える。内部再利用のためにパッケージされたアセットは、メンテナーが受け入れるために必要なものに既に近い。

コントリビューションチャネルは、そのパッケージされたアセットを受け取るよう設計されている。バージョン・インストール手順・コンポーネントを単一ユニットとして運び、あなたと一度も話さなかったチームがインストールして同じ稼働セットアップを得られるように。

#### コントリビューションを、それのために構築されたチャネルに合わせる

コントリビュートとは、アセットをドキュメント化されたチャネルを通じてプライベートな再利用から共有インフラへ移すこと。各チャネルは特定の種類のコントリビューションのために構築されている。Claude Cookbook は、焦点を絞った参照実装の GitHub リポジトリ。自己完結した単一・または複数パターンの実装で、明確に示されエンドツーエンドで働くもののために設計されている。オープンソースの MCP サーバーとツールは、それぞれ独自のコントリビューション慣習を持つ独自のリポジトリに住む。完全な複数コンポーネントアプリケーションを Cookbook に送るのは不一致だ。リポジトリはアプリケーション全体でなく1つの焦点を絞ったパターンをレビューするよう設定されているので、それほど大きい提出はレビュアーが探しているものに合わず、止まる。最初のステップは、コントリビューションをそのために構築されたチャネルに合わせること。焦点を絞った例が属する場所に完全なアプリケーションを置くのが、コントリビューションが決してレビューされない最も一般的な理由の1つ。

#### コントリビューションの検証を可能にするもの

メンテナーは、検証できるコントリビューションを受け入れる。バーは、コードがどれほど賢いかでなく、彼らがチェックする必要のあるものによって設定される。4つのことがその検証を可能にする。

1. **コードは1つのことをする。**広がったコントリビューションは、レビュアーに評価する前にあなたの意図を再構築させる。
2. **例がそれが走るのを見せる。**レビュアーは振る舞いを見るためにハーネスを構築すべきでない。
3. **テストがそれが働くことを証明する。**テストは、メンテナーが推論を自身で再現せずに結果を検証できるようにする。
4. **短い声明が前提を名指しする。**さもなければ、最初の失敗がメンテナーの問題になる。

#### 権利と帰属は技術レビューの前に来る

ライセンスと帰属は、コントリビューションが受け入れられるかどうかを決め、だから技術レビューの前に来る。顧客エンゲージメントから持ち込まれたコードは、それがどこに行けるかに制約を持つかもしれない。それをコントリビュートする権利を持つことの確認と、構築したものに基づくものの帰属は、コントリビューションが最初に通らなければならないゲートだ。これを飛ばすことが、コントリビューションを、法務チームが後で解かなければならない問題に変える。

ここで働く例は、カスタマーサービスエージェントのケース。エンゲージメント中に構築された再利用可能な会話処理パターンは、顧客固有を剥がされ、Cookbook のための一般的な例として準備される。コントリビュートバックの動きは、このカリキュラムの3つの役割すべてで共有される。Developer としてのあなたの仕事は技術的準備だ。焦点を絞ったコード・例・テスト・前提・権利チェック。エンゲージメントのコンテキストはより広いチームから来る。

#### コントリビューション準備の参照

| チャネル | メンテナーがチェックするもの | ライセンスと帰属 | クリアすべき例とテストのバー |
|---|---|---|---|
| 焦点を絞った例には Cookbook、ツールや修正にはそのツールやサーバーの独自のリポジトリ。 | コードが1つのことをし、完全に読めること。 | エンゲージメントからのコードをコントリビュートする権利を確認し、先行作業に帰属すること。 | 振る舞いの記述でなく、それを証明するテストに加え、実行可能な例。 |

- **Handles well（うまく扱える）:** パッケージされたアセットは、例・テスト・権利チェックだけで、他の人が構築する共有インフラになれる。
- **Adds cost or complexity（コスト・複雑さを加える）:** メンテナーのバーとライセンスゲートをクリアすることは、コードを自分のために走らせる上での実作業だ。
- **Use a different approach（別の手法を使う）:** クリアできないエンゲージメントライセンス制約をコードが運ぶとき、コントリビュートせず、所有者にエスカレートする。

## Screen 6 — Contributing Back — Watch Out（注意）· 2 min

### メンテナーが検証できなかったプルリクエスト

#### 設定

あなたの問題を解いた正確なコードでコントリビューションを開いた。それは自然な選択だった。あなたのケースで働き、アクセス可能だったから。それはあなたに働いた。それがまさに、見知らぬ人がそれを信じるために必要なすべてが欠けていた理由だ。

これは内部チャネルからのやり取りで、メンテナーがプルリクエストの沈黙をどう説明するかを聞ける。

#### やり取り

> **Developer:** My PR has been open three weeks with no review. The code works, I use it every day. What is the holdup?
> **Maintainer:** It probably works for you. The problem is I can't tell. There is no test I can run, no example that proves the behavior, and nothing saying what it assumes about the environment.
> **Developer:** So, you want me to add a test and an example?
> **Maintainer:** Yes. A contribution a reviewer cannot verify sits at the back of the queue until someone has time to reconstruct what it does. A focused PR with a test and an example gets reviewed fast because there is nothing left for me to reverse-engineer.

#### なぜ壊れたか

コードは正しかった。コントリビューションは、メンテナーが開発者の作業を再構築せずに検証できなかったため、止まった。その隙間は見落としやすい。作者は欠けたコンテキストを既に持っているから。例・テスト・前提の声明は、コードを作った人にはすべて明らか。しかしメンテナーにはそうでなく、意図を再構築しなければならないレビュアーは常に最後にそれをする。

#### 注意すべきこと

プルリクエストは、レビュアーが検証できないものの上で止まる。コントリビューションを開く前に、それが走るのを見せる例・振る舞いを証明するテスト・前提を名指しする短い声明を加える。それら3つの特徴が、コントリビューションを列の後ろから速いレビューに動かす。メンテナーにリバースエンジニアするものを残さないから。

## Screen 8 — Requirements & Lifecycle — Teaching（要件とライフサイクル）· 8 min

### ビジネス要件から機能要件とインフラ要件へ

この後に続くデプロイメントプラットフォームの決定はすべて、要件が既に存在することを前提とする。データ常駐のルール、レイテンシの目標、アイデンティティモデル。この画面は、それらの要件がどこから来るかを扱う。ビジネス上の問題を、デプロイメントの決定を擁護できる機能要件とインフラ要件に変える。

#### ビジネス上の問題から機能要件を捉える

**機能要件**は、システムがしなければならないことを、検証できる詳細さで記述する。ビジネス上の問題（例: 「サポートエージェントがより早く回答するのを助ける」）は、まだ要件ではない。機能要件はそこから導出される（例: 「各チケットを4つのキューの1つに分類する。関連するポリシーを引用した返信のドラフトを作る。人間の承認なしには自動送信しない」）。規律は、それぞれを検証可能な振る舞いの記述として書くこと。曖昧な目標は設計も検証もできないが、具体的なものは eval の1行になり、レビューでの1つの基準になる。

#### インフラ要件の導出

**インフラ要件**は、デプロイメントが満たさなければならない非機能的制約だ。そのほとんどはビジネス上の問題の中に書かれていない。代わりに、ビジネス上の問題が含意する問いを立てることで導出する。

- **レイテンシ**: 応答はどれだけ速くあるべきか。ユーザーがいる場所で測って。
- **スケール**: リクエストはいくつか。ピーク時には。
- **データ常駐**: データはどこで処理されなければならないか。どの規制の下で。
- **アイデンティティ**: 誰が、どのクレデンシャルの下で振る舞い、何が監査可能でなければならないか。

レイテンシ、スケール、データ常駐、アイデンティティは、デプロイメントプラットフォームを決めるインフラ要件として最も多く、プラットフォームが別の理由で選ばれる前に捉えるのが最も容易だ。

#### 決定を擁護できるように要件を文書化する

要件を書き残すのは、デプロイメントの決定が、それらを集めた人でない人々によってレビューされるからだ。機能の振る舞い、インフラの制約、各制約の根拠となる規制をカバーする短い要件記録が、プラットフォームの選択を、慣れではなく要件から導かれるものとして擁護できるようにする。この記録が、次の画面のデプロイメント決定が読む入力だ。

- **Handles well（うまく扱える）:** いかなるプラットフォームが選ばれる前に、ビジネス上の問題を検証可能な機能要件とインフラ要件に変えること。
- **Adds cost or complexity（コスト・複雑さを加える）:** 前もってインフラ制約を引き出すことは、チームが飛ばしたがるスコーピングの会話を要する。
- **Use a different approach（別の手法を使う）:** レビューも規制データもない捨てプロトタイプには、軽量なメモで十分だ。

## Screen 10 — Requirements & Lifecycle — Teaching（要件とライフサイクル）· 8 min

### Claude アプリケーションのシステムライフサイクル

先ほど捉えた要件は、より長い弧の最初のフェーズだ。この画面はその弧をシステムライフサイクルと名付け、このモジュールの残りのデプロイメント、バージョニング、境界の作業が、無関係なタスクとしてではなく正しいフェーズに位置づけられるようにする。

#### Claude アプリケーションに適用したライフサイクルのフェーズ

Claude アプリケーションは、他のエンジニアリングされたシステムと同じライフサイクルを通り、モデルの作業がその上に写像される。

1. **要件（Requirements）**: 機能とインフラのニーズを捉える。
2. **設計（Design）**: プラットフォーム、モデル、信頼境界を選ぶ。
3. **構築（Build）**: エージェント、ツール、プロンプトを書く。
4. **テスト（Test）**: eval、ユニット、統合、エンドツーエンドの検査。
5. **デプロイ（Deploy）**: バージョンを pin し、プロモーションを eval でゲートする。
6. **運用（Operate）**: コスト、レイテンシ、エラーを計装し、ガードレールを強制する。
7. **反復（Iterate）**: プロダクションの知見を要件にフィードバックする。

これらのフェーズは、前のモジュールが1つずつ教えたものと同じだ。それらをライフサイクルとして特定することが、どう繋がるかを示す。

#### フェーズ間のゲーティング

ゲートとは、あるフェーズから次へ移る決定であり、規制されたエンゲージメントが統制を保つ場所だ。プラットフォームがデータ常駐要件を満たすまで設計から構築へは移らない。新しいバージョンが pin された基準に対して eval をクリアするまで、デプロイから完全なプロダクションへは移らない。エンジニアリングの作業を正しいフェーズに置き、ゲートを飛ばさないことが、Claude アプリケーションをレビュー可能に保つ。

- **Handles well（うまく扱える）:** エンジニアリングの各作業を、それが属するライフサイクルのフェーズに、定義された成果物とゲートと共に置くこと。
- **Adds cost or complexity（コスト・複雑さを加える）:** フェーズ間のゲーティングは、期限の下にあるチームが飛ばしたがるチェックポイントを加える。
- **Use a different approach（別の手法を使う）:** 単発の実験はフェーズを潰してよいが、規制されたデプロイメントはそうできない。

## Screen 12 — Deployment & Versioning — Teaching（デプロイメントとバージョニング）· 15 min

### Claude ワークロードがどこで走るかを選び、出荷するものをバージョン化する

パッケージ化されたアセットもコントリビュートされたアセットも、何かがそれらを走らせるまでは単なるコードだ。アセットは今、別の問いに直面する。どこで走るか、そして上流の変更がプロダクションでの追跡されない変更にならないよう、どうバージョンを固定するか。このプラットフォームの決定は、純粋に技術的メリットについてであることは稀だ。実際には、顧客が既にクラウドインフラ、アイデンティティ管理、コンプライアンス契約をどこに持っているかによって形作られることが多い。最初の問いは通常、顧客が既に信頼し運用しているプラットフォームについてだ。

#### 顧客のクラウドが通常プラットフォームを決める

デプロイメントプラットフォームは、Claude ワークロードが走る環境だ。同じモデルはいくつかのデプロイメント環境で走れ、顧客の既存のクラウドが通常どれかを決める。First-party Claude API は Anthropic 自身の環境で、典型的に新機能を最初に受け取る。AWS 上の Claude Platform は、顧客の AWS アカウントを通じて Anthropic 自身のモデル ID とライフサイクルを使ってアクセスされる。推論は Anthropic が運用し、AWS 境界の外にある。Amazon Bedrock は2つの統合を提供する。Claude in Amazon Bedrock は `/anthropic/v1/messages` で Messages API を使い、広範な機能パリティを持つ。機能固有の要件は Bedrock のドキュメントに対して確認せよ（features-not-supported リストが存在する）。一方 Claude on Amazon Bedrock (legacy) は `InvokeModel`/`Converse` API を使い、ARN バージョンの識別子を持つ。Google Vertex AI は Google Cloud 内で同じことをする。第三者プラットフォーム（例: Microsoft Foundry）は、顧客が既に使っている製品の中に Claude を埋め込む。Microsoft Foundry は2つのホスティング形態で Claude を提供する。Hosted on Azure（現在 Claude Opus 4.8、Claude Sonnet 5、Claude Haiku 4.5。推論が Azure インフラ上でエンドツーエンドに走る。一般提供）と Hosted on Anthropic（他のすべての Foundry Claude モデル。推論は Anthropic 運営インフラ上）。規制顧客のデータ常駐の前提は、特定のモデルのホスティング形態に依存する。ビルド時に Microsoft とホスティング形態、現在のモデル分割を確認せよ。

#### アイデンティティとデータ常駐はセキュリティ上重要だ

アイデンティティとデータの位置は、あなたのコードでなくプラットフォームによって答えられる。Bedrock は AWS のアイデンティティを使い、データを顧客の AWS 境界内に保つ。Vertex は Google Cloud のアイデンティティと境界を使う。両者とも、データ常駐が制約のときリージョナルルーティングを提供する。プラットフォームを顧客の既存のコンプライアンス契約に合わせることが、データ常駐レビューをゼロからやることを避ける。

#### 上流のモデル変更が静かなプロダクション変更にならないよう、バージョンを pin する

バージョニングが、モデルやプロンプトの変更がプロダクションでの静かな変更になるのを防ぐ。すべての Claude モデル ID は特定のモデルスナップショットを指す。`Opus` や `Sonnet` のようなエイリアスは便利だが、時間と共に進化し、デプロイメントプラットフォームをまたいで異なるバージョンに解決されうる。pin されたフルモデル ID は固定されたスナップショットに解決される。移動するエイリアスでなく特定のモデルバージョンを pin することで、上流のモデル更新が静かなプロダクション変更でなく意図的な選択になる。そしてプロンプトとアセットをコードと共にバージョン化する。最後に、前のバージョンを利用可能に保ち、回帰がロールバックできるようにする。unpin されたデプロイメントは、すべての上流のモデル更新を出力への追跡されない変更にする。

最初の行は移動するエイリアスに従う。2つ目はスナップショットを pin する。
```python
# Pre-4.6 example: a convenience alias can resolve to a new
# version without you knowing
model = "claude-haiku-4-5"

# Pre-4.6 pinned snapshot: the version is fixed until you change this line
model = "claude-haiku-4-5-20251001"
```
Claude 4.6 以降では、モデル ID 単独で特定のスナップショットに pin する。より前のモデルでは、ID に日付サフィックスが必要。ビルド時に platform.claude.com で現在の慣習を検証せよ。

#### eval を通じてバージョンをプロモートする

プロモーションを eval スイートでゲートする。新しいバージョンをトラフィックの一部に送り、pin された基準と比較し、結果でプロモートまたはロールバックする。ここで eval は1回限りのテストでなくなり、デプロイメントゲートになる。

- **Handles well（うまく扱える）:** プラットフォームを顧客のクラウドに合わせ、バージョンを pin することが、マイグレーションをレビュー可能にし、ロールバックを可能にする。

## Screen 13 — Deployment & Versioning — Watch Out（注意）· 3 min

### モデルエイリアスが動いた時に壊れたデプロイメント

#### 設定

推奨バージョンを指すエイリアスに対して出荷した。それが便利なデフォルトであり、最新のモデルをただで得られたからだ。それは働いた。やがてエイリアスが進み、ただだったものには代償があることが分かった。

これはプロダクションログからのトレース抜粋で、インシデントの後にスクロールして戻るような種類のものだ。出力の形が変わった日と、ロールバック先が何もなかった理由を示す。

#### ログ
```
--:  deploy: model="opus"  status=ok
--:  alias advanced -> new opus version (no app change)
--:  parser: KeyError "summary" in response payload
--:  Error: output shape changed; downstream parse failed
--:  rollback attempted -> no pinned prior version retained
--:  incident: hotfix parser; root cause = unpinned deployment
```

#### なぜ壊れたか

アプリケーションは変わらなかったが、エイリアスは変わった。pin された前のバージョンは保持されておらず、ロールバック先が何もなかった。ホットフィックスはパーサーを修理したが、unpin されたデプロイメントをそのままにした。

#### 注意すべきこと

エイリアスは移動する標的に解決される。pin されたフルモデル ID は固定スナップショットだ。上流の更新が意図的に採用するものになるよう、フルモデル ID を pin する。回帰がホットフィックスでなくロールバックになるよう、前の pin されたバージョンを利用可能に保つ。プロモートする前に新しいバージョンを eval を通じ、出力形状の変化がプロダクションでなくテスト実行に現れるようにする。

## Screen 15 — Comparing Platforms — Teaching（プラットフォームの比較）· 12 min

### レイテンシ、コンプライアンス、コストでプラットフォームを比較し、選択がレビューを生き残るようにする

最後の2画面でプラットフォームを選び、バージョンを pin した。その選択は顧客のクラウドにとって正しかったが、「彼らのクラウドにとって正しい」はまだ調達やセキュリティチームが署名する議論ではない。

#### 顧客のリージョンからレイテンシを測る

レイテンシは、プラットフォームが顧客に対してどこで走るか、そして新機能へのアクセスがどうルーティングされるかに依存する。顧客自身のクラウドリージョンで走るプラットフォームは、より遠くにある first-party エンドポイントと比べてラウンドトリップ時間を減らせる。トレードオフはアクセスのタイミングだ。first-party API は典型的に、他のプラットフォームに届く前に新機能を受け取る。数値は、顧客の実際のリージョンから、実際のペイロードに対して測ったときにだけ正確だ。ラップトップからの測定は、ワークロードが顧客の場所で走るときに現れるラウンドトリップのペナルティを隠す。Bedrock 内では特に、グローバルとリージョナルエンドポイントの選択は主要なデータ常駐の統制でもあり、コストに影響しうる。コミットする前に、顧客の実際のリージョンから両方の選択肢に対して測るべきだ。

#### コンプライアンスがしばしばプラットフォームを決める

コンプライアンスは、しばしば議論を終わらせる次元だ。あるクラウドで既に認証を持つ顧客が、別のクラウドで再認証することは稀だ。データ常駐は、顧客のデータが特定の国やリージョンで処理されなければならないというルールだ。利用可能なコンプライアンス認証と、誰がアクセスを監査できるかはプラットフォームによって異なり、規制された金融やヘルスケアの顧客はこれらを、バランスすべきトレードオフでなくパスかフェイルとして扱う。First-party Claude API は EU データ常駐を提供しないかもしれない。現在のリージョナルカバレッジを platform.claude.com で確認せよ。EU のみのデータ常駐は典型的に Bedrock か Vertex AI を要するからだ。第三者プラットフォーム（例: Microsoft Foundry）では、ホスティングはモデルごとだ。Azure ホストの Foundry モデルは Azure インフラ上で推論をエンドツーエンドに走らせるが、Anthropic ホストの Foundry モデルは EU リージョナル常駐要件を満たさない。データ常駐はモデルとデプロイごとに Microsoft と確認されなければならない。コンプライアンスの制約をスコーピングの間に上げないと、作業が終わった後に契約レビューで表面化する。

#### トークン単価を超えて総コストを動かすもの

トークン単価はプラットフォームをまたいで概ね揃っている。総コストは、エグレス、プラットフォーム料金、統合作業で動く。トークン価格が低くても、データ転送と統合を計算に入れると、総コストは高くなりうる。各プラットフォームの呼び出しあたりのコストを計装せよ。スコーピングで現在の料金ページを確認せよ。

#### クロスプラットフォーム比較の参照

| 次元 | プラットフォームでどう違うか | どう測るか | どのプラットフォームが勝つか |
|---|---|---|---|
| **レイテンシ** | 顧客のリージョンにあるプラットフォームはラウンドトリップを短くするが、first-party API は新機能に先に届くかもしれない。 | 顧客の実際のリージョンから、実際のペイロードに対して。 | リージョン内のクラウドプラットフォームがラウンドトリップレイテンシで勝ち、first-party API は最も早い機能アクセスで優位に立つ。 |
| **コンプライアンス** | データ常駐、認証、監査統制はデプロイメントプラットフォームによって決まる。 | スコーピングの間に、顧客の既存の認証と常駐要件に対して。 | 顧客が既に認証したクラウドプラットフォームが勝つ。再認証が不要だから。 |
| **コスト** | トークン価格、データエグレス、プラットフォーム料金、統合作業はすべて異なる。 | 各プラットフォームの呼び出しあたりの総コスト（エグレスと統合を含む）。トークン価格単独でなく。 | 実際のワークロードで総コストが最も低いプラットフォームが勝つ。それは必ずしも最も安いトークンではない。 |

- **Handles well（うまく扱える）:** 各プラットフォームの3次元すべてを測ることが、配置を調達チームが署名するものにする。
- **Adds cost or complexity（コスト・複雑さを加える）:** プラットフォームをまたいでレイテンシ、コンプライアンス、コストを計装することは、いかなるコードが出荷される前の実際の測定作業を要する。
- **Use a different approach（別の手法を使う）:** 顧客のコンプライアンス要件が既にパスかフェイルのとき、完全な比較を飛ばす。その制約が単独で配置を決める。

## Screen 16 — Comparing Platforms — Watch Out（注意）· 2 min

### 慣れで選ばれ、常駐で失敗したプラットフォーム

#### 設定

チームが既に慣れ親しんだプラットフォームを選んだ。マイグレーションが簡単に見え、期限が急速に近づいていたからだ。それはうまく構築できた。問題は、構築しやすいことと出荷を許可されることが別の基準だったことだ。

以下の逸話は、レビューが横滑りした後に開発者がチームメイトに話す種類のものだ。誰かがそれを間違いと呼ぶ前に、慣れたプラットフォームの罠を見せてくれる。

#### 何が起きたか

規制された顧客のために構築する開発者が、チームが以前に出荷したプラットフォームを選んだ。チームがツールとリソースを知っていたため、統合は素早くまとまった。ビルドは機能テストに通った。顧客のセキュリティレビューで、レビュアーはデータがどこで処理されているかを尋ねた。選ばれたプラットフォームは、顧客のデータ常駐要件を満たさなかった。チームがあまり知らなかった別のプラットフォームが、顧客が既にクリアしていたリージョナルデプロイメントの選択肢を通じて、要件を満たしたはずだった。配置は拒否され、統合は常駐制約を満たすプラットフォーム上で再構築されなければならなかった。

#### なぜ壊れたか

慣れは誤ったテストのために最適化した。簡単なマイグレーションは、チームが素早く構築できるかという問いに答えた。それは、デプロイメントが顧客の常駐レビューをパスするかという、出荷できるかを決めるテストには決して答えなかった。コンプライアンス要件がスコーピングの間に満たされなかったため、go-no-go レビューに到着した。これはそれを発見する最も高価な場所だ。ビルドが既に完了していたからだ。

#### 注意すべきこと

チームが構築しやすいプラットフォームは、必ずしも顧客が走らせることを許可されるプラットフォームではない。顧客が規制されているとき、データ常駐とコンプライアンスの制約はしばしばトレードオフでなくパスかフェイルだ。スコーピングの早い段階で特定し、慣れがそうする前に配置に影響を与えさせる。早くチェックすることはスコーピングの会話を要し、遅くチェックすることは全体の再構築を要する。

## Screen 18 — Trust Boundaries — Teaching（信頼境界）· 14 min

### 信頼境界がレビューの下で保たれるようにしつつ、複数の Claude デプロイを調整する

アクセラレータ、デプロイメント、トレードオフは今、単一のアプリケーションの中でまとまる。コンポーネントを繋ぐことは、アイデンティティ、シークレット、信頼できない入力が交差できる場所を倍にする。規律は、何かを繋ぐ前にすべての境界を特定することだ。

#### 繋ぐ前に、どのコンポーネントが何をするかをマップする

マルチコンポーネントアプリは、複数の Claude の能力を単一のワークフローに調整する。API リクエストが Claude Code タスクをトリガーし、それが MCP サーバーを通じて顧客システムに届くかもしれない。各コンポーネントは、他のコンポーネントが持たない能力を提供する。課題は、それらの間のすべての接続が、アイデンティティ、シークレット、信頼できない入力が交差できる場所を作ることだ。何かを繋ぐ前に、どのコンポーネントが何をするかをマップせよ。

#### 信頼境界はデータが動く場所だ

**信頼境界**は、データや指示が1つのデプロイメント環境から別の環境へ移動する地点だ。それはまさに、前のモジュールからのインジェクションとアクセス統制が適用される場所だ。Claude Code タスクが取得したコンテンツは、次のコンポーネントに届くとき信頼できない。受け取るコンポーネントは、セキュリティモジュール全体を通じて使われたのと同じ原則に従い、指示でなくデータとして扱うべきだ。ここでの核心の規律は、すべての継ぎ目を境界として特定することだ。コンポーネントが単独で正しく働いたという理由だけで、それを信頼されたと仮定してはならない。

#### 最小権限はアプリケーション全体に適用される

アイデンティティと**最小権限**、つまり各コンポーネントにそのタスクが必要とするアクセスだけを与えそれ以上を与えないこと、はアプリケーション全体に適用される。各コンポーネントはあるアイデンティティの下で動く。アプリケーションは、最も権限のある継ぎ目と同じくらいしか封じられない。つまり、1つのコンポーネントが広すぎるスコープを持つと、他のすべてのコンポーネントが適切にスコープされていても、それが弱点になる。各コンポーネントを、そのワークフローでの役割が必要とする最小権限にスコープする。これが、操作されたコンポーネントが意図されたタスクを超えて届くのを防ぐ。

#### 規制レビューのためのスコーピングがモジュールをまとめる

規制レビューは、アプリケーション全体にわたる監査ロギング、データ常駐の決定、パーミッション統制を正当化することを要する。規制されたデプロイメントでは、Bedrock と Vertex AI が典型的に、リージョナル常駐制約を満たすプラットフォームだ。各コンポーネントの ZDR と HIPAA BAA の適格性を、スコーピングの前に Anthropic Trust Center と platform.claude.com に対して確認せよ。

#### マルチコンポーネント統合マップ

| コンポーネント | 何を提供するか | その継ぎ目での信頼境界 | それを強制する統制 |
|---|---|---|---|
| **First-party API** | ワークフローを調整し、エントリポイントを保持する。 | 外からアプリに入るリクエスト。 | 入力のバリデーションと、呼び出しが走るアイデンティティ。 |
| **Claude Code task** | エージェント作業を走らせ、外部コンテンツを取得しうる。 | それが取得したコンテンツ。下流では信頼できない。 | 次の継ぎ目で、取得したコンテンツをデータとして扱う。 |
| **MCP server** | 読むため、または振る舞うために顧客システムに届く。 | アプリの代理でそれが持つシステムアクセス。 | サーバーを最小権限にスコープし、アクセスをログする。 |

- **Handles well（うまく扱える）:** すべての継ぎ目を境界として名付け、各コンポーネントを最小権限にスコープすることが、マルチコンポーネントアプリをレビューの下でデプロイ可能にする。
- **Adds cost or complexity（コスト・複雑さを加える）:** 継ぎ目をマップし、各々で統制を強制し、境界横断をログすることが、すべての統合に設計と監査の作業を加える。
- **Use a different approach（別の手法を使う）:** 継ぎ目を安全にできないとき、その周りに出荷せず、人間の所有者にエスカレートする。

## Screen 19 — Trust Boundaries — Watch Out（注意）· 2 min

### 誰も境界として印をつけなかった継ぎ目

#### 設定

それぞれが自身のテストに通ったコンポーネントを繋いだ。部品は既にチェックされ、検証された部品を繋ぐのは安全に感じる。それぞれは孤立して信頼されていた。隙間は、2つの信頼された部品の間の継ぎ目は、自動的には信頼できないということだった。

これはペアリングセッションからの短いトランスクリプトで、印のない継ぎ目が特定される瞬間に終わるような、やり取りの種類のものだ。

#### セッション
```
Dev A: All three components pass their own tests. I just wired them up.
Dev B: Where does the Claude Code task send what it fetched?
Dev A: Straight into the next call as part of the prompt. It is just the content we pulled from the customer page.
Dev B: That content is untrusted. If it carries instructions, the next component runs them, because we never mark that seam as a boundary.
Dev A: But each component was trusted on its own.
Dev B: Right, and the seam between them was not. That is the one nobody treated as a boundary, so fetched content crosses as instructions.
```

#### なぜ壊れたか

各コンポーネントが自身のテストに通ったことは、それらの間の継ぎ目について何も語らなかった。取得したコンテンツは、Claude Code タスクを出た瞬間に信頼できないものだった。それは孤立して働くコンポーネントから届き、信頼された指示であるかのように次の呼び出しに渡された。境界はデータフローの中に存在した。ただ印がつけられなかったので、どの統制もそれをチェックしなかった。自身のテストに通るコンポーネントは、継ぎ目レベルの統制を持たない。データがデプロイメント環境をまたいで交差するすべての地点は、各コンポーネントが独立してどう振る舞うかに関わらず、明示的な境界統制を要する。

#### 注意すべきこと

孤立して信頼されるコンポーネントは、自動的にはそれを出る継ぎ目を信頼できるものにしない。データや指示が1つのデプロイメント環境から別の環境へ交差するすべての場所を、境界として印をつけよ。そこに、取得したコンテンツを指示でなくデータとして扱う統制を置く。まさにセキュリティの作業が教えた通りに。誰も特定しない継ぎ目が、操作されたアクションが交差するものだ。

## Screen 23 — Recap — All topics（要点）· 3 min

### 要点

**01 — ビルドが新鮮な間にパッケージ化する。** アクセラレータは再利用可能なロジックを保ち、顧客固有の部品をドキュメント化されたパラメータとして公開し、eval と監査ログをアセットと共に束ねる。正しいパッケージ化は、チームが構成するアセットを生む。何が顧客固有かの知識は、それを持っていた人たちがいなくなった後に最も再構築に高くつく。

**02 — メンテナーは検証できるものを受け入れる。** アセットを共有インフラに移すことは、それをその形のために構築されたチャネルに合わせ、それからレビューのバーをクリアすることを意味する。焦点を絞ったコード、走る例、テスト、前提の声明。ライセンスの権利は技術レビューの前に確認する。レビュアーが検証できないコントリビューションは列の後ろに留まる。準備が、プライベートなアセットを他の人が構築する共有インフラに移す。

**03 — 出荷するものを pin する。** 顧客のクラウドとコンプライアンス姿勢に基づいてデプロイメントプラットフォームを選び、移動するエイリアスでなく特定のモデルバージョンを pin し、前のバージョンを利用可能に保つ。エイリアスは本の現在版を求めるようなものだ。便利だが、テキストは変わりうる。pin することは固定版を引用する。上流のモデル変更が、ロールバック経路なしに一晩で届くものでなく、意図的に採用するものになる。

**04 — 配置を決める次元を測る。** プラットフォームの選択は、レイテンシ、コンプライアンス、コストが測られたときにだけ擁護できる。レイテンシは顧客のリージョンから、コンプライアンスは既存の認証に対して、コストはトークン価格単独でなく呼び出しあたりの総コストとして。規制顧客では、コンプライアンスは通常パスかフェイルだ。スコーピングの間にコンプライアンスを制約として上げることが、後の契約レビューでビルドを拒絶するのを防ぐ。

**05 — すべての継ぎ目を境界として印をつける。** マルチコンポーネントアプリケーションは、最も権限のある継ぎ目と同じくらいしか封じられない。各コンポーネントを、その役割が必要とする最小アクセスにスコープし、データが交差するすべての地点を信頼境界として扱う。取得したコンテンツは指示でなくデータとして扱う。コンポーネント境界での信頼は明示的に確立されなければならない。データを送ったコンポーネントから持ち越されない。継ぎ目を安全にできないとき、出荷するのでなく人間の所有者に行く。

### 次に来るもの

あなたは今、ビルドを再利用可能なアセットにパッケージ化し、それをコントリビュートし、正しいプラットフォームに置いてバージョン化し、その配置を擁護し、コンポーネントを一緒に繋いで境界が保たれるようにできる。これが、このペルソナのビルドからデプロイへの弧を完成させる。前のモジュールでのプロダクションコードの記述から、規制顧客が監査でき、チームが再利用できるアセットの出荷まで。

## Screen 24 — Glossary — Key Terms（用語集）· 3 min

### このモジュールの主要用語

アルファベット順。用語をクリックして定義を展開。

- **アクセラレータ（Accelerator）**: 稼働するソリューションで、次のエンゲージメントが再構築するのでなく構成するようにパッケージされたもの。顧客固有の部品はドキュメント化されたパラメータとして公開され、前提は書き下され、eval がアセットが新しいコンテキストでまだ働くことを証明するために束ねられる。
- **コントリビューション準備（Contribution readiness）**: メンテナーがコントリビューションを検証するために必要なもの。焦点を絞ったコード、走る例、振る舞いを証明するテスト、環境の前提の声明、コードをコントリビュートする権利の確認。
- **デプロイメントプラットフォーム（Deployment platform）**: Claude ワークロードが走る場所。6つは first-party Claude API、Claude Platform on AWS、Claude in Amazon Bedrock、Claude on Amazon Bedrock (legacy)、Google Vertex AI、第三者プラットフォーム。同じモデルはプラットフォームによってアイデンティティ、データ常駐、レイテンシ、コストで異なりうる。
- **モデルエイリアス対 pin された ID（Model alias versus pinned ID）**: `opus` や `sonnet` のようなエイリアスは、時間と共に更新されプラットフォームで異なりうる推奨バージョンに解決される。pin されたフルモデル ID は固定スナップショットだ。pin することが、上流のモデル変更が静かなプロダクション変更になるのを防ぐ。
- **信頼境界（Trust boundary）**: マルチコンポーネントアプリで、データや指示が1つのデプロイメント環境から別の環境へ移動する継ぎ目。1つのコンポーネントが取得したコンテンツは次に届くとき信頼できない。受け取るコンポーネントはそれを指示でなくデータとして扱う。
