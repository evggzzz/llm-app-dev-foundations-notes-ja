## Screen 1　オリエンテーション（Orientation）

> 2 min
> このモジュールを終えると、できるようになること

ビルドをしっかり文書化しておけば、次のエンゲージメントはゼロからではなく、そこから出発できる。

これまで3つのモジュールで、プロダクション品質のエージェントを構築し、適切なパーミッションとコンテキストの統制を整えて Claude Code に組み込み、セキュリティレビューを通る MCP 接続を設定した。それぞれが、動くビルドだった。

このモジュールは、ビルドが動くようになった後のことを扱う。次のエンゲージメントでゼロから作り直すか、一度パッケージ化して次のチームが設定するだけで済むようにするか。後者の道を選ぶことで、同じものを繰り返し作り直す時間が解放され、新しい仕事に回せる。

このモジュールを終えるとき、次のことができるようになる。

1. 稼働するソリューションを、再利用可能な**アクセラレータ**としてパッケージ化する。パラメータ化したエージェントテンプレート、構成可能な MCP サーバー、持ち運べる eval スイートのいずれであれ、次のエンゲージメントはゼロから作り直すのではなく、アセットを設定するだけで済む。
2. ツール、パターン、修正を、文書化されたチャネルを通じてコントリビュートし、メンテナーが受け入れられる形に整え、プライベートなアセットを共有インフラに変える。
3. Claude のワークロードを、first-party API、Amazon Bedrock、Google Vertex AI、サードパーティプラットフォームのどこで動かすかを選び、モデルやプロンプトの変更がプロダクションを暗黙的に壊さないように、出荷するものをバージョン管理する。
4. それらのプラットフォームを、レイテンシ、コンプライアンス、コストで比較する。チームがなんとなく手に取ったデフォルトではなく、調達やセキュリティのチームが承認できる選択にする。
5. 複数の Claude デプロイを一つのワークフローにまとめるアプリケーションを構築し、データとアイデンティティの境界がセキュリティやコンプライアンスのレビューに耐えるようにスコープを決める。

このモジュールは、動くビルドを手にし、今それを長持ちさせなければならない Developer 向けだ。これまでのモジュールは、読者が実践的で、コードを中心に据え、パターンで考える人であることを前提に進んできた。この時点で、プロダクションのエージェントを書き、Claude Code で構成し、セキュリティレビューを通る MCP 接続を組み、eval でそのすべてを証明できるはずだ。本モジュールは、それらをもう一度教え直すことはしない。コードが正しく動き始めた瞬間を起点にして、より難しい問いを投げる。他人が再利用できるか。メンテナーが受け入れられるか。モデルの更新を生き残れるか。セキュリティや調達のチームが、実行場所に対してGOを出せるか。ここでの作業はコードを書くことよりも、書き上げたコードを、再利用可能で、デプロイ可能で、コードを書いていない人にも擁護できるものにするための判断の方だ。

### このモジュールで言う「ビルド」

このモジュールのすべては、繰り返し現れる一つの隙間にかかっている。動くビルドは、まだ再利用やレビュー、デプロイを生き残れるビルドではない。開発中は、テンプレートは動き、コントリビューションは自分の問題を解き、モデルは応答を返し、プラットフォームは構築しやすく、各プラットフォームは独自のテストに合格した。そのどれもが完成状態ではない。同じテンプレートを、自分と一度も話したことのないチーム向けに構成し直さなければならない。同じコントリビューションを、メンテナーが何も再構築せずに検証できる形にしなければならない。同じモデルを、上流の変更が突然の事故ではなく計画的な決定になるようにピンしなければならない。同じプラットフォームが、顧客のデータレジデンシーとコンプライアンスのレビューをクリアしなければならない。同じ繋がったプラットフォーム群が、監査の下で信頼境界を保ち続けなければならない。このモジュールの各トピックは、同じ教訓の別の姿である。コードが動き始めた地点こそが、このモジュールの作業の始まりだ。思いのほか多くの判断は、顧客がすでに持っているクラウド、コンプライアンスの姿勢、レビュープロセスによって決まる。

## Screen 2　再利用のためのパッケージ化（Packaging for Reuse）

> Teaching · 16 min
> 動くビルドをパッケージ化し、次のエンゲージメントがアセットから始められるようにする

前のモジュールを終えて、あなたは動くビルドを手にしている。エージェントのループ、構成済みの MCP サーバー、プロンプトが機能することを証明する eval だ。チームで最も時間を食い、コストがかかるのは、次の顧客向けに同じものを作り直すエンジニアリングの時間だ。

### アクセラレータの役割：再利用可能な部分を残し、残りを切り分ける

アクセラレータとは、将来のエンゲージメントが、空のリポジトリではなく動く土台から始められるようにパッケージ化されたソリューションだ。設計の言葉で言えば、これは再利用のためのパッケージ化（packaging for reuse）にあたる。エンゲージメント固有のコードを再利用可能なコアから切り離し、残りをパラメータ化する。動くビルドを取り、顧客固有の部分を切り分け、ドキュメント化されたデフォルト値を持つパラメータとして外に出す。するとアセットは、全体を書き直されるのではなく、構成されるようになる。ビルドが新しいうちに再利用向けのパッケージ化を済ませておけば、数ヶ月後に意図を復元するよりも安く済む。ある値がなぜハードコードされたかを知っていた人がいなくなった後での復元は、最も高くつく。

### 再利用可能な成果物は複数のアセット型に分かれ、それぞれパッケージ化の方法が違う

再利用可能な成果物の多くは、このモジュール全体で使う3つの型のいずれかに当てはまる。テンプレート、構成可能なサーバー、持ち運べる eval だ。それぞれが異なる種類の作業を含み、それぞれの方法でパッケージ化する必要がある。型を取り違えると、アセットは再利用可能に見えながら、実際には適用しにくいものになる。

| アセット型 | 同梱するもの | 正しいパッケージ化に必要なこと |
|---|---|---|
| エージェントテンプレート | 動くエージェントから取り出した、システムプロンプト、ツールのスキーマ、ループ構造。 | ドメイン固有の値を、ドキュメント化されたデフォルト付きで構成値に引き出す。新しいチームがループを編集するのではなく、値を設定するだけで済むように。 |
| MCP サーバーパッケージ | サーバーが公開するツールと、その入力、インストール側のチームが制御するスコープ。 | 各ツールの入力をドキュメント化し、インストール側のチームがスコープを設定できるようにする。コードを編集せずに、新しい環境へインストールできるように。 |
| eval スイート | アセットが機能することを証明する、採点済みのテストセットとジャッジルーブリック。 | データセットとルーブリックをセットで出荷する。新しいチームが自分のコンテキストで実行し、そこでもアセットが機能することを確認できるように。同じ eval スイートはデプロイ時のゲートも兼ねる。新しいモデルバージョンをプロダクションに昇格させるときは、バージョンをプロダクションに出す前に、ピンしたベースラインスコアに対して実行する。 |

エージェントをテンプレートではなく、緩いスクリプト集として出荷するのが、間違ったやり方で最もよくある形だ。スクリプトは動くので再利用可能に見える。しかし顧客固有の値はすべて別々のファイルに埋まり、次のチームは一つのアセットを構成するのではなく、スクリプトをコピーしては派生させることになる。

### コードと前提を両方ドキュメント化する

コードが記述するのは振る舞いだ。ドキュメントが担うのは、後の開発者がソースを読むだけでは確実に推論できない部分である。アセットが環境について置く前提、期待する入力、すでに扱っている失敗モード、そして「まだ機能するか」を定義する eval。これがなければ、次のチームはアセットをブラックボックス扱いし、作り直すことになる。

### 監査ログをパッケージの一部として同梱する

規制の効く顧客のレビュー担当者は、アセットがどのデータに触れるか、どのアイデンティティで動くか、どんなログを残すかを尋ねる。これらを持たないアクセラレータは、デモは通っても最初のセキュリティレビューで止まる。監査ログをパッケージの一部として扱ってほしい。

### パッケージ化のチェックリスト

パッケージ化を進める間、このチェックリストをビルドのそばに置いてほしい。各列は、アセットごとに一度決める判断だ。

| アセット型 | パラメータ化するもの | ドキュメント化するもの | 監査用に同梱するもの |
|---|---|---|---|
| エージェントテンプレート | 顧客ごとに変わるすべての値。プロンプト、パス、スコープ、参照渡しのクレデンシャル、閾値。 | 環境の前提、期待する入力、扱っている失敗モード、機能することを定義する eval。 | 触れるデータ、動くアイデンティティ、アセットが行ったことのログ。 |
| MCP サーバー | スコープ、参照渡しのクレデンシャル、顧客や環境ごとのパス。 | ツールごとの期待する入力、スコープの境界、扱っている失敗モード。 | 触れるデータ、動くアイデンティティ、アセットが行ったことのログ。 |
| eval スイート | 顧客や環境ごとに変わる閾値とデータセットのパス。 | ルーブリックのロジック、スコアの意味、アセットをピンしているベースライン。 | 触れるデータ、動くアイデンティティ、アセットが行ったことのログ。 |

**うまく機能する場合**
ビルドが新しいうちにパラメータ化を済ませると、一度の納品が、次のエンゲージメントが数時間で構成できるアセットになる。

**コストや複雑さが増す場合**
一般化できる部分と顧客固有の部分を切り分け、前提をドキュメント化する作業は、最初のビルドに実質的な時間を加える。

**別の手法を選ぶ場合**
顧客が二度と再利用しない一回限りのものなら、パッケージ化のオーバーヘッドは割に合わない。ビルドを出荷して次に進む。

## Screen 3　再利用のためのパッケージ化（Packaging for Reuse）

> Watch Out · 2 min
> すぐに出荷できたが、再利用できなかったテンプレート

**セットアップ**
ハードコードの方が早く出荷でき、締め切りに追われていたため、デモを動かした値をハードコードした。テンプレートは動いた。だからこそ、次のチームが再利用しようとするまで、誰ももう一度見ることはなかった。

これはポストモーテム（事後分析）だ。再利用の試みが失敗した後にチームが書くのと同じ書き方で書いてある。誰かがそれをミスと呼ぶ前に、失敗の形を見られるようにするためだ。

### 何が起きたか

あるチームが顧客のエンゲージメント向けにエージェントテンプレートを構築し、期日通りに出荷した。締め切りに間に合わせるため、顧客固有の値はそのままコードに書き込まれた。リポジトリのパス、モデル名、レビューの閾値、その顧客のドメインに固有のプロンプト断片だ。テンプレートは動き、エンゲージメントは終わり、ビルドは再利用可能というラベル付きで共有リポジトリに置かれた。

数ヶ月後、別のチームが似たエンゲージメントでこれを拾い上げた。しかし構成できなかった。構成すべきものが何もなかったからだ。変えるべき値はすべてループの中に焼き込まれており、ファイル全体を読まなければ第二のチームには見えなかった。どの値が顧客固有で、どの値が荷重を支える（load-bearing）かを示すドキュメントもなかった。eval も同梱されていなかったため、編集を推測で行った後でも、テンプレートが新しいコンテキストでまだ機能するかを確かめる手段が何もなかった。彼らはゼロから書き直さなければならなかった。

### どこで壊れたか

ビルドは、再利用できる瞬間ではなく、動いた瞬間に完成したと扱われた。ハードコードは締め切り下では妥当な判断だったが、その後見直されることはなかった。動くテンプレートは、自分が再利用できないことを自ら知らせることはない。コストは、パッケージ化が防ぐはずだった再構築を、第二のチームが背負ったときに初めて現れた。テンプレートが行き止まりだとわかるまでに失われた時間も、そのチームが支払った。

### 注意すべきこと

動くテンプレートは、まだ再利用のためにパッケージ化されていない。これらは違う完成状態だ。警告サインは3つの欠如にある。顧客固有の値が入るべき場所にパラメータがないこと、前提を記述したドキュメントがないこと、異なるコンテキストでもアセットが機能することを証明する eval が同梱されていないこと。ビルドが新しいうちにパッケージ化してほしい。何が顧客固有かという知識は、それを持っていた人が去った後に復元するのが最も高い。

## Screen 5　コミュニティへのコントリビュート（Contributing Back）

> Teaching · 12 min
> アセットを、プライベートな再利用から、メンテナーが受け入れる共有インフラへ移す

あなたはすでに、アセットを共有可能にする作業の大部分を終えている。自分のチームで再利用するためにパッケージ化したとき、パラメータを引き出し、前提を書き下し、eval を同梱した。パラメータは、アセットが書き直しではなく構成できることを示す。ドキュメント化された前提は、アセットが期待する環境をメンテナーに伝える。同梱された eval は、まだ機能することを確認する手段を与える。社内再利用向けにパッケージ化されたアセットは、すでにメンテナーが受け入れるのに必要なものに近い。

コントリビューションチャネルは、そのパッケージ化されたアセットを受け取るように作られている。バージョン、インストール手順、構成要素を一つの単位として運ぶ。そのため、一度も話したことのないチームでも、インストールするだけで同じ稼働構成を手に入れられる。

### コントリビューションを、それ用に作られたチャネルに合わせる

コントリビューション（contributing back）とは、文書化されたチャネルを通じて、アセットをプライベートな再利用から共有インフラへ移すことだ。各チャネルは、特定の種類のコントリビューションに合わせて作られている。Claude Cookbook は、焦点を絞ったリファレンス実装を集めた GitHub リポジトリだ。自己完結型の、単一または複数パターンの実装を、明確に示し、エンドツーエンドで動く形で見せることに向けて作られている。オープンソースの MCP サーバーやツールは、それぞれ独自のリポジトリとコントリビューションの慣習を持つ。Cookbook に完全な複数コンポーネントのアプリケーションを送るのはミスマッチだ。そのリポジトリはアプリケーション全体ではなく、一つの焦点を絞ったパターンをレビューする前提で作られている。その規模の提案は、レビュー担当者が探しているものに合わず、止まってしまう。第一歩は、コントリビューションを、それ用に作られたチャネルに合わせることだ。焦点を絞った例が置かれるべき場所に、完全なアプリケーションを置くのが、コントリビューションがレビューされないまま終わる最も一般的な理由の一つだ。

### コントリビューションの検証を可能にするもの

メンテナーが受け入れるのは、検証できるコントリビューションだ。そのハードルは、コードの賢さではなく、彼らが確認しなければならないことで決まる。検証を可能にするのは、次の4つだ。

1. コードが一つのことをする。広がりすぎたコントリビューションは、レビュー担当者に、評価する前に意図を復元することを強いる。
2. 動いているところを示す例がある。レビュー担当者が、振る舞いを見るためにハーネスを組み立てる必要はないはずだ。
3. それが機能することを証明するテストがある。テストにより、メンテナーは推論を自分で再現せずに結果を検証できる。
4. 前提を明記した短い一文がある。そうでなければ、最初の失敗がメンテナーの問題になる。

### 権利と帰属は技術レビューに先立つ

ライセンスと帰属は、コントリビューションを受け入れられるかどうか、そのものを決める。だから技術レビューより前に来る。顧客のエンゲージメントから持ち込まれたコードには、どこに行けるかについて制約があるかもしれない。自分がコントリビュートする権利を持っていること、そして自分が基にしたものを帰属させることを確認するのが、コントリビューションがまず通らなければならないゲートだ。これを飛ばすと、コントリビューションは法務チームが後から解かなければならない問題に変わる。

ここで扱う例は、カスタマーサービスエージェントのケースだ。エンゲージメント中に構築された再利用可能な会話処理パターンを、顧客固有の部分を剥がして、Cookbook 向けの汎用例として整える。コントリビュートして戻す動きは、このカリキュラムの3つのロールすべてに共通する。Developer としてのあなたの仕事は技術的な準備だ。焦点を絞ったコード、例、テスト、前提、そして権利確認。エンゲージメントのコンテキストは、より広いチームから来る。

### コントリビューション準備の参照表

| チャネル | メンテナーが確認すること | ライセンスと帰属 | 例とテストのハードル |
|---|---|---|---|
| 焦点を絞った例なら Cookbook。ツールや修正なら、そのツールやサーバー自身のリポジトリ。 | コードが一つのことをしているか、全体を読めるか。 | エンゲージメントからのコードをコントリビュートする権利を確認し、先行する成果物の帰属を示すこと。 | 振る舞いを証明するテストと、実行可能な例があること。説明だけではないこと。 |

**うまく機能する場合**
パッケージ化されたアセットは、例、テスト、権利確認を加えるだけで、他の人が基にする共有インフラになる。

**コストや複雑さが増す場合**
メンテナーのハードルとライセンスのゲートをクリアするのは、自分のコードを動かすための作業に加えて必要な、本来の手間だ。

**別の手法を選ぶ場合**
コードが、クリアできないエンゲージメントのライセンス制約を担っているときは、コントリビュートしない。代わりにオーナーにエスカレーションする。

## Screen 6　コミュニティへのコントリビュート（Contributing Back）

> Watch Out · 2 min
> メンテナーが検証できなかったプルリクエスト

**セットアップ**
自分の問題を解いたそのままのコードで、コントリビューションをオープンした。自分のケースで機能し、アクセスしやすかったため、自然な選択だった。あなたには機能した。しかしそれこそが、見知らぬ人が信じるために必要なものが、すべて抜けていた理由だ。

これは内部チャンネルでのやり取りだ。プルリクエストが沈黙する理由を、メンテナーがどう説明するかを聞けるようにするためだ。

### やり取り

```
Developer: My PR has been open three weeks with no review. The code works, I use it every day. What is the holdup?
Maintainer: It probably works for you. The problem is I can't tell. There is no test I can run, no example that proves the behavior, and nothing saying what it assumes about the environment.
Developer: So, you want me to add a test and an example?
Maintainer: Yes. A contribution a reviewer cannot verify sits at the back of the queue until someone has time to reconstruct what it does. A focused PR with a test and an example gets reviewed fast because there is nothing left for me to reverse-engineer.
```

### どこで壊れたか

コードは正しかった。コントリビューションが止まったのは、メンテナーが開発者の作業を復元しないと検証できなかったからだ。その隙間は見落としやすい。作者には欠けているコンテキストがすでにあるためだ。例、テスト、前提の記述は、コードを作った人にはどれも自明に見える。しかしメンテナーにはそうではない。そして、意図を復元しなければならないレビュー担当者は、必ず最後にそれをやる。

### 注意すべきこと

プルリクエストは、レビュー担当者が検証できないものの上で止まる。コントリビューションをオープンする前に、動いていることを示す例、振る舞いを証明するテスト、前提を明記した短い一文を追加してほしい。この3つが、コントリビューションを列の最後から早いレビューへと動かす。メンテナーがリバースエンジニアリングするものを残さないからだ。

## Screen 7A　要件とライフサイクル（Requirements & Lifecycle）

> Teaching · 8 min
> ビジネス要件から、機能要件とインフラ要件へ

この後に続くデプロイメントプラットフォームの判断はすべて、要件がすでに存在することを前提にしている。データレジデンシーの規則、レイテンシの目標、アイデンティティモデルだ。この画面は、その要件がどこから来るかを扱う。ビジネス上の問題を、デプロイの判断がそれに対して擁護できる機能要件とインフラ要件に変換する。

### ビジネス問題から機能要件を取り出す

機能要件は、システムがすべきことを、確認できるほどの詳細度で記述する。ビジネス上の問題（例「サポート担当者の回答を速くする」）は、まだ要件ではない。そこから機能要件が導かれる（例「各チケットを4つのキューのいずれかに分類する。関連するポリシーを引用して返信案を作成する。人の承認なしに自動送信しない」）。ここでの作法は、それぞれを確認可能な振る舞いの記述として書くことだ。曖昧な目標は設計の根拠にも検証の対象にもならない。一方、具体的な記述は eval の一行になり、レビューでの基準になる。

### インフラ要件を導出する

インフラ要件とは、デプロイが満たさなければならない非機能の制約だ。その多くはビジネス問題の中には書かれていない。ビジネス問題が暗示する問いを立てることで導出する。レイテンシは、ユーザーのいる場所で測ったとき、応答はどれだけ速く必要があるか。スケールは、何件のリクエストか、ピークはどれだけか。レジデンシーは、データがどこで処理されなければならないか。どの規制の下でか。アイデンティティは、誰が、どのクレデンシャルで動くか。何を監査可能にしなければならないか。レイテンシ、スケール、レジデンシー、アイデンティティは、デプロイメントプラットフォームを決める最も頻出のインフラ要件であり、別の理由でプラットフォームが選ばれる前の、最初の段階で最も捉えやすい。

### 決定が擁護できるように要件を文書化する

要件を書き留めておくのは、デプロイの判断が、要件の収集に関わらなかった人たちによってレビューされるからだ。機能の振る舞い、インフラの制約、各制約の根拠となる規則をカバーする短い要件記録があれば、「慣れ親しんだから」ではなく「要件から導かれるから」としてプラットフォーム選択を擁護できる。この記録が、次の画面のデプロイ判断が読み込む入力になる。

**うまく機能する場合**
どのプラットフォームを選ぶ前よりも先に、ビジネス問題を確認可能な機能要件とインフラ要件に変換しておく。

**コストや複雑さが増す場合**
前もってインフラの制約を引き出すには、チームが飛ばしたがるスコーピングの会話が必要だ。

**別の手法を選ぶ場合**
レビューがなく、規制対象のデータも扱わない、使い捨てのプロトタイプなら、軽いメモで十分だ。

## Screen 7C　要件とライフサイクル（Requirements & Lifecycle）

> Teaching · 8 min
> Claude アプリケーションのシステムライフサイクル

直前で取り出した要件は、より長い流れの第1フェーズだ。この画面はその流れをシステムライフサイクルと名付ける。これによって、このモジュールの残りの部分で扱うデプロイ、バージョニング、境界の作業が、バラバラのタスクとしてではなく、正しいフェーズに収まるようになる。

### Claude アプリケーションに当てはめたライフサイクルの各フェーズ

Claude アプリケーションは、他のエンジニアリングされたシステムと同じライフサイクルをたどる。モデルの作業は、そのフェーズに割り当てられる。

1. **要件**：機能要件とインフラ要件を取り出す。
2. **設計**：プラットフォーム、モデル、信頼境界を選ぶ。
3. **構築**：エージェント、ツール、プロンプトを書く。
4. **テスト**：eval、ユニット、結合、エンドツーエンドの各検証。
5. **デプロイ**：バージョンをピンし、昇格を eval でゲートする。
6. **運用**：コスト、レイテンシ、エラーを計測し、ガードレールを効かせる。
7. **反復**：プロダクションでの発見を要件に戻す。

これらは、前のモジュールが一つずつ教えたフェーズと同じだ。ライフサイクルとして捉え直すことで、どう繋がっているかが見えてくる。

### フェーズ間のゲーティング

ゲートとは、あるフェーズから次へ進む判断であり、規制対象のエンゲージメントが統制を保つ場所だ。プラットフォームがレジデンシー要件を満たすまで、設計から構築へは進まない。新しいバージョンが、ピンしたベースラインに対する eval をクリアするまで、デプロイから完全なプロダクションへは進まない。エンジニアリングの作業を正しいフェーズに置き、ゲートを飛ばさないことが、Claude アプリケーションをレビュー可能な状態に保つ。

**うまく機能する場合**
エンジニアリングの作業の各ピースを、それが属するライフサイクルのフェーズに、成果物とゲートを定義して置く。

**コストや複雑さが増す場合**
フェーズ間にゲートを置くと、締め切りに追われるチームが飛ばしたがるチェックポイントが増える。

**別の手法を選ぶ場合**
一回限りの実験ならフェーズを畳み込んでもよいが、規制対象のデプロイでは畳み込めない。

## Screen 8　デプロイメントとバージョニング（Deployment & Versioning）

> Teaching · 15 min
> Claude ワークロードがどこで動くかを選び、出荷するものをバージョン管理する

パッケージ化されたアセットも、コントリビュートされたアセットも、どちらも何かが動かすまでは単なるコードだ。アセットはここで別の問いに向き合う。どこで動かし、バージョンをどう固定するか。上流の変更が、プロダクションの追跡されない変更にならないようにするためだ。プラットフォームの判断が、技術的な優劣だけで決まることはまずない。実際には、顧客がすでにどこにクラウドインフラ、アイデンティティ管理、コンプライアンスの契約を持っているかで大抵は形作られる。最初の問いは通常、顧客がすでに信頼し、運用しているプラットフォームはどれか、だ。

### プラットフォームは、通常、顧客のクラウドで決まる

デプロイメントプラットフォームとは、Claude のワークロードが動く環境のことだ。同じモデルを複数のデプロイ環境で動かせるが、通常は顧客が既存で持っているクラウドがどれかを決める。first-party Claude API は Anthropic 自身の環境で、通常は新しい機能を最初に受け取る。Claude Platform on AWS は、顧客の AWS アカウント経由でアクセスし、Anthropic 自身のモデル ID とライフサイクルを使う。推論は AWS の境界の外で Anthropic が運用する。Amazon Bedrock は2つの統合を提供する。Claude in Amazon Bedrock は、`/anthropic/v1/messages` で Messages API を使い、first-party と広く機能パリティを持つ。サポートされない機能のリストが存在するため、機能固有の要件は Bedrock のドキュメントに対して確認してほしい。一方、Claude on Amazon Bedrock（レガシー）は、ARN でバージョン管理された識別子を使い、InvokeModel／Converse API を使う。Google Vertex AI は、Google Cloud の中で同じことをする。Microsoft Foundry のようなサードパーティプラットフォームは、顧客がすでに使っている製品の中に Claude を組み込む。Microsoft Foundry は Claude を2つのホスティング形態で提供する。Hosted on Azure（現在 Claude Opus 4.8、Claude Sonnet 5、Claude Haiku 4.5。推論は Azure インフラ上でエンドツーエンドで動く。一般提供）と、Hosted on Anthropic（その他すべての Foundry Claude モデル。推論は Anthropic 運営のインフラ上で動く）。規制対象の顧客のレジデンシーの前提は、そのモデルのホスティング形態に依存する。ビルド時に、ホスティング形態と現在のモデル分割を Microsoft と確認してほしい。

### アイデンティティとデータレジデンシーはセキュリティ上重要

アイデンティティとデータの場所は、コードではなくプラットフォームが決める。Bedrock は AWS のアイデンティティを使い、データを顧客の AWS 境界の中に保つ。Vertex は Google Cloud のアイデンティティと境界を使う。両者とも、レジデンシーが制約になるときにはリージョナルルーティングを提供する。プラットフォームを顧客の既存のコンプライアンス契約に合わせれば、データレジデンシーのレビューをゼロからやり直さずに済む。

### 上流のモデル変更がプロダクションへの暗黙的な変更にならないよう、バージョンをピンする

バージョン管理は、モデルやプロンプトの変更がプロダクションへの暗黙的な変更になるのを防ぐ。すべての Claude モデル ID は、特定のモデルスナップショットを指す。Opus や Sonnet のようなエイリアスは便利だが、時間とともに変化し、デプロイメントプラットフォームによって異なるバージョンに解決されうる。ピンした完全なモデル ID は、固定のスナップショットに解決される。エイリアスではなく、特定のモデルバージョンをピンしてほしい。そうすれば、上流のモデル更新は偶然ではなく計画的な選択になる。次に、プロンプトとアセットも、コードと並べてバージョン管理する。最後に、前のバージョンを利用可能なまま残し、リグレッション時にロールバックできるようにする。ピンされていないデプロイは、上流のモデル更新のたびに、出力への追跡されない変更を作る。

最初の行は動くエイリアスを追う。2行目はスナップショットをピンする。

```
# Pre-4.6 example: a convenience alias can resolve to a new
# version without you knowing
model = "claude-haiku-4-5"

# Pre-4.6 pinned snapshot: the version is fixed until you change this line
model = "claude-haiku-4-5-20251001"
```

Claude 4.6 以降では、モデル ID だけで特定のスナップショットにピンされる。それより前のモデルでは、ID に日付サフィックスの追加が必要だ。ビルド時に platform.claude.com で現在の慣習を確認してほしい。

### eval を通じてバージョンを昇格させる

昇格を eval スイートでゲートする。新しいバージョンを一部のトラフィックに流し、ピンしたベースラインと比較し、その結果で昇格かロールバックかを決める。ここで eval は一回限りのテストではなくなり、デプロイのゲートになる。

### デプロイメントプラットフォームの判断表

| プラットフォーム | アイデンティティとデータモデル | 選ぶべきとき | バージョニングのピン方法 |
|---|---|---|---|
| first-party Claude API | Anthropic のアイデンティティと利用規約。 | 顧客にクラウドやレジデンシーの拘束がなく、最新の機能を求めている。 | 完全なモデル ID をピンし、前のスナップショットを残す。 |

## Screen 9　デプロイメントとバージョニング（Deployment & Versioning）

> Watch Out · 3 min
> モデルエイリアスが動いて壊れたデプロイ

**セットアップ**
推奨バージョンを指すエイリアスに対して出荷した。便利なデフォルトであり、最新モデルがただで手に入ったからだ。それは機能した。その後エイリアスが進み、ただだと思っていたものには代償があることがわかった。

これはプロダクションログのトレース抜粋で、インシデント後に巻き戻して読むようなものだ。出力の形が変わった日と、ロールバック先がなかった理由がわかる。

### ログ

```
--:  deploy: model="opus"  status=ok
--:  alias advanced -> new opus version (no app change)
--:  parser: KeyError "summary" in response payload
--:  Error: output shape changed; downstream parse failed
--:  rollback attempted -> no pinned prior version retained
--:  incident: hotfix parser; root cause = unpinned deployment
```

### どこで壊れたか

アプリケーションは一度も変わらなかったが、エイリアスは変わった。ピンされた前のバージョンが保持されていなかったため、ロールバック先が何もなかった。ホットフィックスはパーサーを直したが、ピンされていないデプロイをそのまま残した。

### 注意すべきこと

エイリアスの指す先は常に動く。ピンされた完全なモデル ID は固定のスナップショットだ。上流の更新を、自分が意図して採用するものにするために、完全なモデル ID をピンしてほしい。リグレッションがホットフィックスではなくロールバックで済むように、前のピンされたバージョンを利用可能なまま残す。出力の形の変化がプロダクションではなくテスト実行で現れるように、新しいバージョンを昇格させる前に、自分の eval を通してゲートする。

## Screen 11　プラットフォームの比較（Comparing Platforms）

> Teaching · 12 min
> レイテンシ、コンプライアンス、コストでプラットフォームを比較し、選択がレビューを生き残るようにする

直前の2画面で、プラットフォームを選び、バージョンをピンした。その選択は顧客のクラウドにとって正しかった。しかし「彼らのクラウドに合っている」は、調達やセキュリティのチームがGOを出す論拠には、まだならない。

### 顧客のリージョンからレイテンシを測る

レイテンシは、プラットフォームが顧客に対してどこで動くかと、新機能へのアクセスがどうルーティングされるかに依存する。顧客自身のクラウドリージョンで動くプラットフォームは、遠くにある first-party エンドポイントと比べてラウンドトリップ時間を短くできる。ただしトレードオフはアクセスのタイミングだ。first-party API は通常、他のプラットフォームに届くより前に新機能を受け取る。数値が正確になるのは、顧客の実際のリージョンから、顧客の実際のペイロードに対して測ったときだけだ。ノートPCからの測定は、ワークロードが顧客の場所で動き始めたときに現れるラウンドトリップのペナルティを隠してしまう。Bedrock に限って言えば、グローバルとリージョナルのエンドポイントの選択は、主要なレジデンシー制御でもあり、コストにも影響する。決める前に、顧客の実際のリージョンから両方の選択肢に対して測ってほしい。

### コンプライアンスがプラットフォームを決めることが多い

コンプライアンスは、しばしば議論を終わらせる次元だ。あるクラウドで認定を既に持っている顧客は、別のクラウドで再認定するとは考えにくい。データレジデンシーとは、顧客のデータを特定の国やリージョンで処理しなければならないという規則だ。利用可能なコンプライアンス認定と、アクセスを監査できる主体は、プラットフォームによって異なる。規制対象の金融や医療の顧客は、これらをバランスを取るトレードオフではなく、合格か不合格かとして扱う。first-party Claude API は EU データレジデンシーを提供しない場合がある。EU 専用のレジデンシーは通常 Bedrock か Vertex AI を必要とするため、現在のリージョンカバレッジを platform.claude.com で確認してほしい。Microsoft Foundry のようなサードパーティプラットフォームでは、ホスティングはモデルごとに決まる。Azure ホストの Foundry モデルは Azure インフラ上で推論をエンドツーエンドで実行する一方、Anthropic ホストの Foundry モデルは EU リージョンのレジデンシー要件を満たさない。レジデンシーはモデルとデプロイごとに Microsoft と確認しなければならない。コンプライアンスの制約はスコーピングの段階で上げておかないと、作業が終わった後の契約レビューで表面化する。

### トークン単価以外に総コストを動かすもの

トークン単価はプラットフォーム間で概ね揃っている。総コストを動かすのは、エグレス、プラットフォーム料金、統合作業だ。データ転送と統合を加味すると、トークン価格が低いほうが総コストとして高くつくことがある。プラットフォームごとに1回あたりのコストを計測してほしい。スコーピングの段階で現在の価格ページを確認すること。

### プラットフォーム間比較の参照表

| 次元 | プラットフォームによる違い | 測り方 | どこが勝つか |
|---|---|---|---|
| レイテンシ | 顧客のリージョンにあるプラットフォームはラウンドトリップを短くする。一方、first-party API は新機能に早く届く。 | 顧客の実際のリージョンから、実際のペイロードに対して。 | リージョン内のクラウドプラットフォームはラウンドトリップレイテンシで勝ち、first-party API は機能への早期アクセスで優位。 |
| コンプライアンス | データレジデンシー、認定、監査のコントロールはデプロイメントプラットフォームが決める。 | スコーピングの段階で、顧客の既存の認定とレジデンシーの要件に対して。 | 顧客がすでに認定を取っているクラウドプラットフォームが勝つ。再認定がいらないため。 |
| コスト | トークン価格、データエグレス、プラットフォーム料金、統合作業がすべて変わる。 | トークン単価だけでなく、エグレスと統合を含めた1回あたりの総コストをプラットフォームごとに。 | 実際のワークロードに対して総コストが最も低いプラットフォームが勝つ。それは必ずしも最も安いトークンではない。 |

**うまく機能する場合**
3つの次元をすべてプラットフォームごとに測ると、配置の判断が調達チームが承認できるものになる。

**コストや複雑さが増す場合**
レイテンシ、コンプライアンス、コストをプラットフォームをまたいで計測するには、コードを出荷する前に本当の測定作業が必要だ。

**別の手法を選ぶ場合**
顧客のコンプライアンス要件がすでに合格か不合格かで決まるなら、完全な比較は飛ばしてよい。その制約が単独で配置を決める。

## Screen 12　プラットフォームの比較（Comparing Platforms）

> Watch Out · 2 min
> 馴染みで選んだプラットフォームがレジデンシーで落ちた

**セットアップ**
チームがすでに馴染みのあるプラットフォームを選んだ。移行が簡単に見え、締め切りが迫っていたためだ。ビルドはうまくいった。問題は、「構築しやすい」と「出荷してよい」が違う基準だったことだ。

以下は、レビューがこけた後に開発者がチームメイトに話すような話だ。馴染みのプラットフォームの罠を、誰かがミスと呼ぶ前に見られるようにするためだ。

### 何が起きたか

規制対象の顧客向けに構築する開発者が、チームが以前出荷したことのあるプラットフォームを選んだ。チームはツールとリソースをよく知っていたため、統合はすぐにまとまった。ビルドは機能テストに通った。顧客のセキュリティレビューで、レビュー担当者はデータがどこで処理されているかを尋ねた。選ばれたプラットフォームは、顧客のレジデンシー要件を満たさなかった。チームがあまり詳しくない別のプラットフォームが、顧客がすでにクリアしていたリージョンデプロイの選択肢を通じて、その要件を満たしていたはずだ。配置は却下され、レジデンシーの制約を満たすプラットフォーム上で統合をやり直さなければならなかった。

### 注意すべきこと

## Screen 14　信頼境界（Trust Boundaries）

> Teaching · 14 min
> 複数の Claude デプロイを、信頼境界がレビューの下で保たれたまま調整する

アクセラレータ、デプロイ、トレードオフはここで一つのアプリケーションにまとまる。コンポーネントを繋ぐと、アイデンティティ、シークレット、信頼できない入力が交差する場所が掛け算で増える。ここでの作法は、何かを繋ぐ前に、すべての境界を特定することだ。

### 繋ぐ前に、どのコンポーネントが何をするかをマップする

複数コンポーネントのアプリケーションは、複数の Claude の能力を一つのワークフローにまとめる。API リクエストが Claude Code のタスクを起動し、そのタスクが MCP サーバーを通じて顧客のシステムに届く、というかもしれない。各コンポーネントは、他が持たない能力を一つずつ提供する。難しいのは、コンポーネント間の接続がそれぞれ、アイデンティティ、シークレット、信頼できない入力が交差する場所を作ることだ。何かを繋ぐ前に、どのコンポーネントが何をするかをマップしてほしい。

### 信頼境界は、データが動く場所だ

**信頼境界**とは、データや指示があるデプロイ環境から別のデプロイ環境へ移動する点のことだ。前のモジュールのインジェクション対策とアクセス制御がまさに適用される場所だ。Claude Code のタスクが取得したコンテンツは、次のコンポーネントに届いたとき、信頼できないものだ。受け取る側のコンポーネントは、セキュリティモジュール全体で使ったのと同じ原則に従い、それを指示ではなくデータとして扱うべきだ。ここで中心となる原則は、すべての継ぎ目を境界として特定することだ。あるコンポーネントが単体で正しく動いたからといって、それを信頼してよいとは仮定しない。

### 最小特権はアプリケーション全体に適用される

アイデンティティと**最小特権**（それぞれのコンポーネントに、そのタスクが必要なアクセスだけを与え、それ以上は与えないこと）は、アプリケーション全体に適用される。各コンポーネントは、あるアイデンティティの下で動く。アプリケーションのコンテインメントの強度は、最も特権の大きな継ぎ目で決まる。つまり、他のすべてのコンポーネントが適切にスコープされていても、一つでも広くスコープされすぎたコンポーネントがあれば、そこが弱点になる。各コンポーネントを、ワークフロー上の役割が必要とする最小特権にスコープする。こうしてはじめて、操作されたコンポーネントが、本来のタスクの範囲を越えられなくなる。

### 規制対象のレビューに向けたスコーピングがモジュールをまとめる

規制対象のレビューは、アプリケーション全体にわたる監査ログ、データレジデンシーの決定、パーミッションのコントロールを正当化することを求める。規制対象のデプロイでは、通常、Bedrock と Vertex AI がリージョンレジデンシーの制約を満たすプラットフォームになる。スコーピングの前に、各コンポーネントについて、ZDR と HIPAA BAA の適格性を Anthropic Trust Center と platform.claude.com で確認してほしい。

### 複数コンポーネントの統合マップ

| コンポーネント | 提供するもの | 継ぎ目にある信頼境界 | それを強制するコントロール |
|---|---|---|---|
| first-party API | ワークフローを指揮し、入口を持つ。 | アプリの外から入ってくるリクエスト。 | 入力のバリデーションと、その呼び出しが動くアイデンティティ。 |
| Claude Code タスク | エージェント的な作業を走らせ、外部コンテンツを取得することがある。 | それが取得したコンテンツ。下流では信頼できない。 | 次の継ぎ目で、取得したコンテンツをデータとして扱う。 |
| MCP サーバー | 顧客のシステムに届き、読み取る、または作用する。 | アプリの代わりに保持する、システムへのアクセス。 | サーバーを最小特権にスコープし、そのアクセスをログに残す。 |

**うまく機能する場合**
すべての継ぎ目を境界として明示、各コンポーネントを最小特権にスコープすれば、複数コンポーネントのアプリはレビューの下でデプロイ可能になる。

**コストや複雑さが増す場合**
継ぎ目をマップし、それぞれにコントロールを効かせ、境界横断をログに残す作業は、統合ごとに設計と監査の作業を加える。

**別の手法を選ぶ場合**
ある継ぎ目が安全にできないなら、そのまま出荷せず、人間のオーナーへエスカレーションする。

## Screen 15　信頼境界（Trust Boundaries）

> Watch Out · 2 min
> 誰も境界として明示しなかった継ぎ目

**セットアップ**
それぞれが単体のテストに通ったコンポーネントを繋いだ。各パーツはすでに検査済みで、検査済みのパーツを繋ぐのは安全に感じる。それぞれは単体では信頼されていた。隙間はこうだ。信頼された二つのパーツの間の継ぎ目そのものは、自動的には信頼できない。

これはペア作業の短いトランスクリプトで、明示されなかった継ぎ目が特定される瞬間に終わるようなやり取りだ。

### セッション

```
Dev A: All three components pass their own tests. I just wired them up.
Dev B: Where does the Claude Code task send what it fetched?
Dev A: Straight into the next call as part of the prompt. It is just the content we pulled from the customer page.
Dev B: That content is untrusted. If it carries instructions, the next component runs them, because we never mark that seam as a boundary.
Dev A: But each component was trusted on its own.
Dev B: Right, and the seam between them was not. That is the one nobody treated as a boundary, so fetched content crosses as instructions.
```

### どこで壊れたか

各コンポーネントが自身のテストに通ったことは、その間の継ぎ目について何も語らない。取得されたコンテンツは、Claude Code タスクを出た瞬間に信頼できないものだった。それが、単体では機能したコンポーネントから届き、まるで信頼された指示であるかのように次の呼び出しに渡された。境界はデータフローの中に存在した。ただ明示されなかったため、どのコントロールもそれを検査しなかった。自身のテストに通ったコンポーネントは、継ぎ目レベルのコントロールを持たない。デプロイ環境間でデータが交差するすべての点は、各コンポーネントが独立してどう振る舞うかとは無関係に、明示的な境界のコントロールを必要とする。

### 注意すべきこと

単体で信頼されたコンポーネントが、そこから出ていく継ぎ目を自動的に信頼できるわけではない。データや指示があるデプロイ環境から別の環境へ交差するすべての場所を、境界として明示してほしい。そこに、セキュリティの作業で教わったのと同じように、取得したコンテンツを指示ではなくデータとして扱うコントロールを置く。誰も特定しない継ぎ目を、操作されたアクションが越えていく。

## Screen 19　ふりかえり（Recap）

> All topics · 3 min
> 重要なポイント

**01**
ビルドが新しいうちにパッケージ化する。アクセラレータは、再利用可能なロジックを残し、顧客固有の部分をドキュメント化されたパラメータとして外に出し、eval と監査ログをアセットに同梱する。正しいパッケージ化は、チームが構成するアセットを作る。何が顧客固有かという知識は、それを持っていた人が去った後に復元するのが最も高い。

**02**
メンテナーが受け入れるのは、検証できるものだ。アセットを共有インフラに移すとは、その形に合わせたチャネルに対応させ、レビューのハードルをクリアすることだ。焦点を絞ったコード、実行可能な例、テスト、前提の記述。そしてライセンスの権利を技術レビューの前に確認すること。レビュー担当者が検証できないコントリビューションは、列の最後に置かれる。準備が、プライベートなアセットを、他の人が基にする共有インフラへと移す。

**03**
出荷するものをピンする。顧客のクラウドとコンプライアンスの姿勢に基づいてデプロイメントプラットフォームを選び、動くエイリアスではなく特定のモデルバージョンをピンし、前のバージョンを利用可能なまま残す。エイリアスは、本の「最新版」を頼むようなものだ。便利だが、中身は変わりうる。ピンは、固定された版を指す。だから上流のモデル変更は、自分が意図して採用するものになり、ロールバック経路のないまま一晩で届くものではなくなる。

**04**
配置を決める次元を測る。プラットフォームの選択が擁護できるのは、レイテンシ、コンプライアンス、コストが測られているときだけだ。レイテンシは顧客のリージョンから、コンプライアンスは既存の認定に対して、コストはトークン単価ではなく1回あたりの総コストとして。規制対象の顧客にとって、コンプライアンスは通常、合格か不合格かだ。コンプライアンスをスコーピングの段階で制約として上げておけば、後の契約レビューでビルドが却下されるのを防げる。

**05**
すべての継ぎ目を境界として明示する。複数コンポーネントのアプリケーションのコンテインメントは、最も特権の大きな継ぎ目で決まる。各コンポーネントを、その役割が必要とする最小のアクセスにスコープし、データが交差するすべての点を信頼境界として扱う。取得したコンテンツは、指示ではなくデータとして扱う。コンポーネント境界での信頼は、明示的に確立しなければならない。データを送ったコンポーネントからは持ち越されない。ある継ぎ目が安全にできないときは、出荷するのではなく、人間のオーナーへ回す。

## Screen 19B　用語集（Glossary）

> Key Terms · 3 min
> このモジュールの主要な用語

アルファベット順。用語をクリックすると定義が展開する。

**アクセラレータ（Accelerator）**
次のエンゲージメントが、ゼロから作り直すのではなく、構成するだけで済むようにパッケージ化された、稼働するソリューション。顧客固有の部分はドキュメント化されたパラメータとして外に出され、前提は書き下され、アセットが新しいコンテキストでもまだ機能することを証明する eval が同梱される。

**コントリビューション準備（Contribution readiness）**
メンテナーがコントリビューションを検証するために必要なもの。焦点を絞ったコード、実行可能な例、振る舞いを証明するテスト、環境の前提の記述、そしてコードをコントリビュートする権利の確認。

**デプロイメントプラットフォーム（Deployment platform）**
Claude のワークロードが動く場所。6つある。first-party Claude API、Claude Platform on AWS、Claude in Amazon Bedrock、Claude on Amazon Bedrock（レガシー）、Google Vertex AI、サードパーティプラットフォーム。同じモデルでも、プラットフォームによって、アイデンティティ、データレジデンシー、レイテンシ、コストが異なりうる。

**モデルエイリアスとピンされた ID（Model alias versus pinned ID）**
`opus` や `sonnet` のようなエイリアスは、時間とともに更新される推奨バージョンに解決され、プラットフォームによって異なりうる。ピンされた完全なモデル ID は、固定のスナップショットだ。ピンすることで、上流のモデル変更がプロダクションへの暗黙的な変更になるのを防ぐ。

**信頼境界（Trust boundary）**
複数コンポーネントのアプリにおいて、データや指示があるデプロイ環境から別のデプロイ環境へ移動する継ぎ目。あるコンポーネントが取得したコンテンツは、次に届いたときには信頼できないため、受け取るコンポーネントはそれを指示ではなくデータとして扱う。
