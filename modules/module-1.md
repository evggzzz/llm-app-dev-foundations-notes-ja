## Screen 1 — Orientation（オリエンテーション）（2 min）

### このモジュールを終えるとできること

Claude に向けてコードを一行書く前に、まず言葉の意味を知っておくと役立つ。

このモジュールでは、モデルの基礎と技術的な土台を説明する。Developer コースの残りは、これらをすでに身につけている前提で進む。

このモジュールを終えると、次のことができるようになる。

1. トークンとは何か、コンテキストウィンドウが固定の予算としてどう働くか、サンプリングがなぜ出力を変えるのか、非決定性がテストや eval で何を意味するのかを説明する。
2. Claude のモデルファミリーと能力階層を説明し、モデルの選択と、extended thinking のような推論モードを有効にすることを区別する。
3. zero-shot、one-shot、multi-shot のプロンプトモードから選び、例を追加することのコストと品質のトレードオフを評価する。
4. 開発者が Claude にどうアクセスするかを説明する。SDK か生の REST か、同期かストリーミングか、大量処理向けの非同期パターンの違いを含む。

## Screen 2 — Teaching: How LLMs Behave（LLM の振る舞い）（12 min）

**取り上げる主題:** LLM の振る舞い。トークン、コンテキスト、サンプリング、非決定性。

**タブ:** `Tokens` | `Context Window` | `Sampling` | `Non-determinism`

### トークン：入力、出力、コストの単位

Claude は文字や単語をそのまま読むのではない。読むのはトークンである。トークンあたりの平均文字数は対象モデルのトークナイザに依存し、モデルの世代が変われば値も変わる。文字数をトークンで割った大まかな目安はモデル依存と考え、ビルド時には現在のトークナイザの振る舞いを確認すること。

モデルが処理するものは、すべてトークンとして数えられる。プロンプト、会話履歴、ツール定義、ツール結果、そしてモデルが生成する応答。トークンは料金と予算の共通単位でもある。ある機能のコストを見積もるときも、入力が収まるかを判定するときも、数えるのは単語ではなくトークンである。API の課金もコンテキストウィンドウの計測もトークン単位で行われるため、トークンで考える習慣を持つとよい。

### コンテキストウィンドウ：固定の予算

コンテキストウィンドウは、1回のリクエストでモデルが取り込めるトークンの総量である。システムプロンプト、それまでの会話全体、差し込んだドキュメント、すべてのツール結果、モデルの出力を一度に保持する。これは固定の予算で、2種類の境界挙動がある。

入力がすでにウィンドウを超えているリクエストは、生成が始まる前にバリデーションエラーで拒否される。一方、入力時点では収まっていても、生成の途中で上限に達することがある。現在のモデルはその時点で生成を止め、エラーを送出する代わりに、それまでに生成した出力を `model_context_window_exceeded` という stop reason とともに返す。

どちらの場合も、長いセッションを維持するには、アプリケーションが呼び出しごとに履歴を切り詰めるか要約する必要がある。開発時はテスト入力が短いため、ウィンドウが埋まることはほとんどない。プロダクションでは入力が長くターン数も増えるため、ウィンドウが早く埋まる。この失敗については Module 2 で詳しく扱う。

### サンプリング：同じプロンプトで違う答えが返る理由

言語モデルは、次のトークンをあらかじめ一つに決めて選ぶわけではない。各ステップで、取りうる次トークンの確率分布を生成し、そこからサンプリングする。temperature のような設定が、その分布の形を決める。温度を低くすると確率が最も高いトークンに集中し、出力が再現しやすくなる。温度を高くすると分布が広がり、出力が多様になる。

選択が固定ではなくサンプリングで決まるため、同じプロンプトを2回実行すると、両方とも正解であっても表現が変わることがある。これはモデルの生成方法に由来する性質である。

サンプリングの制御はモデル依存であることにも注意したい。最新の Claude モデルは、デフォルト以外のサンプリングパラメータを受け付けない。temperature、top_p、top_k を設定すると 400 エラーが返り、それらのモデルではプロンプトで振る舞いを制御する。temperature を受け付けるモデルでも、temperature を 0 にすれば出力は再現しやすくなるが、呼び出し間で完全に同一になることは保証されない。ビルド時には API リファレンスで現在のパラメータ対応状況を確認すること。

### 非決定性：テストと eval が受ける影響

非決定性は、サンプリングがもたらす主要な帰結である。同じ入力でも、同じ出力は保証されない。これが、Claude の機能に対するテストの考え方を変える。

応答のテキストを厳密に assert するテストは安定しない。モデルは同じ正しい答えをさまざまな言い回しで表現できるからである。代わりに、満たすべき性質を assert する。必須フィールドが存在する、値が範囲内にある、構造がパースできる、といったことである。構造ではなく意味を判定したいときは、モデル採点のジャッジ（model-graded judge）を使う eval を使う。

だから本コースは eval を「機能が正しいと知る」ための標準と位置づけ、Module 3 でその能力を構築する。

## Screen 3 — Teaching: Models & Reasoning（モデルと推論）（10 min）

**取り上げる主題:** モデルの選択肢と推論モード。

**タブ:** `The Model Family` | `Reasoning Modes` | `How They Work Together`

### Claude のモデルファミリー

Claude は現在4つの階層からなるモデルファミリーである。Fable、Opus、Sonnet、Haiku。各モデルは、コスト、レイテンシ、能力の異なるトレードオフを表す。

Sonnet は、ほとんどのプロダクションワークロード向けのバランスの取れたデフォルトである。Haiku は、自身の能力の範囲に収まるタスク向けに、スピードとコスト効率を高めて作られている。Opus は Sonnet の範囲を超える、要求の高い作業を扱う。Fable は最も能力の高い階層で、最大の知性が優先される、もっとも要求の高い推論、コーディング、エージェント作業向けである。

実用的な進め方は、Sonnet から始めることである。eval が現在の階層では品質基準に届かないことを示したときだけ1階層上げ、eval が品質の低下を許容できることを示したときだけ Haiku に下げる。Claude ファミリーは進化しているため、ビルド時には platform.claude.com/docs で現在のモデルラインナップと識別子を確認すること。

### 推論モードはモデル選択とは別の設定

どのモデルを実行するかが1つの決定である。モデルが答える前に推論するかどうかは、呼び出しごとに決める、もう1つの決定である。

現在のモデルでは、推論モードは adaptive thinking である。モデルがいつ、どれだけ考えるかを決め、固定のトークン予算ではなく effort 設定で深さを調整する（古い `budget_tokens` 制御は非推奨で、最新のモデル世代では 400 エラーを返す）。thinking の内容は、最新モデルではデフォルトで応答から省かれる。表示が必要なときは、要約された形式をリクエストする。推論は難しく段階の多い問題でそのコストに見合う価値があり、単なる参照や分類では無駄になる。

このモジュールの要点は、2つの設定要素が独立して組み合わさるということである。モデル選択がファミリーのメンバーを選び、推論モードはリクエストごとに設定する。モデルごとにデフォルトは異なる（最新モデルの一部は、デフォルトで、あるいは常に適応的に考える）。対象モデルの現在の thinking のデフォルトは、ビルド時に確認しておく必要がある。

### 2つの設定をどう組み合わせるか

モデル選択と推論モードは独立しているため、それぞれ別々に設定できる。推論をオフにした能力の高いモデルは速く直接的である。推論をオンにした小さなモデルは、考えるために多くのトークンを消費する。もっとも要求の大きいタスクでは、能力の高いモデルに高い effort 設定を組み合わせる。

Module 2 は、推論を有効にし、返される thinking ブロックを扱う方法を説明する。コスト、レイテンシ、品質と天秤にかけてどのモデルを実行するかは、Module 4 で扱う。

## Screen 4 — Teaching: Prompting Modes（プロンプトモード）（8 min）

**取り上げる主題:** プロンプトモード。zero-shot、one-shot、multi-shot。

**タブ:** `The Three Modes` | `Cost & Quality Trade-off` | `Mode & Model Choice`

### 3つのモード

プロンプトの言葉選びとは別に、プロンプトの中にいくつの実例を与えるかという問題がある。

zero-shot は指示だけを与え、例は与えない。タスクを記述し、結果を求める。one-shot は、入力と望ましい出力のペアを例として1つ加える。multi-shot（few-shot とも呼ぶ）は、そのような例を複数含める。これらの例は訓練データではない。プロンプトの中に置かれ、求める答えの正確な形をモデルに示す。記述だけでは、その形を最後まで決めきれないことが多い。

### コストと品質のトレードオフ

追加する例は、それぞれ毎回の呼び出しでトークンを消費し、コンテキストの予算を圧迫する。つまり品質とコストのトレードオフになる。

タスクが単純で出力の形が明らかなら、zero-shot で十分である。出力に、記述では拾いきれない特定の構造や大文字小文字の区別、エッジケースがあるときは、one-shot や multi-shot に進む。多くの場合、正しい例を1つか2つ追加するだけで、指示をもう1段落追加するより早く問題が解決する。Module 2 でも繰り返す基本は、信頼できる結果を生み出す最小限のプロンプトを追加することである。

### モード選択とモデル選択の相互作用

プロンプトモードとモデル選択は、相互に関連する設定要素である。より能力の高いモデルは、小さなモデルが構造を合わせるために例を必要とするようなタスクでも、zero-shot で成功することが多い。つまり例を追加すれば、より安いモデルに仕事を任せられる場合がある。

この2つの決定は、一緒に検討する価値がある。eval を満たす、最もシンプルなモデルと最少の例を試し、eval が必要だと言う箇所だけ、能力や例を追加する。

## Screen 5 — Teaching: Technical Substrate（技術的土台）（12 min）

**取り上げる主題:** 技術的土台。SDK、REST、ストリーミング、非同期。

**タブ:** `SDK vs. REST` | `Sync, Streaming & Real-time` | `Async for High-Volume Work`

### 開発者が Claude にアクセスする経路：SDK か生の REST か

Claude には HTTP REST API 経由でアクセスする。コードは、API キーと JSON ボディを伴ったリクエストをエンドポイントに送り、JSON 応答を読み取る。このエンドポイントは、任意の HTTP クライアントから直接呼べる。

より一般的なのは公式 SDK を使う方法である（Python や TypeScript 向けなど）。SDK は同じ REST API の上に被せた薄い便利レイヤーで、認証、リクエストの構築、リトライ、応答のパースを引き受けるため、定型文を書かずに済む。SDK も生の REST も、同じ API、同じモデルにアクセスする。SDK はリクエストの手組みを省いてくれる。Module 2 では、この同じ土台の上にある SDK と Messages API を使って実装を解説する。

### 同期、ストリーミング、リアルタイムの応答

同期リクエストは最もシンプルなパターンである。リクエストを送り、完全な応答がひとまとまりで戻ってくるのを待ってから処理する。短い応答や、誰も待っていないバックエンドジョブにはこれで十分である。

応答が長い場合やユーザーが見ている場合は、ストリーミングを使う。モデルが生成するのに従って応答が小片で送られ、画面が空白のまま待たされることなく出力がすぐ現れる。コードはその小片を最終的なメッセージに組み直す。Claude は Server-Sent Events を使い、同じ HTTP 接続上でストリーミングを提供する。Module 2 は、ストリームを安全に消費し、中断時に復旧する方法を説明する。

### 大量処理向けの非同期パターン

大量処理に対応するパターンは2つあり、それぞれ異なる問題を解く。

Python SDK は非同期クライアント（`AsyncAnthropic`）を提供し、非ブロッキングの async/await を使い、アプリケーションのスレッドを占有せずに API 呼び出しを行う。TypeScript SDK の標準クライアントは Promise ベースなので、そのまま await で呼び出せる。非同期クライアントクラスは別にない。いずれの場合もリクエストはリアルタイムで戻るが、待っているあいだにアプリケーションはほかの処理を進められる。ブロックせずに並行性が必要な場面に適する。

Message Batches API は、バルクのオフラインワークロード向けのもう1つのパターンである。大量のリクエストを1回の呼び出しで送信し、識別子を受け取り、完了をポーリングする。バッチジョブは完了まで最大24時間かかり、そのレイテンシと引き換えにトークン単価が下がる。ユーザーが各結果を待っておらず、ターンアラウンドよりもコストが重要な、オフラインパイプライン、評価実行、バルクジョブに適する。

- A. そのとおり。モデルは決定論的（deterministic）である。
- B. 必ずしもそうではない。モデルは次のトークンを確率分布からサンプリングするため、両方の答えが正しくても表現が変わることがある。

- A. 同じ設定である。
- B. extended thinking は別のモデルである。
- C. モデル選択はファミリーのどのメンバーを実行するかを選び、extended thinking は、対応する任意のモデルがオン・オフで実行できる呼び出しごとの設定である。

## Screen 7 — Exercise: Predict the Behavior（演習：振る舞いを予測する）（6 min）

以下の各シナリオは、このモジュールの4つの土台（サンプリング、プロンプトモード、リクエスト形態、コンテキスト予算）のいずれかから取り出した構成を示す。各シナリオについて、正しい振る舞いを予測し、その理由も示す選択肢を選べ。4問中3問正解なら部分点がある。

**ボタン:** `Submit` | `Skip for now`

**Scenario 1.** 分類タスクを `temperature` 0 で実行した場合と、同じタスクを高い `temperature` で実行した場合を比べる。繰り返し実行したとき、出力がどう違うかを予測せよ。
- A. `temperature` が低いと、モデルは確率を最も高いトークンに集中させる。そのため繰り返し実行しても同じラベルがはるかに一貫して返る。ただし `temperature` 0 でさえ決定性は保証されない。`temperature` が高いと分布が広がり、表現だけでなく選ばれるラベルまで変わることがある。分類器には、低い温度で再現しやすい振る舞いが望ましい。
- B. どちらの設定でも毎回同一の出力が返る。`temperature` は選ばれるトークンではなく、応答の長さにだけ影響するからである。
- C. `temperature` が高いほうが精度が高い。分布を広げることで、モデルはより多くの正しい答えを検討できるからである。
- D. 分類タスクに `temperature` は影響しない。分類はサンプリングに関係なく常に固定ラベルを返すからである。

> **訳注（推定正解）:** A。低い温度では最も確率の高いトークンに集中して一貫性が増し、高い温度では分布が広がってラベルすら変わりうる。

**Scenario 2.** zero-shot プロンプトで構造の誤った出力が返り続けるタスクを考える。multi-shot に切り替えると何が変わるかを予測せよ。
- A. multi-shot への切り替えは、新しい構造でモデルを再訓練する。一度例を送れば、以後のすべての呼び出しで変更が永続する。
- B. 正しい入出力の例を2つか3つ追加すると、モデルに合わせるべき正確な構造を示せる。多くの場合、これで指示を足しても直らなかった構造の問題が解決する。コストは毎回の呼び出しで余分にトークンがかかることなので、出力を信頼できるようにする最少の例にとどめる。
- C. multi-shot は構造の問題に効かない。出力の形を変えるには `temperature` を上げるしかない。
- D. multi-shot は呼び出しあたりのトークンコストを下げる。例のおかげでモデルがより短い応答を生成できるからである。

> **訳注（推定正解）:** B。例はモデルを再訓練するのではなく、プロンプト内で正確な構造を示すことで構造のズレを直す。

**Scenario 3.** ユーザーを待たせずに1晩で50,000件のドキュメントを処理しなければならないパイプラインを考える。どのリクエスト形態が適するか、そしてその理由を予測せよ。
- A. 同期のループが最も適する。ドキュメントごとに1回 API を呼ぶのが最もシンプルなパターンで、バッチ送信のオーバーヘッドを避けられるからである。
- B. ストリーミングが最も適する。応答を小片で送ることで、パイプラインが各ドキュメントの処理を早く始められるからである。
- C. バッチパターンが適する。リクエストをバッチで送信し、完了をポーリングする。レイテンシが長くなることを引き換えに、トークン単価が下がる。同期ループはレートリミットに当たり、アプリケーションを占有する。誰も見ていないのでストリーミングには何の得もない。
- D. より大きなコンテキストウィンドウが最も適する。50,000件すべてを1リクエストに収めれば、呼び出しを繰り返さずに済むからである。

> **訳注（推定正解）:** C。バッチはレイテンシと引き換えに低コストで、ユーザーが待たないバルク処理に適する。同期ループはレートリミットとスレッド占有の問題がある。

**Scenario 4.** 長いマルチターンのエージェントセッションで、コンテキストウィンドウが埋まり続ける場合を考える。症状を予測し、該当する予算の名前を答えよ。
- A. モデルは場所を作るため、最も古いターンを黙って捨てる。セッションは続くが、エラーなしに初期のコンテキストが静かに失われる。
- B. コンテキストウィンドウは固定のトークン予算である。履歴とツール結果が蓄積するにつれて埋まっていく。すでに大きすぎる入力は生成前にエラーで拒否される。入力時には収まるが生成中に上限に達したリクエストは、切り詰められた出力と `model_context_window_exceeded` という stop reason で戻る。症状は、テストでは問題なく動いたセッションが、入力が大きくなると失敗することである。だからアプリケーションが履歴を切り詰めるか要約する。
- C. 症状はサンプリングの遅延である。該当する予算は `temperature` 設定で、セッションが長くなるにつれて下げなければならない。
- D. 固定の予算はない。ウィンドウは蓄積する履歴をすべて保持するため自動的に広がり、長いセッションがこの理由で失敗することはない。

> **訳注（推定正解）:** B。コンテキストウィンドウは固定予算で、超過時には生成前エラーまたは生成中の `model_context_window_exceeded` による切り詰めとして現れる。

## Screen 8 — Recap: Five Takeaways（ふりかえり：5つの要点）（2 min）

1. **トークンは入力、出力、コストの単位である。** 単語ではなくトークンで考え、予算を立てる。API が計量し、コンテキストウィンドウが測るのはトークンだからである。
2. **コンテキストウィンドウは、リクエスト全体を一度に保持する固定のトークン予算である。** 大きすぎる入力は生成前にエラーになる。生成中に上限に達すると `model_context_window_exceeded` という stop reason で切り詰められた出力が返る。だから履歴の管理はアプリケーションの仕事である。
3. **サンプリングは生成を非決定的にする。** 同じプロンプトでも、実行のたびに違う表現が返りうる。だから厳密なテキストでテストするのは当てにならない。これが eval の出番である。
4. **モデル選択と推論モードは、別々に組み合わせられる設定要素である。** eval を満たす、最も小さなモデルと最もシンプルな推論、プロンプトを選ぶ。そして eval が必要だと言うところだけ、能力を追加する。
5. **開発者は REST API 経由で、通常は SDK を通じて Claude にアクセスする。** 同期、ストリーミング、async/await、バッチのどれを選ぶかは、ユーザーが待っているかどうかと、ワークロードがリアルタイムかバルクのオフラインかで決まる。
